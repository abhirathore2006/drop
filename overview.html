Static site publishing · on company infra

Push a folder.
Get a URL.

Drop is a self-hosted, Surge.sh-style platform for publishing pre-built static sites (and React SPAs) to *.drop.company.com — gated by Google login, backed by S3, with no database.

$ drop publish ./dist myapp
  ▸ packing dist/ (42 files)
  ▸ dropping…
  ✓ live at https://myapp.drop.company.com
TypeScript · Node 24 S3 only — no DB Google SSO CLI · MCP · dashboard atomic deploys + rollback
01 — The idea

Surge.sh, but it never leaves your network

Developers constantly need to share a built static site — a demo, a docs build, a React app — at a real URL, without provisioning anything. Public tools (Surge, Netlify, Vercel) are perfect UX but can't host internal builds. Drop brings that exact "push a folder, get a URL" flow inside the company, on your own S3 and your own network.

FOR

Any developer

Build locally, run one command (or ask an AI agent), and the site is live on an internal subdomain in seconds.

NOT

A build platform

No server-side builds, no SSR. You ship the finished dist/; Drop just stores and serves it.

SECURE

Internal-only

Publishing is gated by Google Workspace login; sites are reachable only on the corporate network/VPN.

02 — Implementation

Three small pieces over one S3 bucket

A thin CLI uploads a tarball; a control-plane API unpacks it into a versioned S3 prefix and flips a pointer; a stateless edge streams it back. That's the whole system.

CLI
tars the built folder, POSTs it with a session token
→
API
auth, claim name, unpack to files/<ver>/, flip pointer
→
S3
the only state: site record + versions + files
→
Edge
resolve version, stream bytes, SPA fallback

No database — by design

All state lives as S3 objects. The one operation needing atomicity — "first publisher owns the name" — uses S3 conditional writes (If-None-Match to claim, If-Match compare-and-swap to edit). That makes both services stateless and horizontally scalable with no RDS to run.

Atomic deploys + rollback

Each publish writes a brand-new immutable files/<version>/ tree, then atomically flips currentVersion. Readers see all-old or all-new, never a mix. Rollback is just flipping the pointer back; old versions stay in S3.

SPA-aware serving. A missing path that looks like a navigation route (/dashboard) falls back to index.html (200) so React Router deep-links work; a missing asset (.js/.css) returns a real 404 — never HTML where the browser expects JS.

Auth — server-mediated Google login

Clients hold no Google credentials. The API is the OAuth client: it runs the Google flow, verifies the email + domain (and an optional per-person allowlist), and mints its own signed session token. Clients (CLI, MCP, dashboard) only ever need the API URL.

Caching — memory for pointers, disk for bytes

Process memory holds only the tiny per-site version pointer, so it stays flat across hundreds of sites. Asset bytes go to an ephemeral local-disk cache (keyed by the immutable versioned path), where the OS page cache keeps hot files at RAM speed — no heap-OOM risk.

03 — Data model

Everything is an S3 object

Three object types per site. No tables, no migrations.

# one private bucket
sites/<name>/site.json              owner, collaborators, currentVersion   (claim + CAS)
sites/<name>/versions/<id>.json    per-publish audit (who, when, size)
sites/<name>/files/<id>/...        the served files (immutable per version)

The edge reads site.json for the live version (cached), then streams files/<currentVersion>/<path>. A publish never mutates an existing version — it writes a new one and flips the pointer.

05 — Infra requirements

Small footprint, no stateful tier

Two tiny stateless Node services + S3. No database, no static credentials. Runs on EC2, ECS, or Kubernetes (a Helm chart ships in the repo).

Kubernetes

Helm chart deploys api + edge Deployments, Services, one Ingress (api host + wildcard), ConfigMap/Secret, an IRSA ServiceAccount, optional HPA. The edge cache is a per-pod ephemeral emptyDir — scales to any replica count.

Why it's cheap

No RDS is the big saving. Fixed costs are basically the ALB + NAT; Fargate baseline for two tiny services is small; S3/ACM/Route 53 are pennies. Low hundreds/month at internal scale.

One config rule. Clients (CLI/MCP/dashboard) need only DROP_API. Everything sensitive — Google credentials, session secret, S3 role — lives only on the API/edge.
