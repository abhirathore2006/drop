Guides

Getting started

Bring up the full Drop stack locally, publish your first site, and learn the handful of make targets you'll use every day.

Drop is self-hosted static-site publishing: push a built folder, get a URL. Everything runs on your own machine for development — a local S3 (Floci), Postgres, the control-plane api, and the serving edge. This page gets you from a clean checkout to a live local site.

Run it locally

Prerequisites

  • Node 24 — the exact version is pinned in .nvmrc. Run nvm use (or nvm install) in the repo first; make setup will also install it via nvm.
  • podman — used to run Floci (S3) and Postgres as local containers. Install Podman Desktop if you don't have it.

(Bun is only needed to run the test suite — not for local dev, which runs the servers as Node processes.)

One-time setup

make setup

This ensures Node (via nvm + .nvmrc), installs npm dependencies, initializes and starts the podman VM, and pulls the Floci and Postgres images. Behind a TLS-inspecting proxy, point podman at your root CA so it can pull images:

make setup CORP_CA=~/certs/your-root-ca.cer

Start the stack

make start

This brings up everything and builds the bundles first:

ServicePortRole
Floci (S3)4566Object storage for the served file bytes (persistent named volume)
Postgres5432All metadata: users, sites, members, versions (persistent named volume)
api8473Control plane + dashboard; runs schema migrations on boot
edge8474Stateless serving edge; routes by Host header
Note

With no .env file, DROP_DEV_AUTH defaults to 1 locally — so dev-auth is on and you don't need any Google credentials to develop. If you do set the Google env vars (DROP_GOOGLE_CLIENT_ID, DROP_GOOGLE_CLIENT_SECRET, DROP_SESSION_SECRET) in .env, browser Google login is enabled even in dev-auth — and the dev sub:email tokens keep working too. Published sites and metadata persist across restarts in named volumes. See Configuration for the env knobs.

Publish your first site

Point Drop at a built folder and give it a name:

make publish DIR=./your/dist NAME=myapp

Under dev-auth, make publish first does a throwaway dev-login as alice@example.com, then publishes. View the result — by default the edge serves at:

http://myapp.drop.localhost:8474/

The .localhost TLD auto-resolves to 127.0.0.1 on macOS, so that URL just works in a browser. The edge routes purely by the Host header, so you can also curl it directly:

curl -H "Host: myapp.drop.localhost" http://localhost:8474/
Tip

If you'd rather not rely on .localhost resolution, add 127.0.0.1 myapp.drop.localhost to /etc/hosts and browse normally.

Compute quickstart (apps, databases & secrets)

Beyond static sites, Drop runs container apps, managed Postgres, and write-only secrets — all under the same names, auth, and CLI. This plane is opt-in: it lights up only when the API has a Kubernetes cluster.

The static stack above needs no cluster. To exercise apps / databases / secrets locally, bring up the compute plane — a single-node k3s (in podman) with KEDA + the HTTP add-on (scale-to-zero), CloudNativePG + the Barman Cloud Plugin (managed Postgres), and the External Secrets Operator + a floci ClusterSecretStore (so the aws secrets backend runs against Floci's Secrets-Manager emulation):

make compute-up                  # k3s + KEDA + CNPG + ESO (bash infra/local/compute-up.sh under the hood)

Run the API against the cluster

Point the API at the cluster kubeconfig with DROP_KUBECONFIG (this is what enables /v1/apps + /v1/databases; without it Drop is static-only and those endpoints return 501). For the Floci secrets path, also select the aws backend against Floci's Secrets-Manager emulation:

DROP_KUBECONFIG=~/.kube/drop-local.config DROP_SECRET_BACKEND=aws \
  DROP_SECRET_MANAGER_ENDPOINT=http://localhost:4566 DROP_SECRET_STORE_NAME=floci \
  ... node dist/api.js
Note

DROP_SECRET_BACKEND defaults to kube (writes the <app>-secret Secret directly). Setting aws with DROP_SECRET_STORE_NAME=floci exercises the same path as prod's AWS Secrets Manager + External Secrets Operator — locally against Floci. Every backend converges on the <app>-secret Secret the Deployment envFroms.

Deploy an app, a database & a secret

With the cluster up and the API running, the compute commands work locally exactly as in prod. Declare an app: section in drop.yaml (with a build: Dockerfile, no prebuilt image needed), then:

drop db create myapi-db                                  # managed Postgres (CloudNativePG, single instance)
drop deploy ./api myapi --build --no-start               # build + push through Drop, deploy — but don't boot yet
drop db password myapi-db --set-secret myapi:PGPASSWORD  # rotate the DB password straight into the app secret (never printed)
drop start myapi                                         # first boot — already has the password

--build is the simplest path: Drop builds your Dockerfile locally and streams the image through Drop so the cluster can pull it — no registry credentials on your machine, and the same command works locally and in prod (locally it imports into k3s's containerd; in prod it pushes to a registry/ECR). The builder is docker by default — set DROP_BUILDER=podman to use podman. Want just the build + push (it prints the in-cluster ref)? Run drop push ./api myapi. Already have an image? Pin app.image in drop.yaml and drop --build.

Why --no-start + --set-secret? An app with scale.min: 1 boots a pod the moment it's deployed — so deploying before its DB password exists makes that first pod crash-loop. --no-start deploys it stopped; drop db password <db> --set-secret <app>:<KEY> then rotates the password directly into the app's write-only secret (it never returns to your terminal); and drop start gives it a healthy first boot. Add --show if you also need the password printed; an app with no required secrets can skip --no-start and go live immediately.

The app goes live at https://myapi.drop.example.com (locally the edge dispatches <app>.drop.localhost to the KEDA interceptor, which wakes a scaled-to-zero pod on the first request). Lifecycle is drop restart (roll the pods) / drop stop (true offline — scale to 0 and don't wake on traffic) / drop start (resume the configured scale).

Organisations

Every site, app, and database belongs to one organisation. You get a personal org automatically — these commands target it by default — so nothing above changes. To deploy into a shared team org, create it once with drop org create <slug> (you become owner; drop org add <slug> <email> [role] grants org-wide owner/admin/member/viewer), then pass --org <slug> to drop publish / drop deploy / drop db create.

Secrets are write-only

App secrets are set / rotated / deleted but never readable back — they're injected as env vars and never touch drop.yaml/git, API/CLI/MCP responses, logs, or the metastore. Inspect with drop secrets ls myapi (key names + when changed, never values) and remove with drop secrets rm myapi DB_PASSWORD.

Runnable example apps

See examples/DATABASE_APPS.md for the end-to-end flow with several DB-backed examples — guestbook-node (Node + pg), tasks-node-ts (Node + TypeScript via tsx), blog-express (Express 5 + EJS), notes-next (Next.js), and board-tanstack (TanStack Start) — each binding to a managed Postgres via the libpq PG* env vars and a write-only PGPASSWORD secret: create a DB, set the secret, and drop deploy … --build. The same commands are also exposed as MCP tools (incl. org_create / org_list / org_add_member; image push is CLI-only for now) and in the React console — see CLI, MCP & Dashboard.

WebSockets

examples/chat-ws is a real WebSocket chat app, but WebSockets don't yet traverse the public edge — the edge proxy doesn't tunnel HTTP Upgrade requests and the KEDA HTTP add-on interceptor returns 403 for WS upgrades. Today you exercise chat-ws via kubectl port-forward straight to the pod; the public hostname serves the HTML but the browser's WebSocket connection fails until the fix lands.

Trusted local HTTPS (optional, nginx)

The make flow serves over plain http://…:<port>. For trusted HTTPS with stable names — and a setup that mirrors production — run the whole stack in containers behind nginx (works on macOS, Linux, and Windows via Docker/Podman Desktop or WSL2):

make stop                                              # free the shared host ports first
node build.mjs                                         # build the self-contained bundles
./infra/nginx/gen-certs.sh                             # local TLS cert for *.drop.localhost
docker compose -f infra/docker-compose.yml up --build  # postgres + floci + api + edge + nginx

You then get, behind nginx on :443:

  • https://api.drop.localhost/ → control plane + dashboard (and /docs/)
  • https://<name>.drop.localhost/ → any published site (edge, wildcard)

nginx terminates TLS on :443 and routes by hostname (api.drop.localhost → api, *.drop.localhost → edge), setting X-Forwarded-Host which the edge honors — the same shape as the production ingress. The images copy the prebuilt bundles (hence node build.mjs first), so there's no in-image npm install — they build offline / behind a TLS-inspecting proxy. gen-certs.sh uses mkcert for a browser-trusted cert if installed, else a self-signed one — in which case run make trust-cert to add it to your OS trust store (macOS/Linux/Windows; make untrust-cert reverses it) and clear the browser warning.

Note

This stack reuses the make start host ports (8473/8474/5432/4566) plus 443 — run make stop first. If the host can't bind 443 (rootless Docker/Podman) or it's taken, set a high one: DROP_HTTPS_PORT=8443 docker compose … up --build → https://api.drop.localhost:8443/. To trust your org's root CA for runtime egress (e.g. Google login through a corporate proxy), drop it in infra/ca/*.crt. Real Google login locally still uses the loopback URL (Google won't accept a .localhost OAuth redirect).

Publish from anywhere (one-line install)

You don't have to clone the repo to publish to a running Drop. The deployed API serves its own installer at /install.sh, templated with that instance's URL. It puts a persistent drop command on your PATH and auto-configures the API URL — no clone, no npm, no Google secrets on your machine (login is server-mediated). Needs only Node 18+:

curl -fsSL https://api.drop.example.com/install.sh | sh
drop login
drop publish ./dist myapp

Your instance's dashboard shows this command with the real API URL already filled in — copy it from there. The installer is idempotent; re-run it any time to update the CLI.

Prefer no install at all? Any machine with Node can run the CLI straight from git via npx (set DROP_API first). See CLI, MCP & Dashboard for the full command surface.

export DROP_API=https://api.drop.example.com
npx git+https://git.example.com/scm/<team>/drop.git publish ./dist myapp

Other make targets

TargetWhat it does
make statusShow whether api, edge, floci, and pg are up
make logsTail the api + edge logs
make restartStop, then start the whole stack
make loginReal Google sign-in (server-mediated); needs .env with the web-client config and DROP_DEV_AUTH=0
make stopStop api + edge + Floci + Postgres (podman VM stays up)
make stop-allSame as stop, and also stops the podman machine
make resetWipe the Floci + Postgres volumes — deletes all sites + metadata
Danger

make reset removes the persistent named volumes. Every published site and all metadata are gone. There's no undo.
