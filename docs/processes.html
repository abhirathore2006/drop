Processes, health & releases

Additions to the app: section of drop.yaml, designed together: healthcheck: (readiness/liveness probes), release: (a migration Job that gates the rollout), processes: (a web process plus long-running workers), and schedule: (a CronJob instead of a web app). All are optional — an app that declares none behaves exactly as before.

How they fit together

At drop deploy the API, in order: runs the release Job (if any) and halts on failure, then applies the Deployment(s) with the healthcheck probes attached to the web container, wiring one Service + KEDA HTTPScaledObject for the web process and a plain Deployment for each worker. The release + rollout are serialized per app under an advisory lock, so two overlapping deploys can't interleave migrations — the second returns 409.

healthcheck:

Probes on the web container. The readiness probe gates traffic (Kubernetes won't route to a pod until it passes, so rolling deploys are actually zero-downtime); the liveness probe restarts a wedged pod. Both hit the same endpoint by default.

app:
  image: myapi:1
  healthcheck:
    path: /healthz                  # required; must start with "/"
    interval: 10s                    # periodSeconds; 1–300s, default 10s
    timeout: 2s                      # timeoutSeconds; 1–60s, default 2s
    grace: 15s                       # initialDelaySeconds; 0–600s, default 15s
KeyTypePurpose
pathstringRequired. HTTP path probed on the container port. Must start with /; otherwise the whole block is ignored and the default probe (below) applies.
intervaldurationHow often to probe (periodSeconds). 1–300s, default 10s. Junk values fall back to the default.
timeoutdurationPer-probe timeout (timeoutSeconds). 1–60s, default 2s.
gracedurationStartup grace before the first probe (initialDelaySeconds). 0–600s, default 15s.
Note

With no healthcheck, Drop still attaches a TCP-socket readiness probe on the container port — better than nothing, and honest about it: a listening socket isn't proof the app is healthy, but it stops traffic reaching a pod that hasn't bound its port yet. Add an HTTP healthcheck for a real readiness signal and for liveness restarts. Probes apply to the web process only.

keep_warm

A fourth key, keep_warm: true, opts a scale-to-zero app into synthetic uptime probing — Drop otherwise won't probe an app that can scale to zero, because a probe would wake the pod. It's a pure observability toggle: it changes nothing about scaling (KEDA's min stays 0), and it doesn't require an HTTP path to be set. See Observability.

healthcheck:
  keep_warm: true

release:

A one-shot Job run before the new version rolls out — the home for database migrations. It runs with the same image, env, secrets, and uses bindings as the app (so PG* and the write-only <app>-secret are all present), backoffLimit: 0 / restartPolicy: Never — one shot, never a silent retry of a migration.

app:
  image: myapi:1
  release: "npm run migrate"            # shorthand; runs as ["/bin/sh","-c", …]
  # — or the object form —
  release:
    command: "npm run migrate"
    timeout: 5m                      # wait bound; default 5m, hard cap 15m
Halt on failure

If the release Job fails or times out, the deploy halts: the new Deployment/HTTPScaledObject is never applied, so the old version keeps serving. The drop deploy call returns an error whose body includes the tail of the Job pod's logs, and the failure is recorded in the audit trail. Fix the migration and redeploy. Read the full logs any time with drop logs <app> --release.

  • The Job is named <app>-release-<version> and is kept briefly after it finishes so you can retrieve its logs; prior release Jobs are garbage-collected on the next deploy (and on app delete).
  • A --no-start deploy skips the release phase — the point of --no-start is to set secrets first, and a migration would fail against a not-yet-configured app. Set secrets, then drop start, then redeploy to run the release.
  • Two overlapping deploys can't run migrations at once: the release + rollout hold a per-app advisory lock, so a concurrent drop deploy returns 409.

processes:

Replace the implicit single process with a named map. The web process gets today's treatment — a Service + KEDA HTTPScaledObject on your app's host; every other key is a worker: a plain Deployment with no Service and no HTTP scaling. All processes share the same image, env, secrets, and uses bindings; per-process resources override the app-level default.

app:
  image: myapp:1
  processes:
    web:    { command: "node server.js" }                 # Service + HTTPScaledObject
    worker: { command: "node worker.js",
              scale: { min: 1, max: 1 },
              resources: { cpu: "250m", memory: "256Mi" } }  # plain Deployment
KeyTypePurpose
commandstring / arrayWhat the process runs. A string runs in shell form (["/bin/sh","-c", …]); an array is passed through unchanged (exec form). Omit to use the image's own entrypoint.
scaleobject{ min, max }. For web: KEDA replica bounds (as app.scale). For a worker: static replicas — min is enforced ≥ 1 (see note) and is the running count today.
resourcesobject{ cpu, memory } limits for this process, overriding the app-level default.
scale_onobject{ queue, target } — turns this worker into a queue-scaled worker: a KEDA ScaledObject watches queue's length on the app's bound cache and scales replicas so each one carries roughly target jobs. See below. Not valid on the web process.
Rules
  • At most one web. Zero is fine — a worker-only app is legal (no Service, no host, just workers). More than one web process is rejected at deploy with a 400.
  • Workers are min ≥ 1 — unless scale_on. A plain scale-to-zero worker has no wake source (nothing routes traffic to it), so it would never run; Drop enforces at least one replica. A web process may still scale to zero, and so may a scale_on worker (see below) — the queue itself is the wake source.
  • Zero migration. Omit processes: entirely and the app is exactly one web process — identical to before.

scale_on: queue-scaled workers

Pair a worker with a cache and it scales on queue depth instead of sitting at a fixed replica count — the same primitive that backs BullMQ-style job queues, with zero new infrastructure. Under the hood: a KEDA ScaledObject with the redis-lists trigger watches the length of scale_on.queue on the app's bound Valkey and drives replicas between scale.min and scale.max so each replica carries roughly scale_on.target queued jobs.

app:
  image: myapp:1
  uses:
    - cache: jobs                       # the queue lives in this bound Valkey
  processes:
    web:    { command: "node server.js" }
    worker: { command: "node worker.js",
              scale: { min: 0, max: 10 },
              scale_on: { queue: "jobs", target: 10 } }  # ~10 queued jobs per replica
KeyTypePurpose
queuestringRequired. The name of the Valkey list (the BullMQ queue name, or whatever your worker LPUSH/BRPOPs against).
targetnumberRequired. Desired queue length per replica — KEDA computes the replica count from queueLength / target. Clamped to 1–1000.
Rules
  • Requires a {cache} binding. The app's uses must declare at least one { cache: <name> } — that's where the queue lives. Missing one is rejected at deploy with a 400.
  • Not valid on web. The web process already scales on HTTP traffic (via the KEDA HTTP add-on); scale_on there is rejected with a 400.
  • min: 0 is allowed and is the default. Unlike a plain worker, a scale_on worker has a real wake source — an empty queue means legitimately zero replicas, and KEDA scales it back up the moment something is enqueued. max defaults to 3 if unset.
  • Multiple caches: first one wins. If the app binds more than one cache, the trigger points at the first {cache} entry in uses — declare exactly one cache binding if you need to be unambiguous.
  • Password stays out of the manifest. The trigger's address (host:port) is plain — a ClusterIP name isn't sensitive — but the cache's password is wired through a KEDA TriggerAuthentication that references the cache's own write-only Secret; it's never copied anywhere.

schedule:

A 5-field cron expression that turns the app into a Kubernetes CronJob instead of a Deployment + Service + KEDA HTTPScaledObject — no listener, no host, just a container that runs to completion on a schedule.

app:
  image: myjob:1
  schedule: "0 3 * * *"                    # every day at 03:00 UTC
  command: "python nightly_report.py"  # optional; omit to use the image's entrypoint
KeyTypePurpose
schedulestringRequired to opt in. A standard 5-field cron expression (minute hour day-of-month month day-of-week) — numbers, ranges (1-5), steps (*/15), lists (0,12), and * are all accepted per field. An invalid expression is dropped silently (the app deploys as an ordinary web app instead), so double-check it round-trips through drop deploy before relying on it.
commandstring / arrayWhat the Job runs. Same convention as a process's command: a string runs in shell form, an array is exec-form passthrough. Omit to use the image's own entrypoint.
Mutually exclusive

schedule cannot be combined with processes, an explicitly-declared services, or healthcheck — a CronJob has no web/worker processes, no listener, and no probes to attach. (Simply omitting services is fine — Drop's own default single-port service doesn't count as a declaration.) Combining any of these returns a 400 at deploy.

  • One run at a time. concurrencyPolicy: Forbid — if a run is still in flight when the next fire comes around, the next one is skipped rather than piling up.
  • History. The last 3 successful and 3 failed Jobs are kept for inspection; older ones are garbage-collected automatically.
  • release: is unaffected. It still runs once per drop deploy of the CronJob object itself — not before every scheduled fire.
  • Stop/start. drop stop <app> suspends the CronJob (spec.suspend: true) — no new runs fire, though one already in flight finishes; drop start unsuspends it.
Tip

Before schedule: existed, the only way to run something on a timer was a worker process that polled forever — sometimes fronted by a dummy HTTP server just to satisfy the "must be a web app" assumption. schedule: ends that workaround: declare the cron expression and let Kubernetes own the timing.

Observing it

drop ps <app> shows one row per process — its ready/replicas, restart count, and state — aggregated across the web + worker Deployments:

$ drop ps myapp
  PROCESS   READY   RESTARTS   STATE
  web       2/2     0          Running
  worker    1/1     0          Running

drop logs <app> --release reads the latest release Job's pod logs (the migration output) instead of the app pods — the same one-shot logs endpoint, pointed at the release Job:

$ drop logs myapp --release
  running migrations…
  applied 0003_add_index
  done
Tip

Each process counts as pods against your tenant quota, same as any workload. A worker that's idle most of the time is still an always-on replica (min ≥ 1) — size it accordingly, or reach for scale_on to scale it with queue depth instead (down to zero when idle).
