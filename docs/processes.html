Processes, health & releases

Three additions to the app: section of drop.yaml, designed together: healthcheck: (readiness/liveness probes), release: (a migration Job that gates the rollout), and processes: (a web process plus long-running workers). All three are optional — an app that declares none behaves exactly as before.

How they fit together

At drop deploy the API, in order: runs the release Job (if any) and halts on failure, then applies the Deployment(s) with the healthcheck probes attached to the web container, wiring one Service + KEDA HTTPScaledObject for the web process and a plain Deployment for each worker. The release + rollout are serialized per app under an advisory lock, so two overlapping deploys can't interleave migrations — the second returns 409.

healthcheck:

Probes on the web container. The readiness probe gates traffic (Kubernetes won't route to a pod until it passes, so rolling deploys are actually zero-downtime); the liveness probe restarts a wedged pod. Both hit the same endpoint by default.

app:
  image: myapi:1
  healthcheck:
    path: /healthz                  # required; must start with "/"
    interval: 10s                    # periodSeconds; 1–300s, default 10s
    timeout: 2s                      # timeoutSeconds; 1–60s, default 2s
    grace: 15s                       # initialDelaySeconds; 0–600s, default 15s
KeyTypePurpose
pathstringRequired. HTTP path probed on the container port. Must start with /; otherwise the whole block is ignored and the default probe (below) applies.
intervaldurationHow often to probe (periodSeconds). 1–300s, default 10s. Junk values fall back to the default.
timeoutdurationPer-probe timeout (timeoutSeconds). 1–60s, default 2s.
gracedurationStartup grace before the first probe (initialDelaySeconds). 0–600s, default 15s.
Note

With no healthcheck, Drop still attaches a TCP-socket readiness probe on the container port — better than nothing, and honest about it: a listening socket isn't proof the app is healthy, but it stops traffic reaching a pod that hasn't bound its port yet. Add an HTTP healthcheck for a real readiness signal and for liveness restarts. Probes apply to the web process only.

release:

A one-shot Job run before the new version rolls out — the home for database migrations. It runs with the same image, env, secrets, and uses bindings as the app (so PG* and the write-only <app>-secret are all present), backoffLimit: 0 / restartPolicy: Never — one shot, never a silent retry of a migration.

app:
  image: myapi:1
  release: "npm run migrate"            # shorthand; runs as ["/bin/sh","-c", …]
  # — or the object form —
  release:
    command: "npm run migrate"
    timeout: 5m                      # wait bound; default 5m, hard cap 15m
Halt on failure

If the release Job fails or times out, the deploy halts: the new Deployment/HTTPScaledObject is never applied, so the old version keeps serving. The drop deploy call returns an error whose body includes the tail of the Job pod's logs, and the failure is recorded in the audit trail. Fix the migration and redeploy. Read the full logs any time with drop logs <app> --release.

  • The Job is named <app>-release-<version> and is kept briefly after it finishes so you can retrieve its logs; prior release Jobs are garbage-collected on the next deploy (and on app delete).
  • A --no-start deploy skips the release phase — the point of --no-start is to set secrets first, and a migration would fail against a not-yet-configured app. Set secrets, then drop start, then redeploy to run the release.
  • Two overlapping deploys can't run migrations at once: the release + rollout hold a per-app advisory lock, so a concurrent drop deploy returns 409.

processes:

Replace the implicit single process with a named map. The web process gets today's treatment — a Service + KEDA HTTPScaledObject on your app's host; every other key is a worker: a plain Deployment with no Service and no HTTP scaling. All processes share the same image, env, secrets, and uses bindings; per-process resources override the app-level default.

app:
  image: myapp:1
  processes:
    web:    { command: "node server.js" }                 # Service + HTTPScaledObject
    worker: { command: "node worker.js",
              scale: { min: 1, max: 1 },
              resources: { cpu: "250m", memory: "256Mi" } }  # plain Deployment
KeyTypePurpose
commandstring / arrayWhat the process runs. A string runs in shell form (["/bin/sh","-c", …]); an array is passed through unchanged (exec form). Omit to use the image's own entrypoint.
scaleobject{ min, max }. For web: KEDA replica bounds (as app.scale). For a worker: static replicas — min is enforced ≥ 1 (see note) and is the running count today.
resourcesobject{ cpu, memory } limits for this process, overriding the app-level default.
scale_onobjectReserved. { queue, target } is accepted and round-tripped for forward compatibility but ignored today; queue-scaled workers (KEDA on a Valkey list) land in a later slice.
Rules
  • At most one web. Zero is fine — a worker-only app is legal (no Service, no host, just workers). More than one web process is rejected at deploy with a 400.
  • Workers are min ≥ 1. A scale-to-zero worker has no wake source (nothing routes traffic to it), so it would never run; Drop enforces at least one replica. A web process may still scale to zero.
  • Zero migration. Omit processes: entirely and the app is exactly one web process — identical to before.

Observing it

drop ps <app> shows one row per process — its ready/replicas, restart count, and state — aggregated across the web + worker Deployments:

$ drop ps myapp
  PROCESS   READY   RESTARTS   STATE
  web       2/2     0          Running
  worker    1/1     0          Running

drop logs <app> --release reads the latest release Job's pod logs (the migration output) instead of the app pods — the same one-shot logs endpoint, pointed at the release Job:

$ drop logs myapp --release
  running migrations…
  applied 0003_add_index
  done
Tip

Each process counts as pods against your tenant quota, same as any workload. A worker that's idle most of the time is still an always-on replica (min ≥ 1) — size it accordingly, or wait for queue-scaled workers.
