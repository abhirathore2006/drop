Guides

Stacks

A stack is a declarative, multi-resource graph — one stack: section in drop.yaml that describes a group of databases, apps and sites and the edges that wire them together. drop up reconciles it: it diffs your spec against what's live and applies an ordered plan. It replaces the three-command imperative dance (create a DB, deploy an app, publish a site, and wire the env by hand) with one desired-state file.

Note

A stack is grouping + desired state, not a new resource kind. Every resource in a stack is an ordinary database / app / site row — every existing route, role and dashboard page keeps working on it. You can still drop deploy, drop db create or drop publish a stack's resources individually.

The stack: section

A stack has a name and a map of resources (≤ 16 in v1). Each resource has a type (database, app or site) and type-specific fields — the same fields the standalone app: / database: sections accept.

stack:
  name: myproduct
  resources:
    db:  { type: database, storage: 1Gi }
    api:
      type: app
      dir: ./api                # build context → built + pushed through Drop at `up`
      uses: [{ database: db }]  # app→database edge — references the resource KEY `db`
      expose: { tcp: false }    # A2 TCP opt-in (parsed + stored now)
      env: { NODE_ENV: production }
    web:
      type: site
      dir: ./web/dist
      env_from: [{ resource: api, output: url, as: API_BASE }]  # site→app edge

Resource names

A resource with key k in stack s materializes as the site name s-k — so db above becomes myproduct-db, reachable at myproduct-db-rw for Postgres. Set an explicit name: on a resource to override that (e.g. to adopt an existing database). The mapping (resource_key → site_name) is recorded so a rename in your spec never silently forks a second copy.

Edges

Edges reference resource keys within the same stack; the reconciler resolves each key to its site name. The edge kinds:

EdgeDeclared onFieldWhat it does
app → databasean appuses: [{ database: <key> }]The first-class DB binding: injects the CNPG <db>-app secret via envFrom, mounts the cluster CA, and sets PGSSLMODE=verify-full. The <key> is resolved to the materialized DB name.
app → appan appuses: [{ app: <key> }]Service discovery (see below): injects a plain <KEY>_URL env pointing at the peer app's in-cluster Service — or, when the peer scales to zero, at its public host so a call wakes it.
site → appa siteenv_from: [{ resource, output: url, as }]Publish-time substitution: the referenced app's url replaces the ${as} placeholder in the site's text files (.html/.js/.css/.json/…) as the tarball is packed — client-side. The server validates the edge and stores it; it never rewrites bytes. A static site has no runtime env, so a site consumes an app only this way — never via uses.

(An app can also uses-bind a bucket, cache, or auth resource — same shape, credential injection via the write-only secret path.)

Edges define the apply order: databases before the apps that use them, apps before the sites that read them, and a used app before the app that calls it. A cycle is rejected with a clear error — there's no valid order.

app → app: service discovery

Declaring uses: [{ app: <key> }] on an app injects an env var named <KEY>_URL (the referenced resource key, uppercased) into the consumer — so uses: [{ app: api }] gives the consumer API_URL. Multiple app uses inject multiple <KEY>_URL vars. It's a plain, non-secret container env (a Service URL carries nothing sensitive), resolved fresh at every deploy, not stored in the secret path.

What the URL points at depends on the target's scale floor:

  • Always-on target (scale.min ≥ 1) → its in-cluster Service, http://<name>.<ns>.svc.cluster.local:80. This is a direct, lowest-latency hop, allowed by the tenant's default intra-namespace network policy.
  • Scale-to-zero target (scale.min = 0) → its public host, https://<name>.<baseDomain>. There's no pod to dial directly, so the call hairpins out through the edge → KEDA HTTP interceptor (the same wake path external traffic takes), which scales the target up on first hit. Added latency: the first call after idle pays an external round-trip plus a cold start; steady-state calls are normal edge-proxied requests. If a peer is on a hot path, give it scale: { min: 1 } to keep the direct in-cluster URL.
Same org only

An app→app edge may only target another app in the same organisation (which shares the namespace). A cross-org reference is refused (400) — the tenant isolation boundary is a feature, not a hole to punch. Within a stack every resource already shares the namespace, so this is automatic; a direct drop deploy that names a peer in another org (or another namespace) is rejected up front. Mutually-referencing apps (a uses b and b uses a) are rejected as a cycle in v1.

drop up

drop up [dir] reads the stack: section of dir/drop.yaml (default .), shows the plan, and applies it. The split of responsibility: the server owns existence + config (claim, DB create, app deploy, config updates, flagged deletes); the CLI owns content (it builds & pushes each app image and publishes each site's bytes).

$ drop up ./myproduct
  ACTION  KIND      KEY  RESOURCE        REASON
  create  database  db   myproduct-db    not present — will create
  create  app       api  myproduct-api   not present — will create
  create  site      web  myproduct-web   not present — will create
  ▸ building image for api (…/api)…
  ▸ applying stack myproduct…
  ▸ publishing site web → myproduct-web…
  ✓ stack myproduct reconciled (spec v1)
ActionMeaning
createThe resource isn't live yet — claim it and apply.
updateThe resource exists; re-apply its (possibly changed) config. Idempotent — re-applying converges.
noopLive and unchanged vs the last applied spec. Skipped.
deletePreviously managed, now gone from the spec. Flagged only unless you pass --prune (see below).
Flags
  • --dry-run — print the plan and stop. Applies nothing. (This is the same endpoint the canvas and agent tools use to preview changes.)
  • --prune — actually execute the delete steps. Without it, a resource dropped from the spec is flagged in the plan but left running, so removing a key never silently destroys data.
  • --org <slug> — reconcile into a team organisation (default: your personal org).
Halt on failure

A stack up runs under a per-stack advisory lock (a concurrent up returns 409) and halts on the first failing step — the steps that already applied stay applied, and a retry converges (server-side-apply mental model). The desired spec is committed only on a fully successful run, so a partial failure re-plans correctly on retry.

Inspecting & removing stacks

$ drop stack ls                    # your stacks across orgs
$ drop stack status myproduct      # the spec + each resource's live status
$ drop stack rm myproduct          # delete the stack, ORPHAN its resources (they keep running)
$ drop stack rm myproduct --cascade # delete the stack AND tear down its resources

drop stack rm defaults to orphaning: it removes the stack grouping but leaves the databases/apps/sites in place (the response records which were orphaned). --cascade tears each resource down using the same delete path as drop rm. Both are audited (stack.up, stack.delete).

drop detect

drop detect [dir] proposes a stack: section from what's already on disk — no server call, purely local heuristics, so it works before you've ever logged in. It's the fastest way to bootstrap a stack for an existing repo instead of hand-writing the resources:

SignalProposes
a Dockerfile (any casing) in a directoryan app resource, dir: that directory
a prisma/ dir; pg/postgres/postgres.js/drizzle-orm in package.json deps; PG*/DATABASE_URL in .env.examplea database resource + uses: [{ database }] on that directory's app
ioredis/redis/bullmq in package.json deps; REDIS_URL in .env.examplea cache resource + uses: [{ cache }] on that directory's app
a package.json build script mentioning dist/out/build, or an existing dist/index.htmla site resource (no Dockerfile in the same directory — a Dockerfile always wins)
package.json workspaces (resolved one level; members must have their own package.json/Dockerfile)one resource per workspace member, keyed from its sanitized dirname, each analyzed with the same rules above

Deterministic — the same directory always proposes the same spec (sorted, stable resource keys). It only ever proposes what it found real evidence for: scale, resources, env, healthchecks, and everything else are left for you to add by hand.

$ drop detect ./guestbook-node
stack:
  name: guestbook-node
  resources:
    app:
      type: app
      dir: .
      uses: [{ database: db }]
    db:
      type: database
  · .: Dockerfile → app resource (dir: .)
  · .: "pg" in package.json deps → database resource + uses binding on app
  tip: pass --write to merge this into ./guestbook-node/drop.yaml

--write merges the proposal into <dir>/drop.yaml (creating the file if it doesn't exist yet); it refuses to clobber an existing stack: section unless you also pass --force. Review the printed proposal — and its resources' fixture-suite regression coverage (every example under examples/ is checked against an exact expected spec) — before trusting it on a real repo.

Agents & the console

The MCP tools mirror the CLI with an agent-safe shape: detect (local heuristics → a proposed spec, as JSON — review before applying), stack_plan (dry-run — show the plan before touching anything), then stack_up to apply, and stack_status to inspect. The dry-run plan is the same one the read-only canvas renders as a pending-changes overlay, so an agent, the CLI and the console all reconcile through one endpoint and can never diverge.
