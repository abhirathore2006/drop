Guides

CLI, MCP & Dashboard

Three ways to drive Drop: the drop command line, an MCP server for AI clients, and a browser dashboard. All three share one identity and the same /v1/* endpoints.

TERMINAL

CLI

Run drop via npx from git, or install a persistent command. Best for scripts and CI.

AI CLIENT

MCP server

Publish and manage in natural language from any MCP client — "publish ./dist as myapp".

BROWSER

Dashboard

Sign in with Google to list sites, roll back, manage collaborators, transfer, and delete.

Note

Publishing happens in the CLI or MCP. The dashboard manages existing sites but does not upload new ones.

CLI

The CLI is bin/drop.ts, bundled to dist/drop.js. It targets *.drop.example.com and stores its session in ~/.config/drop/session.json — the same file the MCP server uses.

Install options

Option 1 — one-line install from your Drop server (recommended). The deployed API serves its own installer at /install.sh, already templated with that instance's URL. It downloads the CLI bundles the same API serves, drops a drop command on your PATH (no sudo, no git, no npm), and auto-configures the API URL — so drop login and drop publish work immediately. It also records where it installed from, so later you just run drop update (which re-runs this installer). Idempotent; re-run any time to update. Needs only Node 18+.

curl -fsSL https://api.drop.example.com/install.sh | sh
drop login
drop publish ./dist myapp

The dashboard for your instance shows this exact command with your real API URL filled in — copy it from there.

Option 2 — npx straight from git (no install). npx clones the repo, runs the prepare step (esbuild bundles both dist/drop.js and dist/mcp.js), and runs it. Good for one-off CI invocations.

npx git+https://git.example.com/scm/<team>/drop.git publish ./dist myapp
# pin a tag/branch:  npx git+https://…/drop.git#v0.1.0 ls
# set the API once:  export DROP_API=https://api.drop.example.com

Auth

drop login signs in with Google through the Drop server — the API owns the Google credentials and hands back a session token, so no Google client id/secret lives on your machine. Clients need only DROP_API (or pass --api <url>); it defaults to https://api.drop.example.com. For local work against make start, drop dev-login <sub> <email> saves a session without Google, and requires DROP_DEV_AUTH=1 on the API.

drop exec — an authenticated shell into a running app

drop exec <app> [-- cmd…] gives you a live shell in the app's container without cluster credentials: no kubectl exec, no bypass of Drop's authz. Each session mints a single-use, 60-second, user+app+command-bound ticket, upgrades a WebSocket to the API, which authorizes it against the exec permission, records an app.exec audit event including the exact command, opens the pod's exec stream, and bridges your terminal to it. When your terminal is a TTY it goes raw (so Ctrl-C and arrow keys pass through to the remote shell, and window resizes forward); a piped stdin is streamed for scripting. The command you run is bound into the ticket at issuance, so a redeemed upgrade can never run a different command than the one you were authorized for. Idle sessions and a per-user concurrency cap are enforced. Databases are excluded — a shell on the managed-Postgres operand pod is an operator action, not a tenant one; use drop db proxy for authenticated psql.

Warn

A shell can read the container's environment, so exec exposes an app's write-only secrets. An app's injected credentials (its DATABASE_PASSWORD, API keys, and friends) are environment variables to the process, so anyone who can exec can env them out. This is stricter framing than logs: logs merely might echo a secret a process happens to print, whereas a shell hands the whole environment over on demand. For that reason exec is an editor+ permission (never a viewer), it is never granted to a static-site or database, and every session is audited with its command. Grant it deliberately.

No MCP tool

drop exec is CLI-only — there is deliberately no MCP tool for it, the same posture as the drop db proxy tunnel: an interactive raw-byte terminal session is not a natural request/response MCP surface, and the secret-exposure framing above means it should be an explicit human action, not something an agent reaches for. Service tokens can still drive it non-interactively via the exec:<app> scope.

Organisations

Every resource belongs to exactly one organisation. You always have a personal org (auto-created on first login, reusing your existing namespace), so commands without --org behave exactly as before. Create a team org with drop org create <slug>, then pass --org <slug> to publish/deploy/db create to put new resources there. --org appears exactly where an org is a choice: a create target (those commands), a list filter (drop ls --org <slug>), and a re-home target (drop transfer <name> --org <slug>). Per-resource commands (share, secrets, restart, rollback, …) take a globally-unique resource name that already implies its org, so they don't need --org; org-wide access is granted with drop org add. Org roles apply org-wide — see Roles & Visibility.

Image push

drop deploy <dir> --build builds the Dockerfile locally and streams a docker save tarball to Drop, which makes it pullable by the cluster (locally imported into the k3s node's containerd; in prod pushed to a registry/ECR), then deploys — same command everywhere, no registry credentials on the client. drop push <dir> [name] does just the build+push and prints the in-cluster ref (locally drop.local/<app>:<tag>). The builder is docker by default; set DROP_BUILDER=podman to use podman. To build a specific Dockerfile (e.g. multiple per-env files), pass -f, --dockerfile <path> (relative to your CWD, like docker's -f; it implies --build). A prebuilt app.image in drop.yaml (without --build) still works unchanged. Image push is CLI-only for now — there is no MCP tool yet.

MCP server

Drop ships an MCP server, so any MCP client (Claude Code/Desktop, Cursor, …) can publish and manage sites in natural language. Use the drop-mcp command the installer puts on your PATH, or launch it on demand via npx — both are shown below.

Option A — installed drop-mcp (recommended). The served installer (curl … /install.sh | sh) puts drop-mcp on your PATH alongside drop. Point your MCP config (e.g. Claude Code .mcp.json) straight at it:

{
  "mcpServers": {
    "drop": {
      "command": "drop-mcp",
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}

Option B — npx (no install). The client launches the server on demand straight from git — the prepare step on clone builds dist/mcp.js, which is what makes npx … drop-mcp work without a separate build:

{
  "mcpServers": {
    "drop": {
      "command": "npx",
      "args": ["-y", "--package", "git+https://git.example.com/scm/<team>/drop.git", "drop-mcp"],
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}
Note

The client needs only DROP_API — login is server-mediated, so no Google client id/secret lives on anyone's machine. The server reads/writes the same ~/.config/drop/session.json as the CLI, so login once and both work.

Dashboard

The control-plane API serves a React console at its root — open the API URL in a browser (locally http://localhost:8473/). Click Sign in with Google (server-mediated, sets an HttpOnly cookie), then you get your workloads grouped by type (sites · apps · databases) with live status. Each workload opens as its own full, deep-linkable page (/app/<name>, /database/<name>, /site/<name>, plus /admin) with detailed sections to:

  • sites — roll back versions, set visibility;
  • apps — see image/scale/status, view logs, manage write-only secrets, and drive lifecycle (restart / stop / start);
  • databases — see the connection ref and set/rotate the password;
  • and for all — add/remove collaborators, transfer ownership, delete.

It uses the same /v1/* endpoints and identity as the CLI and MCP. Platform admins get an extra all workloads view with type/owner filters and user suspension.

Reminder

Publishing/deploying stays in the CLI and MCP — the dashboard manages existing workloads but does not upload or build them. See examples/DATABASE_APPS.md for the app + database + secret walkthrough.
