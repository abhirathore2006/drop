Guides

CLI, MCP & Dashboard

Three ways to drive Drop: the drop command line, an MCP server for AI clients, and a browser dashboard. All three share one identity and the same /v1/* endpoints.

TERMINAL

CLI

Run drop via npx from git, or install a persistent command. Best for scripts and CI.

AI CLIENT

MCP server

Publish and manage in natural language from any MCP client — "publish ./dist as myapp".

BROWSER

Dashboard

Sign in with Google to list sites, roll back, manage collaborators, transfer, and delete.

Note

Publishing happens in the CLI or MCP. The dashboard manages existing sites but does not upload new ones.

CLI

The CLI is bin/drop.ts, bundled to dist/drop.js. It targets *.drop.example.com and stores its session in ~/.config/drop/session.json — the same file the MCP server uses.

Install options

Option 1 — one-line install from your Drop server (recommended). The deployed API serves its own installer at /install.sh, already templated with that instance's URL. It downloads the CLI bundles the same API serves, drops a drop command on your PATH (no sudo, no git, no npm), and auto-configures the API URL — so drop login and drop publish work immediately. Idempotent; re-run any time to update. Needs only Node 18+.

curl -fsSL https://api.drop.example.com/install.sh | sh
drop login
drop publish ./dist myapp

The dashboard for your instance shows this exact command with your real API URL filled in — copy it from there.

Option 2 — npx straight from git (no install). npx clones the repo, runs the prepare step (esbuild bundles both dist/drop.js and dist/mcp.js), and runs it. Good for one-off CI invocations.

npx git+https://git.example.com/scm/<team>/drop.git publish ./dist myapp
# pin a tag/branch:  npx git+https://…/drop.git#v0.1.0 ls
# set the API once:  export DROP_API=https://api.drop.example.com

Auth

drop login signs in with Google through the Drop server — the API owns the Google credentials and hands back a session token, so no Google client id/secret lives on your machine. Clients need only DROP_API (or pass --api <url>); it defaults to https://api.drop.example.com. For local work against make start, drop dev-login <sub> <email> saves a session without Google, and requires DROP_DEV_AUTH=1 on the API.

MCP server

Drop ships an MCP server, so any MCP client (Claude Code/Desktop, Cursor, …) can publish and manage sites in natural language. Use the drop-mcp command the installer puts on your PATH, or launch it on demand via npx — both are shown below.

Option A — installed drop-mcp (recommended). The served installer (curl … /install.sh | sh) puts drop-mcp on your PATH alongside drop. Point your MCP config (e.g. Claude Code .mcp.json) straight at it:

{
  "mcpServers": {
    "drop": {
      "command": "drop-mcp",
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}

Option B — npx (no install). The client launches the server on demand straight from git — the prepare step on clone builds dist/mcp.js, which is what makes npx … drop-mcp work without a separate build:

{
  "mcpServers": {
    "drop": {
      "command": "npx",
      "args": ["-y", "--package", "git+https://git.example.com/scm/<team>/drop.git", "drop-mcp"],
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}
Note

The client needs only DROP_API — login is server-mediated, so no Google client id/secret lives on anyone's machine. The server reads/writes the same ~/.config/drop/session.json as the CLI, so login once and both work.

Dashboard

The control-plane API serves a React console at its root — open the API URL in a browser (locally http://localhost:8473/). Click Sign in with Google (server-mediated, sets an HttpOnly cookie), then you get your workloads grouped by type (sites · apps · databases) with live status, and a per-workload drawer to:

  • sites — roll back versions, set visibility;
  • apps — see image/scale/status, view logs, manage write-only secrets, and drive lifecycle (restart / stop / start);
  • databases — see the connection ref and set/rotate the password;
  • and for all — add/remove collaborators, transfer ownership, delete.

It uses the same /v1/* endpoints and identity as the CLI and MCP. Platform admins get an extra all workloads view with type/owner filters and user suspension.

Reminder

Publishing/deploying stays in the CLI and MCP — the dashboard manages existing workloads but does not upload or build them. See examples/DATABASE_APPS.md for the app + database + secret walkthrough.
