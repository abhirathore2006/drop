Guides

Environments

An environment is a named, durable instantiation of one stack — staging and prod from a single drop.yaml, each with its own per-environment variables. Every stack already has an unnamed default environment (the one that exists the moment you first drop up); named environments are additional, parallel copies that live alongside it. Where a preview is an ephemeral branch that expires, an environment is a permanent parallel instantiation you promote between.

The model

A stack spec instantiates into environments the same way a template instantiates into a stack — it's the same variable machinery (${var.x}) all the way down. Each environment gets:

  • a row in environments(stack_id, name, variables, …) holding its variable overlay;
  • its own resource namespace — resources materialize as <stack>-<env>-<key> (see naming);
  • a full up run with ${var.x} resolved from its variables.

The default environment keeps today's names (<stack>-<key>) and needs zero migration — an existing stack simply is its default environment.

Preview or environment?

Both give you a parallel copy of your app — they solve different problems. Reach for the right one:

PreviewEnvironment
LifetimeEphemeral — expires (default 7 d), swept automaticallyDurable — lives until you delete it
ShapeA single label-named branch of one workloadA full parallel instantiation of the whole stack
Hostname<name>--<label> (double dash)<stack>-<env>-<key> (single dash)
VariablesNone — mirrors the parent's configPer-environment ${var.x} overlay
DatabaseReuses the parent's (opt-in empty clone)Its own, provisioned by the stack spec
Typical useReview a pull request, share a WIPRun staging / prod, promote releases
Commanddrop deploy … --preview pr-42drop env create staging …

Rule of thumb: if it should disappear on its own, it's a preview; if it's a place you keep running and ship to, it's an environment.

Creating an environment

An environment stores a variable overlay: values for the ${var.x} placeholders in your drop.yaml. Provide them with repeatable --set key=value:

# the stack already exists (a plain `drop up` created its default env)
$ drop env create staging --stack shop --set db_size=512Mi --set region=eu
  ✓ environment staging created for stack shop
  reconcile it:  drop up --env staging

$ drop env ls shop
$ drop env rm shop staging --cascade   # --cascade also tears down its resources

Variables & the overlay

Your drop.yaml stack spec can carry ${var.x} placeholders anywhere a value goes — an app env: value, a database storage:, an image tag. At up time the placeholders are substituted from the target environment's stored variables:

stack:
  name: shop
  resources:
    db:
      type: database
      storage: "${var.db_size}"      # 512Mi in staging, 20Gi in prod
    api:
      type: app
      uses: [{ database: db }]
      env:
        REGION: "${var.region}"
$ drop up --env staging     # db → 512Mi, REGION → eu
$ drop up --env prod        # db → 20Gi,  REGION → us  (prod's own overlay)

A referenced variable with no value in the target environment is a hard, clear error — missing required variable(s) for env "staging": db_size — surfaced before anything touches the cluster. The default environment carries no variables, so a placeholder-bearing spec is instantiated through named environments.

Promotion

Promotion moves a tested spec from one environment to another. It re-runs the target's up with the source's currently-applied spec — and because container images are pinned refs, the exact artifact your source environment is running lands in the target, byte-for-byte. It is not a variable copy:

$ drop env promote shop staging prod
  ✓ promoted staging → prod for stack shop
  • Spec, not variables. The target keeps its own variables; it receives the source's resolved spec shape. You promote what staging is running, not what staging is configured with.
  • Exact images. Each app is pinned to the precise image the source environment is serving, so prod runs the identical build you validated in staging — never a fresh rebuild that could drift.

Naming — single dash, not --

Named-environment resources materialize as <stack>-<env>-<key> with single dashes: stack shop, env staging, resource db → shop-staging-db. The default environment keeps the original <stack>-<key> (shop-db).

The double dash -- is reserved for preview hostnames (<name>--<label>) and is rejected by site-name validation, so environments deliberately use a single dash to stay ordinary, valid, collision-free site names. Environment names are short DNS labels; default is reserved (it's the unnamed default env) and -- is not allowed inside a name.

In the console

The stack page carries an environment switcher in its header — a dropdown listing default plus every named environment. Switching re-scopes the whole page (the live canvas, the per-resource metric chips, and the pending-changes drawer) to that environment's resources. A small form there creates a new environment (name + optional variables) inline.

Quota — environments are not free

Each environment's resources are ordinary workloads: a staging and a prod copy of a three-resource stack is six workloads, and every one of them counts against your organisation's workload cap and storage budget exactly like any other. Environments are a first-class way to run parallel copies — they are not a way to run them for free. Delete an environment (drop env rm … --cascade) to reclaim its share.
