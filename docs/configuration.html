Guides

Configuration — drop.yaml

A single optional drop.yaml at your build root configures a deploy. It has three independent sections: site: (static-site routing/headers/CORS/auth), app: (a container app), and database: (a managed Postgres). A bundle uses whichever section fits what it deploys.

How it works

Put a drop.yaml at the root of your build/app folder (the same directory you publish or deploy — e.g. ./dist). The static site: config is applied by the edge on every request; the app: and database: sections drive the compute plane (see App & database config below). At publish/deploy time the API parses and sanitizes it; it is not served as a file, so visitors can never fetch it back. The edge then applies the parsed site: config on every request — resolving clean URLs, running redirects, merging headers, answering CORS, and challenging for basic auth.

The config is versioned with the deploy. Each publish can ship a different drop.yaml, and a rollback restores the exact config that shipped with that version — routing rules and all.

Note

Parsing is lenient: invalid JSON is rejected outright, but unknown fields and malformed entries are silently ignored rather than failing the publish. List fields are capped at 500 entries.

A complete example

Every key is optional. This annotated example uses all of them at once.

site:                                  # static-site config (app: / database: are documented below)
  name: my-site                        # target site — lets `drop publish ./dist` skip the name arg
  spaFallback: index.html              # doc served on navigation misses; false to disable
  cleanUrls: true                        # /about → try /about.html
  notFound: 404.html                   # custom 404 document (served with status 404)
  redirects:
    - { from: /old, to: /new, status: 301 }
    - { from: "/docs/*", to: /help/:splat, status: 302 }
  headers:
    - { source: "/assets/*", headers: { cache-control: "public, max-age=31536000, immutable" } }
    - { source: "/*", headers: { x-frame-options: DENY, x-content-type-options: nosniff } }
  cors: { allowOrigins: ["https://app.example.com"], allowMethods: [GET, HEAD], maxAge: 600 }
  basicAuth:
    realm: Staging
    users: { team: "sha256:2bb80d537b1da3e38bd30361aa855686bde0eacd7162fef6a25fe97bf527a25b" }

Key reference

KeyTypePurpose
namestringSelf-identifying target site (≤ 63 chars, valid site name).
spaFallbackstring · falseDoc served on navigation misses. Default "index.html"; false disables.
cleanUrlsbooleanResolve extensionless paths to their .html file.
notFoundstringCustom document served with a 404 status.
redirectsarrayOrdered {from, to, status} rules with glob + :splat.
headersarray{source, headers} rules matched by path glob.

name

Lets the bundle identify itself. drop publish ./dist (no name argument) uses drop.yaml's site.name; with neither, a friendly name is generated (e.g. twilight-cherry-8f3a). The server still validates that you own that name, and rejects a bundle whose name contradicts the explicit publish target. It must pass site-name validation (≤ 63 chars) or it is ignored.

{ "name": "my-site" }

spaFallback

For single-page apps: when a navigation request matches no file, serve this document so client-side routing can take over. Defaults to "index.html". Set to false to disable it entirely (e.g. a multi-page site that wants real 404s).

{ "spaFallback": false }

cleanUrls

When true, a request for /about tries /about.html. Lets you link to extensionless paths.

{ "cleanUrls": true }

notFound

The document to serve (with an actual 404 status) when nothing else matches. Pairs naturally with spaFallback: false on multi-page sites.

{ "notFound": "404.html" }

redirects

An ordered list of { from, to, status? } rules; the first match wins. A trailing * in from is a glob whose captured tail can be substituted into to via :splat. status must be a 3xx code (300–399); anything else falls back to 301.

{
  "redirects": [
    { "from": "/old", "to": "/new", "status": 301 },
    { "from": "/docs/*", "to": "/help/:splat", "status": 302 }
  ]
}

headers

A list of { source, headers } rules. source is a path glob (a trailing * matches any tail); headers is a map of header name → value. Every rule whose glob matches the request path is applied, in order — so a broad /* rule and a specific /assets/* rule can both contribute. Header names are lower-cased. This is also how you set cache-control.

{
  "headers": [
    { "source": "/assets/*", "headers": { "cache-control": "public, max-age=31536000, immutable" } }
  ]
}

cors

Set to true for permissive defaults (allow any origin, methods GET, HEAD, OPTIONS), or pass an object to tune it:

FieldTypeDefault / notes
allowOriginsstring[]["*"]. Use explicit origins to restrict.
allowMethodsstring[]["GET","HEAD","OPTIONS"]
allowHeadersstring[]Sent as access-control-allow-headers when set.
exposeHeadersstring[]Sent as access-control-expose-headers when set.
maxAgenumberPreflight cache seconds.
credentialsbooleanWhen true with a * origin, the request origin is echoed back.
 
{
  "cors": {
    "allowOrigins": ["https://app.example.com"],
    "allowMethods": ["GET", "HEAD"],
    "maxAge": 600
  }
}
Tip

With explicit origins, a Vary: Origin header is added automatically so caches don't leak the wrong allow-origin between callers.

basicAuth

Gate the whole site behind HTTP Basic auth. users maps a username to a password, stored either as plaintext ("plain") or hashed as "sha256:<hex>". An optional realm is shown in the browser prompt. Comparisons are constant-time.

{
  "basicAuth": {
    "realm": "Staging",
    "users": {
      "team": "sha256:2bb80d537b1da3e38bd30361aa855686bde0eacd7162fef6a25fe97bf527a25b",
      "guest": "plain"
    }
  }
}
Note

Publishing a drop.yaml with basicAuth — like setting visibility=password — makes the edge require an Authorization: Basic header before serving anything. See Roles & Visibility for how this interacts with org-level visibility modes.

App & database config

Alongside site:, a drop.yaml can declare an app: section (a container app) or a database: section (a managed Postgres). These drive the compute plane, so they only take effect when it is enabled (DROP_KUBECONFIG set — see below); against a static-only instance the app/database routes return 501. The three sections are independent — a bundle uses whichever fits what it ships.

Note

Runnable end-to-end walkthroughs (build an image, create a DB, wire a secret, deploy) live in the repo's examples/ — start with examples/DATABASE_APPS.md and the per-example drop.yaml files. See also Getting started §Compute.

app:

Declared under app:, parsed at drop deploy. The API translates it into a Deployment + Service + KEDA HTTPScaledObject + a default-deny NetworkPolicy in your tenant namespace. Every key but image is optional.

app:
  name: myapi                          # optional; `drop deploy ./dir myapi` overrides it
  image: myapi:1                        # required — OR run `drop deploy --build` to build ./Dockerfile and push it through Drop
  services:
    - { internal_port: 8080 }              # the port your app listens on (v1: exactly one HTTP service)
  scale: { min: 0, max: 3 }              # KEDA replicas; min 0 = scale-to-zero when idle
  resources: { cpu: "250m", memory: "256Mi" }
  env:                                  # NON-secret config only (secrets: `drop secrets set`, never in drop.yaml)
    PGHOST: mydb-rw
    PGSSLMODE: require
  trusted: true                          # default true; false → run under the gVisor sandbox
KeyTypePurpose
imagestringRequired. Container image reference. Skip a prebuilt image and pass drop deploy --build to build the folder's Dockerfile and stream it through Drop (no registry creds).
namestringSelf-identifying app name (≤ 63 chars). A name argument to drop deploy overrides it; a contradicting name is rejected.
servicesarray{ internal_port, protocol? }. v1: exactly one http service. Default [{ internal_port: 8080, protocol: http }]; protocol: tcp / multi-port is not yet supported.
scaleobject{ min, max } KEDA replica bounds (min ≥ 0, max ≥ 1, max ≥ min). min: 0 = scale-to-zero. Default { min: 0, max: 3 }.
resourcesobject{ cpu, memory } container limits (k8s quantities, e.g. "500m", "512Mi"). Default { cpu: "0.5", memory: "512Mi" } — never unbounded.
envobjectString→string environment variables. Non-secret only — put credentials in write-only secrets (drop secrets set), which are injected as env at runtime.
usesarray{ database } entries binding managed databases in the same org (see uses:). Each injects the DB's <db>-app Secret (PG* env incl. password), mounts the cluster CA, and sets PGSSLMODE=verify-full — no password ever copied into the app. A missing or cross-org database is rejected at deploy with a 400.
healthcheckobject{ path, interval?, timeout?, grace? } → readiness + liveness probes on the web container (readiness gates traffic; liveness restarts a wedged pod). Absent → a TCP-socket readiness probe on the container port. See Processes, health & releases.
releasestring / object"npm run migrate" or { command, timeout? } → a migration Job run before the new version rolls out; a failure halts the deploy (the old version keeps serving). See Processes, health & releases.
processesobjectA map of named processes replacing the implicit single process — one web (Service + HTTPScaledObject) plus plain-Deployment workers. Absent → today's single-process app. See Processes, health & releases.
trustedbooleanDefault true. Set false to run the pod under the gVisor RuntimeClass (stronger isolation for untrusted images).
Tip

An app with scale.min: 1 boots a pod the instant it deploys — so if it needs a secret (e.g. a DB password) to start, deploy with --no-start, set the secret, then drop start. See Getting started.

uses:

Declare a dependency on a managed database and let Drop wire the connection for you — no reading the Secret, building a connection string, or copying the DB password into the app's own secret. Each entry names a database your org owns; at drop deploy the API resolves it and binds it. A database that doesn't exist, or that belongs to a different org, is rejected with a 400 naming it.

app:
  image: todo:1
  uses:
    - { database: tododb }              # a `drop db create tododb` in the same org
    - { database: tododb, via: pooler } # route PGHOST through the DB's connection pooler (see database: below)
    - { bucket: avatars }            # a bucket in the same org → S3_* env (see Buckets)
    - { cache: sessions }           # a cache in the same org → REDIS_URL env (see Cache)

For each bound database, Drop injects into the app's pod:

  • envFrom the CNPG-generated <db>-app Secret — so PGHOST/PGUSER/PGPASSWORD/PGDATABASE come straight from the database. The password is never copied into the app's own secret, so rotation just works (the pod re-reads on restart).
  • the cluster CA (the <db>-ca Secret, ca.crt only — never the CA key) mounted read-only at /var/run/drop/db-ca/<db>/ca.crt.
  • PGSSLMODE=verify-full and PGSSLROOTCERT pointed at that CA — so the app verifies the server's TLS certificate against it instead of trusting the connection blindly.
  • with via: pooler, a PGHOST override pointing at the database's connection pooler Service (<db>-pooler-rw) instead of the primary. It's a container env var, so it wins over the <db>-app value; enable the pooler on the database first (drop db pooler enable).

uses also binds non-database resources: bucket: injects S3_* credentials (see Buckets) and cache: injects REDIS_URL (see Cache) — both through the write-only secret path, both same-org-only. Multiple bindings of a kind get resource-name-prefixed variables.

Tip

uses replaces the manual PG* + drop db password … --set-secret wiring shown in example 4 below. Pass individual PG* only to override a specific value; prefer discrete PG* over a single DATABASE_URL, which some drivers silently prefer even when stale.

database:

Declared under database:, parsed at drop db create. The API translates it into a CloudNativePG Cluster (plus scheduled backups via the Barman Cloud Plugin) in your tenant namespace. A database has no required key — an empty section yields a default Postgres.

database:
  name: mydb                           # optional; `drop db create mydb` sets it
  engine: postgres-18                  # v1 pins to Postgres 18 (other values are ignored, not errors)
  storage: 1Gi                         # PVC size (Mi / Gi / Ti); default 1Gi, hard cap 1Gi (rejected if larger)
  hibernation: none                    # or "scheduled" → an idle-shutdown CronJob (cost intent)
  extensions: [pgvector, pg_trgm]      # Postgres extensions created at bootstrap (allowlisted; create-time only)
KeyTypePurpose
namestringSelf-identifying database name (≤ 63 chars); the drop db create <name> argument sets it.
enginestringPinned to postgres-18 in v1. Unknown values are pinned to the supported engine rather than rejected (future engines are additive).
storagestringPVC size as a binary-SI quantity (Mi/Gi/Ti, e.g. 512Mi). Default 1Gi; a hard cap of 1Gi applies today — a larger request is rejected at create with a 400 (a fuller per-app + per-tenant quota system is planned).
hibernationstringnone (default) or scheduled — the latter opts the cluster into the idle-shutdown CronJob.
extensionsstring[]Postgres extensions to create at bootstrap, validated against a platform allowlist (default pgvector, pg_trgm, pgcrypto, citext). See Extensions.
Note

The database password never appears in drop.yaml. Drop generates it at create; read it once, or rotate it straight into an app's write-only secret with drop db password <db> --set-secret <app>:<KEY>. Connect via the <name>-rw Service and the <name>-app credentials Secret in your namespace.

Reaching a database from your laptop

Apps in the same namespace connect over the in-cluster network (see uses:) — no proxy needed. For a human psql session from outside the cluster you have two paths: drop db proxy <db> — an authenticated, audited tunnel with per-user authz that works even on locked-down deployments (the recommended default) — or drop db expose <db> for direct L4 TLS-SNI reach when you want a long-lived endpoint. See TCP access for the trade-offs.

Extensions

Request Postgres extensions at create — pgvector (the highest-demand one for internal AI tools), pg_trgm, pgcrypto, citext by default (the allowlist is set per platform via DROP_DB_EXTENSION_ALLOWLIST). A name outside the allowlist is rejected with a 400.

drop db create vecdb --ext pgvector,pg_trgm
drop db ext ls vecdb              # pgvector, pg_trgm  (reads the stored config)

Drop runs CREATE EXTENSION IF NOT EXISTS … in the cluster's bootstrap SQL (pgvector's SQL name is vector). None of the default set needs shared_preload_libraries.

Create-time only (v1)

Extensions are created when the database is first initialised — CNPG's initdb bootstrap is immutable, so adding one to an existing database is a v1 limitation: drop db ext add returns a 409 pointing you to recreate the database with --ext, or run the extension SQL yourself via a superuser migration. This is called out honestly rather than pretending an in-place add works.

Connection pooling (PgBouncer)

Enable a first-class CloudNativePG Pooler (PgBouncer) in front of the primary — it matters as soon as per-request-connection apps meet scale-to-zero churn.

drop db pooler enable mydb --mode transaction   # or --mode session; default transaction
drop db pooler disable mydb                       # deletes the pooler

The pooler is reachable at the <db>-pooler-rw Service. Route an app through it by adding via: pooler to its uses entry (see uses:) — Drop then overrides PGHOST to the pooler instead of the primary. Transaction mode multiplexes connections per-transaction (best for many short-lived connections); session mode holds one server connection per client session (needed for session-scoped state like SET/advisory locks). Enable/disable are audited (db.pooler.enable / db.pooler.disable).

Compute environment variables

The drop.yaml sections above are per-bundle client config. The compute plane itself (enabling it, backups, the secrets and image backends) is configured on the API server through environment variables — these are operator settings, not part of any bundle.

Note

Compute is opt-in: the API enables apps and databases only when DROP_KUBECONFIG is set. Without it Drop is static-only and /v1/apps / /v1/databases return 501. See Architecture §7 for how these map to the cluster.

Enabling compute

VariableDefaultPurpose
DROP_KUBECONFIG—Path to the cluster kubeconfig. Setting it enables the compute plane (/v1/apps + /v1/databases); unset = static-only.
DROP_BLOCKED_EGRESS_CIDRS10.0.0.0/8Comma-separated cluster pod+service CIDRs excluded from each tenant's HTTPS egress allowlist (the per-namespace NetworkPolicy), so tenants can't reach each other or the platform DB. The default covers local k3s only — set the real EKS pod/service CIDRs in prod or cross-tenant egress silently stays open.

Managed-database backups (CloudNativePG)

Drive the Barman Cloud Plugin ObjectStore that streams WAL + base backups to S3.

VariableDefaultPurpose
DROP_DB_BACKUP_ROLE_ARN—IRSA role ARN CNPG assumes to write backups to S3. Prod fails closed without it.
DROP_DB_BACKUP_S3_ENDPOINT—S3 endpoint for backups. Leave empty for real AWS S3; set it to point at Floci (local) or another S3-compatible store.
DROP_DB_BACKUP_S3_EGRESS_CIDR—CIDR added to the tenant egress allowlist so CNPG pods can reach the backup endpoint despite the default-deny NetworkPolicy.

App secrets backend

Write-only app secrets are stored behind a SecretStore port whose backend is picked at deploy time. kube writes the <app>-secret Secret directly; aws writes AWS Secrets Manager entries that the External Secrets Operator syncs into the cluster.

VariableDefaultPurpose
DROP_SECRET_BACKENDkubekube | aws. kube keeps secrets in etcd Secrets (use a KMS encryption-at-rest provider); aws routes them through AWS Secrets Manager + the External Secrets Operator.
DROP_SECRET_STORE_NAMEflociName of the ESO ClusterSecretStore the aws backend reconciles ExternalSecrets against.
DROP_SECRET_MANAGER_ENDPOINT—Secrets Manager endpoint. Empty = real AWS; set it to point at Floci's SM emulation locally (e.g. http://localhost:4566).
DROP_SECRET_MANAGER_REGIONDROP_S3_REGION → us-east-1Region for Secrets Manager calls. Falls back to DROP_S3_REGION, then us-east-1.
DROP_SECRET_MANAGER_KEY_ID—Static access key id for Secrets Manager. Omit in prod to use the IRSA / SDK default credential chain.
DROP_SECRET_MANAGER_SECRET—Static secret access key paired with DROP_SECRET_MANAGER_KEY_ID (local/dev only; prefer IRSA in prod).
DROP_SECRET_PATH_PREFIXdropPath prefix for Secrets Manager entries — secrets land at <prefix>/<ns>/<app>/<KEY>.
Note

App-secret values are write-only end to end — set/rotated/deleted via drop secrets but never returned by the API, CLI, MCP, logs, or the metastore. They reach only the secret manager and the pod env. See CLI, MCP & Dashboard for the commands and tools.

Image push backend

drop deploy --build and drop push stream a locally built image into Drop, which makes it pullable by the cluster — no registry credentials on the client, same command local and prod. The server picks how to ingest the image at deploy time. containerd (local k3s) imports the streamed docker save tarball into the node's containerd; registry (prod) pushes it to a registry/ECR.

VariableDefaultPurpose
DROP_IMAGE_BACKENDcontainerdcontainerd | registry. containerd imports the pushed image into the k3s node's containerd; registry pushes it to a registry/ECR.
DROP_IMAGE_RUNTIMEpodmanContainer CLI the server uses to load the tarball into the node (containerd backend).
DROP_K3S_CONTAINERk3sName of the k3s node container the image is imported into (containerd backend).
DROP_CONTAINERD_SOCK/run/k3s/containerd/containerd.sockcontainerd socket inside the k3s node used to import the image (containerd backend).
DROP_IMAGE_REGISTRY—Target registry images are pushed to. Required when DROP_IMAGE_BACKEND=registry.
DROP_IMAGE_REGISTRY_PULL_SECRET—Image-pull Secret the cluster uses to pull from a private registry (registry backend).
DROP_IMAGE_MAX_BYTES2 GiB (2147483648)Maximum accepted size for a streamed image tarball.
Note

The CLI builder is a client-side setting, not a server one: DROP_BUILDER selects the local container CLI used to build the image (default docker; set to podman to use Podman). Image push is CLI-only today — there is no MCP tool for it yet.

Examples

Four complete drop.yaml variations, from a plain static site to a container app with a managed database. Each is the whole file — mix the keys from the reference above as needed.

1 · Static site

A built folder with clean URLs and a custom 404. Ship it: drop publish ./dist.

site:
  name: marketing
  cleanUrls: true                     # /about → /about.html
  notFound: 404.html
  headers:
    - { source: "/assets/*", headers: { cache-control: "public, max-age=31536000, immutable" } }

2 · Static site, password-protected

Same, gated behind HTTP Basic auth — the edge requires credentials before serving anything. Passwords are "plain" or "sha256:<hex>". Ship it: drop publish ./dist.

site:
  name: staging
  spaFallback: index.html              # SPA: serve index.html on navigation misses
  basicAuth:
    realm: Staging
    users:
      team: "sha256:2bb80d537b1da3e38bd30361aa855686bde0eacd7162fef6a25fe97bf527a25b"
      guest: plain                     # stored as plaintext; use sha256:<hex> to hash

3 · Container app

A stateless HTTP app (no database). Build the folder's Dockerfile and deploy in one step: drop deploy ./app --build.

app:
  name: myapi
  image: myapi:1                        # or omit a prebuilt ref and use `drop deploy --build`
  services:
    - { internal_port: 8080 }              # the port your app listens on (v1: one HTTP service)
  scale: { min: 0, max: 3 }              # min 0 = scale-to-zero when idle
  resources: { cpu: "250m", memory: "256Mi" }
  env:
    LOG_LEVEL: info                       # non-secret config only (secrets: `drop secrets set`)

4 · Container app + managed database

One drop.yaml with both sections. They're deployed by two commands — create the DB first, wire its password into the app's write-only secret, then deploy:

# drop db create guestbook-db ./app
# drop deploy ./app --build --no-start
# drop db password guestbook-db --set-secret guestbook:PGPASSWORD   # rotates + injects; never printed
# drop start guestbook
app:
  name: guestbook
  image: guestbook:1
  services:
    - { internal_port: 8080 }
  scale: { min: 1, max: 2 }              # min 1 so it stays connected to the DB
  env:                                  # non-secret DB connection config (safe to commit)
    PGHOST: guestbook-db-rw              # the DB's read-write Service (same namespace → short name resolves)
    PGPORT: "5432"
    PGUSER: app
    PGDATABASE: app
    PGSSLMODE: require
    # PGPASSWORD is a SECRET — never here; injected via `drop db password … --set-secret`

database:
  name: guestbook-db                   # must match the PGHOST above (<name>-rw)
  storage: 1Gi                         # default + current hard cap
Tip

An app that needs a secret to boot (like PGPASSWORD) should deploy with --no-start, get its secret set, then drop start — otherwise a scale.min: 1 pod crash-loops before the secret exists. See Getting started §Compute.
