Guides

AI intent

Describe what you want to deploy in plain language and let an LLM draft a stack for you — inside Drop's own console. The generated spec lands in the visual editor as a proposed, unapplied change: you review it on the canvas, then Apply → dry-run → confirm, exactly as with any hand-authored stack. It is a drafting aid, never an autopilot.

Off by default

AI intent is disabled unless the operator sets DROP_LLM_URL. A self-hosted Drop never calls out to a third-party LLM on its own — turning this on is an explicit, deliberate operator action. With it unset, the prompt box is hidden and POST /v1/stacks/generate returns 501.

Where it fits

Drop already offers two intent-first paths to provisioning: the deterministic drop detect flow and the MCP tools an agent can drive. AI intent is deliberately thin — it adds the same idea in the console, for humans who don't have an MCP client wired up. All the value it leans on — the stack schema, the visual editor, the reconcile plan — was already built. There is no new CLI surface: the CLI/agent path is covered by detect + MCP.

Enabling it (operators)

Point Drop at any OpenAI- or Anthropic-style chat endpoint that can return structured JSON. Drop speaks both shapes behind one adapter — pick your provider, set the model, done. The endpoint receives only the user's prompt and Drop's public stack schema; it never receives any secret.

VariableMeaning
DROP_LLM_URLThe endpoint URL — an OpenAI-style /chat/completions or an Anthropic-style /v1/messages. Unset ⇒ the whole feature is off.
DROP_LLM_API_KEYThe provider API key. Sent as the provider's auth header (Authorization: Bearer … for OpenAI-style, x-api-key for Anthropic-style). Never logged.
DROP_LLM_MODELThe model id the endpoint expects (provider-agnostic — you choose it).
DROP_LLM_STYLEopenai or anthropic. Optional — defaults to openai, or is inferred from the URL (a /v1/messages path implies anthropic).
DROP_LLM_TIMEOUT_MSPer-request timeout for the LLM call (default 30000).
DROP_LLM_MAX_RESPONSE_BYTESUpper bound on the response body Drop will read (default 256 KiB) — a DoS guard against a hostile/broken endpoint.

Provider-agnostic on purpose: run it against a hosted model, a gateway, or a model you host yourself — Drop only needs an endpoint that speaks one of the two shapes and returns JSON.

The flow

  1. Describe. On the Stacks page a prompt box appears (only when the feature is enabled). Type what you want — e.g. "a Node API with a Postgres database and a static site front-end" — and press Generate.
  2. Draft. Drop calls the configured LLM with a versioned, in-repo prompt template plus the stack JSON schema as the structured-output contract, then runs the raw result through the same stack-config sanitizer the CLI uses — junk is dropped, unknown fields ignored.
  3. Review on the canvas. The sanitized spec loads into the editor as proposed, unapplied pending changes: every resource shows as a "create". A banner reminds you it is AI-generated — review before applying. Edit nodes, fix wiring, drag edges — it is an ordinary editor session now.
  4. Apply → dry-run → confirm. When it looks right, Apply runs a dry-run reconcile plan; you confirm it, and only then does Drop execute it — the exact same path as drop up.
Note

Nothing is created by the Generate step. The spec is a proposal held in your browser until you explicitly Apply and confirm the plan.

Guardrails

  • Never trusted. The model's raw output always passes through sanitizeStackConfig. Unknown fields, secret-looking fields, and unknown resource types are stripped before you ever see the spec.
  • Never executes on its own. A generated spec reaches the cluster only through the visual editor's Apply → dry-run → confirm gate. The generate endpoint makes no cluster calls.
  • Never sees secrets. The prompt template and schema carry no secret material, and the endpoint receives only your prompt plus the public schema. Secrets are managed separately through Drop's write-only secret path and are never put in a stack spec.
  • Versioned prompt. The system prompt lives in the repository and is versioned — it is not user-editable, so what the model is asked is auditable and consistent across a deployment.
