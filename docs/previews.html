Previews

A preview is a labeled, expiring extra version of a static site, served at its own hostname alongside — never instead of — the site's normal live version. Publish a build under a label (a branch name, a PR number, anything DNS-safe) and it's reachable at <name>--<label>.drop.example.com immediately, without touching what <name>.drop.example.com is currently serving. This is the "static previews" slice (E1) — app previews (a full parallel deployment, not just bytes) are a separate, later piece.

The flow

# publish a preview under a label (defaults to a short random suffix if you omit one)
$ drop publish ./dist myapp --preview pr-42
  ▸ packing ./dist
  ▸ dropping to myapp (preview: pr-42)…
  ✓ preview live at https://myapp--pr-42.drop.example.com  (expires 2026-07-11T00:00:00.000Z)

# myapp.drop.example.com is UNCHANGED — still serving whatever it served before

$ drop preview ls myapp
$ drop preview rm myapp pr-42

Re-publishing the same label re-points it at the new build (an upsert, not a second preview) — exactly what a CI job does on every push to a pull request: one label per PR, always pointing at its latest commit.

The hostname scheme

A preview lives at <name>--<label>.<baseDomain> — the double hyphen is the separator between the site name and the preview label. It rides the same flat wildcard certificate as every other Drop hostname (*.drop.example.com) — no new DNS or TLS to provision. Because the separator has to be unambiguous, both halves are constrained:

  • Site names may never contain -- (reserved — see Configuration). This is enforced at claim/publish time, instance-wide.
  • Preview labels are a DNS-safe label, 1–20 characters, and may not themselves contain --.

Given those two rules, the edge can always split a request host on its first -- and get exactly the right site + label back out — no ambiguity, no parsing games.

Expiry

Every preview has an expires_at, defaulting to 7 days from publish. Override it with ?expire_days — the CLI exposes it as --expire-days, clamped to 1–30:

$ drop publish ./dist myapp --preview pr-42 --expire-days 1

A background sweep in the API deletes expired preview rows on a short interval (default every 5 minutes, DROP_PREVIEW_SWEEP_INTERVAL_MS). Requesting an expired (or never-existed) label returns a plain 404 — the same response whether the label never existed or has simply aged out, so a probe can't distinguish the two.

A preview's bytes are VERSION bytes

A preview doesn't get its own storage — it's a pointer at an ordinary published version. That version's bytes are cleaned up by Drop's existing version-retention housekeeping (keepVersions), the same as any other version, without any awareness of a live preview still pointing at it. In practice this only bites a preview that outlives many other publishes to the same site before its own expires_at arrives — publish a fresh preview (or shorten --expire-days) if that matters for your workflow. The sweep's job is narrower and simpler: it only ever removes the preview pointer once its own clock runs out.

Never indexed

Every response served from a preview hostname — 200s, redirects, 404s, auth challenges, all of it — carries X-Robots-Tag: noindex. Preview URLs are throwaway by design (a PR number, a random suffix); nothing about them should ever land in a search index or a crawler's cache.

Visibility & access inheritance

A preview is not a separate resource with its own access settings — it inherits the parent site's visibility and password gate verbatim. If myapp is private, myapp--pr-42 is private too, with the identical 403. If it's password-protected, the same credentials that unlock myapp.drop.example.com unlock the preview. There is no way to make a preview more or less visible than its parent — one gate, applied consistently, is easier to reason about (and to audit) than a second one that could quietly drift.

GitHub Action recipe: a URL per pull request

This extends the CI recipe from CI tokens — same DROP_TOKEN service token, same install step, one more publish flag and a comment step:

# .github/workflows/preview.yml — a preview URL on every push to a PR
name: preview
on:
  pull_request:
jobs:
  preview:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write   # needed to post the PR comment
    env:
      DROP_API: https://api.drop.example.com
      DROP_TOKEN: ${{ secrets.DROP_TOKEN }}   # a drop_st_… service token, scoped to publish:myapp-web
    steps:
      - uses: actions/checkout@v4
      - name: Install the Drop CLI
        run: curl -fsSL "$DROP_API/install.sh" | sh
      - name: Build
        run: npm ci && npm run build
      - name: Publish preview
        id: preview
        run: |
          out=$(drop publish ./dist myapp-web --preview "pr-${{ github.event.pull_request.number }}")
          echo "$out"
          url=$(echo "$out" | grep -oE 'https://[^ ]+')
          echo "url=$url" >> "$GITHUB_OUTPUT"
      - name: Comment the preview URL on the PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `Preview: ${{ steps.preview.outputs.url }}`,
            })

The label pr-$PR means every push to the same pull request re-points the same preview at its latest commit — one comment, always current, no accumulating stale URLs. See CI tokens (service accounts) for minting a scoped DROP_TOKEN (grant it publish:myapp-web, nothing wider).
