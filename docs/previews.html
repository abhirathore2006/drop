Previews

A preview is a labeled, expiring extra instance of a workload, served at its own hostname alongside — never instead of — the live version. For a static site (E1) it's an extra published build; for a container app (E2) it's a full parallel deployment. Either way it's reachable at <name>--<label>.drop.example.com immediately, without touching what <name>.drop.example.com is currently serving. Both kinds share the same hostname scheme, expiry, never-indexed rule, and drop preview ls/rm commands — jump to App previews for the container-specific parts.

The flow

# publish a preview under a label (defaults to a short random suffix if you omit one)
$ drop publish ./dist myapp --preview pr-42
  ▸ packing ./dist
  ▸ dropping to myapp (preview: pr-42)…
  ✓ preview live at https://myapp--pr-42.drop.example.com  (expires 2026-07-11T00:00:00.000Z)

# myapp.drop.example.com is UNCHANGED — still serving whatever it served before

$ drop preview ls myapp
$ drop preview rm myapp pr-42

Re-publishing the same label re-points it at the new build (an upsert, not a second preview) — exactly what a CI job does on every push to a pull request: one label per PR, always pointing at its latest commit.

The hostname scheme

A preview lives at <name>--<label>.<baseDomain> — the double hyphen is the separator between the site name and the preview label. It rides the same flat wildcard certificate as every other Drop hostname (*.drop.example.com) — no new DNS or TLS to provision. Because the separator has to be unambiguous, both halves are constrained:

  • Site names may never contain -- (reserved — see Configuration). This is enforced at claim/publish time, instance-wide.
  • Preview labels are a DNS-safe label, 1–20 characters, and may not themselves contain --.

Given those two rules, the edge can always split a request host on its first -- and get exactly the right site + label back out — no ambiguity, no parsing games.

Expiry

Every preview has an expires_at, defaulting to 7 days from publish. Override it with ?expire_days — the CLI exposes it as --expire-days, clamped to 1–30:

$ drop publish ./dist myapp --preview pr-42 --expire-days 1

A background sweep in the API deletes expired preview rows on a short interval (default every 5 minutes, DROP_PREVIEW_SWEEP_INTERVAL_MS). Requesting an expired (or never-existed) label returns a plain 404 — the same response whether the label never existed or has simply aged out, so a probe can't distinguish the two.

A preview's bytes are VERSION bytes

A preview doesn't get its own storage — it's a pointer at an ordinary published version. That version's bytes are cleaned up by Drop's existing version-retention housekeeping (keepVersions), the same as any other version, without any awareness of a live preview still pointing at it. In practice this only bites a preview that outlives many other publishes to the same site before its own expires_at arrives — publish a fresh preview (or shorten --expire-days) if that matters for your workflow. The sweep's job is narrower and simpler: it only ever removes the preview pointer once its own clock runs out.

Never indexed

Every response served from a preview hostname — 200s, redirects, 404s, auth challenges, all of it — carries X-Robots-Tag: noindex. Preview URLs are throwaway by design (a PR number, a random suffix); nothing about them should ever land in a search index or a crawler's cache.

Visibility & access inheritance

A preview is not a separate resource with its own access settings — it inherits the parent site's visibility and password gate verbatim. If myapp is private, myapp--pr-42 is private too, with the identical 403. If it's password-protected, the same credentials that unlock myapp.drop.example.com unlock the preview. There is no way to make a preview more or less visible than its parent — one gate, applied consistently, is easier to reason about (and to audit) than a second one that could quietly drift.

App previews

Everything above is about static-site previews. A container app preview (E2) is the same idea — a labeled, expiring instance at <name>--<label> that leaves the live app untouched — but instead of pointing at some published bytes it stands up a parallel deployment:

# deploy a preview of an already-deployed app (build + push the image the usual way, then preview it)
$ drop deploy ./app myapp --build --preview pr-42
  ▸ deploying myapp  (drop.local/myapp:abc123) (preview: pr-42)…
  ✓ preview live at https://myapp--pr-42.drop.example.com  (expires 2026-07-11T00:00:00.000Z)

$ drop preview ls myapp          # lists site AND app previews (URL, expiry, "own db")
$ drop preview rm myapp pr-42     # tears the parallel workload down (and any --with-db clone)

Under the hood, drop deploy myapp --preview pr-42 applies a parallel manifest set named myapp-p-pr-42 — its own Deployment, Service and HTTPScaledObject — at host myapp--pr-42.<base>. The edge's -- split resolves that host to the suffixed workload, and the KEDA interceptor wakes and routes it exactly like any other app. The parent app's manifests and current version are never touched.

  • Scale is forced to {min:0, max:1} — a preview is cheap and ephemeral. It scales to zero when idle and wakes on the first request, so an open PR costs nothing while nobody's looking at it.
  • Secrets & bindings are the parent's, read-only. The preview's pods envFrom the parent's <name>-env and <name>-secret — it never gets its own secret set — and each uses: binding (database, cache, bucket, auth, app→app) resolves to the same underlying resource the parent points at. A preview is a view of your app against its real dependencies, not a fresh sandbox.
  • The image is the freshly-resolved ref — the same image-stream path a normal deploy takes (the CLI sends the built ref). The preview registry row stores that image ref in place of a static version id.
  • No release Job runs for a preview: it shares the parent's already-migrated database, and re-running the release command against it could mutate production data.
An app preview's bytes are the image

Unlike a site preview (whose bytes are an ordinary version), an app preview stores nothing of its own — its image is the shared, content-addressed layer set the parent already pulls, GC'd with everything else. Only the manifests (a scale-to-zero Deployment + Service + HTTPScaledObject) and, if you opt into --with-db, one database cost anything. Teardown — at expiry or via drop preview rm — deletes exactly those, idempotently, through the same code path as drop app delete.

--with-db: an isolated database per preview

By default a preview shares the parent app's database — fine for read-heavy previews, dangerous if the preview writes. Pass --with-db and Drop instead clones a fresh, empty database from the parent's bound-db spec (same storage size, engine and extensions), named myapp-p-pr-42-db, and rewires just this preview's database binding to point at it:

$ drop deploy ./app myapp --build --preview pr-42 --with-db
  ✓ preview live at https://myapp--pr-42.drop.example.com  (expires …)
     own empty database: myapp-p-pr-42-db

The clone is an empty CloudNativePG cluster — a fresh schema, not a copy of the parent's data. It's created same-org, audited (preview.create records withDb), and torn down at expiry alongside the workload. A --with-db preview starts blank, so pair it with your app's own migrations/seeding if it needs a schema — or, to test against real data, branch the database with --from-backup.

--from-backup: branch the database from real data

An empty clone answers "does my migration run?"; it can't answer "does my change break against production data?". Add --from-backup and the --with-db clone becomes a branch of the parent database — a fresh CloudNativePG cluster bootstrapped (bootstrap.recovery) from the parent db's most recent Barman backup, i.e. a full copy of its data at that point:

$ drop deploy ./app myapp --build --preview pr-42 --with-db --from-backup
  ✓ preview live at https://myapp--pr-42.drop.example.com  (expires …)
     branched database: myapp-p-pr-42-db (from myapp-db@2026-07-04T09:00:00.000Z)

Point-in-time recovery: pass --at <ISO> to recover to a specific moment (a recoveryTarget.targetTime) instead of the latest backup — e.g. reproduce the state just before a bad deploy:

$ drop deploy ./app myapp --preview repro --with-db --from-backup --at 2026-07-01T12:00:00Z
It reads the backup, never the live volume

Recovery restores from the parent's backup objects in S3 (s3://…/databases/<ns>/<db>) — it never touches the parent database's live volume or its primary, so branching a busy production database is safe and puts no load on it. The parent must have at least one completed backup; if it doesn't, the branch is refused with a 409 (trigger a backup first). The branch archives its own WAL to its own path and gets its own generated password, so it is fully isolated from the parent even though the data is a copy.

A branch is a full copy — mind the storage

Unlike an empty clone, a branch is the same size as the source database. It counts against your org's storage budget just like any database (and is refused with a 429 if it would exceed it). It's swept at preview expiry along with everything else, but while it lives it costs real disk — prefer short expiries for branches.

Branching prod data is a privileged action

Because a branch copies production rows, --from-backup requires manage permission on the parent database (the db:create verb — the same tier that provisions a database), not merely deploy on the app. Someone who can deploy the app but doesn't manage its database can still make an empty --with-db clone, but not branch its data — they get a 403. Every branch is separately audited as db.branch with the source database and the recovery target time, and it's strictly opt-in (never the default). The console preview row shows "branched from <db>@<ts>" so the provenance is visible at a glance.

Quota

An app preview is a real running workload, so it counts against your org's workload cap — a preview that would exceed the cap is refused with a 429 (remove a preview or app, or ask an admin to raise the limit). Re-deploying the same label re-points the existing preview and doesn't count again. With --with-db, the clone's storage additionally counts against the org's storage budget. Delete previews you're done with (or let them expire) to free the slot.

Console

The app detail page lists active previews (label, URL, expiry, an "own db" marker) with a remove button, gated on the deploy capability — the same verb that creates one. Creating a preview is a CLI/CI action; the console manages the ones that exist.

GitHub Action recipe: a URL per pull request

This extends the CI recipe from CI tokens — same DROP_TOKEN service token, same install step, one more publish flag and a comment step:

# .github/workflows/preview.yml — a preview URL on every push to a PR
name: preview
on:
  pull_request:
jobs:
  preview:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write   # needed to post the PR comment
    env:
      DROP_API: https://api.drop.example.com
      DROP_TOKEN: ${{ secrets.DROP_TOKEN }}   # a drop_st_… service token, scoped to publish:myapp-web
    steps:
      - uses: actions/checkout@v4
      - name: Install the Drop CLI
        run: curl -fsSL "$DROP_API/install.sh" | sh
      - name: Build
        run: npm ci && npm run build
      - name: Publish preview
        id: preview
        run: |
          out=$(drop publish ./dist myapp-web --preview "pr-${{ github.event.pull_request.number }}")
          echo "$out"
          url=$(echo "$out" | grep -oE 'https://[^ ]+')
          echo "url=$url" >> "$GITHUB_OUTPUT"
      - name: Comment the preview URL on the PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `Preview: ${{ steps.preview.outputs.url }}`,
            })

The label pr-$PR means every push to the same pull request re-points the same preview at its latest commit — one comment, always current, no accumulating stale URLs. See CI tokens (service accounts) for minting a scoped DROP_TOKEN (grant it publish:myapp-web, nothing wider).
