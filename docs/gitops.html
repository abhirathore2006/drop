Guides

GitOps

Point a stack at a git repository and Drop keeps it in sync — it polls the drop.yaml, and whenever it changes, re-runs the standard drop up for you. It is deliberately pull-only: Drop reaches out to the git host on an interval; the git host never reaches in. Nothing is exposed, no inbound webhook, no callback URL to register.

Why pull, not push

An inbound webhook means exposing the control-plane API to your git host and trusting its payloads — exactly the kind of surface an internal platform exists to avoid. So Drop polls instead: a small housekeeping loop fetches the linked file on an interval (default 60 s, jittered), compares a SHA-256 of the file contents against the last-applied one, and only acts when it changed. No signatures to verify, no endpoint to protect, works behind any firewall.

Linking a stack

drop stack link myproduct \
  --repo https://github.com/acme/myproduct \
  --branch main \
  --path drop.yaml

Supported hosts: GitHub, GitLab, and any raw-file URL. For a private repo, pass a token — it is stored on the link row (never returned by stack link status, never logged) and sent as the host's auth header:

drop stack link myproduct --repo … --token ghp_xxx
# or read it from the environment instead of the shell history:
drop stack link myproduct --repo … --token-env GITHUB_TOKEN

Check the sync state any time:

drop stack link status myproduct
#   repo    github.com/acme/myproduct (main:drop.yaml)
#   last    a1b2c3d · synced · 2m ago
drop stack sync myproduct     # poll + apply right now, don't wait for the interval
drop stack unlink myproduct   # stop polling (nothing is torn down)

Spec-only, images pinned

The poller applies spec changes, not source builds. A linked drop.yaml whose apps carry a dir: build context is refused with a clear error — GitOps mode expects images pinned by ref (a tag or digest your CI already pushed). This keeps the pull loop free of a checkout-and-build step; source-build-on-push composes later with in-cluster builds. Everything a normal drop up does — create, update, topologically-ordered apply under the stack's advisory lock — happens exactly the same way here.

Review before apply

By default a change applies automatically. Add --dry-run-only to gate it on a human:

drop stack link myproduct --repo … --dry-run-only

Now when the file changes the poller does not apply it — it records the pending revision and flips the status to pending_review. The stack page shows a “changes pending review” badge with a Review & apply action; applying it runs the standard up at that revision. This is the safe default for anything you'd hesitate to let a merge deploy on its own.

Failures land in the feed

A fetch error, an unparseable drop.yaml, a refused dir: spec, or a halted reconcile records the error on the link (visible in stack link status and the console badge) and emits a GitOps event to the org's activity feed — so a broken sync is visible next to every other platform event, and reaches your org webhook if you've set one. A successful sync emits a gitops_synced event with the new revision.

In the console

The stack page carries a GitOps section: the linked repo/branch/path, the last-sync revision, status and time, and any last error, with a Sync now button. For a --dry-run-only link it also surfaces the pending-review affordance that opens the plan and applies it on confirm — the same dry-run → plan → confirm path the canvas editor uses.
