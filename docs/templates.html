Guides

Templates

A template is a published, reusable stack: a sanitized spec plus a set of variables, stored per-instance in a registry. Anyone who can see it runs drop new <slug> (or clicks "Deploy this stack" in the console) to instantiate their own copy — the same declarative up that reconciles a hand-written stack, with variables substituted and secrets collected. Templates are the golden-path catalog: publish a working stack once, and the next team ships in one command.

Note

Publishing runs a strip pass that refuses (fail-closed) to bake secrets or pinned image digests into a shared spec. Instantiating never stores a secret in the template or the stack spec — secret variables are written through the same write-only secret path as drop secret set.

Model

The registry is per-instance. A template is org-owned, with a slug that is unique across the whole instance (the drop new namespace), and each published version is immutable:

templates(id, slug unique, org_id, name, description, visibility, created_by, created_at)
template_versions(template_id, version, spec, variables, readme, created_by, created_at)
VisibilityWho can see / instantiate it
publicInstance-wide — every user of this Drop. The default for an internal-tool catalog.
orgMembers only — anyone in the owning organisation. It is a 404 for outsiders (list, get and instantiate).

Re-publishing a slug appends a new version (never mutates an existing one) and refreshes the catalog card's name/description/visibility. drop new takes --version to pin an older one.

Variables

A variable is { key, description?, default?, required, secret? }. Two placeholders are substituted across every string value of the spec at instantiate time:

PlaceholderResolves to
${var.KEY}the variable's value — an explicit --set wins, else its default. A required variable with neither is a 400 (the CLI prompts for it on a TTY).
${stack}the new stack's name — so template-relative hostnames like ${stack}-db-rw resolve to the instantiated database.
Secrets never touch the spec

A secret: true variable may only be used as the whole value of a resource env entry (e.g. SESSION_SECRET: ${var.session_secret}). At instantiate that entry is lifted out of the spec into a secretsToSet plan — { app, key, value } — that the CLI (or console, or MCP tool) writes through the write-only secret path after the up, then restarts the app. The secret value is never stored in the template, never in the stack spec, and never in the audit log (only the key name is recorded). Using a secret variable anywhere else is a hard error.

Publishing & the strip pass

Publish from an existing, working stack (--from-stack) or from a drop.yaml's stack: section (--file):

$ drop template publish --from-stack myproduct --slug myproduct \
    --visibility public --var 'db_storage:database volume size:1Gi' \
    --var 'session_secret:app session secret' --required session_secret --secret session_secret \
    --readme README.md

A --from-stack export captures each app's deployed image (a mutable tag) so the template is instantiable without the original source. Then the strip pass runs:

StepWhat it does
DigestsA concrete image digest (repo:tag@sha256:…) is stripped to repo:tag. dir: build contexts and mutable tags are kept.
Stack-relativeExplicit resource name: overrides are dropped (instantiate re-derives <stack>-<key>); the concrete stack-name prefix inside env values is rewritten to ${stack}.
Registered secretsEnv keys already registered as write-only secrets (app_secret_keys) are removed — they never belong in a shared spec.
Credential flagA remaining env value whose key looks secret (/pass|secret|token|key|credential/i) and whose value has high entropy is flagged — unless it is already variable-ized (${var.…}) or explicitly allowed.
Fail closed

Publishing refuses while any flagged value remains. Fix it the right way — replace it with a ${var.…} variable — or, once you've confirmed it is not a secret, pass --allow <ENVKEY> to let that specific value through. Every --allow is recorded in the audit trail (template.publish).

Instantiating — drop new

$ drop new myproduct                          # latest version, stack name = the slug
$ drop new myproduct --name shop --set db_storage=5Gi
$ drop new myproduct --version 2 --org acme     # a pinned version, into a team org

drop new resolves the template, fills variables from --set (which always wins) and prompts for any missing required variable on an interactive terminal, then runs the same reconcile as drop up. The new stack records its provenance (from_template, from_template_version), so drop stack outdated can later compare it against the template's latest. After the up, the CLI writes any returned secrets to their apps and restarts them, printing exactly what it set.

Agents use the MCP tools template_list, template_show and template_deploy. As with stacks, the agent-safe shape is plan first: template_deploy with dry_run: true returns the ordered plan before anything is created.

The console & deep-link badges

The Templates tab lists every template you can see as cards (name, description, visibility badge). Each card deep-links to /template/<slug> — a page that renders the README, a variables form (required, defaults, and secret fields as password inputs), a read-only canvas preview of the stack graph, and a Deploy this stack button that instantiates it and drops you on the new stack's canvas.

That deep link is stable, so a docs page or README can carry a "Deploy on Drop" badge pointing straight at https://<your-drop>/template/<slug>.

Staying up to date — the upstream diff

A template keeps improving after you instantiate it. Because the stack recorded its provenance (from_template, from_template_version) at drop new time, Drop can compare it against the template's latest version and show you exactly what changed — like Dependabot, but for your infrastructure spec. Crucially it is a three-way diff, so upstream changes are separated from your local drift:

AxisComparisonMeaning
upstreamtemplate@pinned → template@latestwhat the template author changed since you deployed.
local drifttemplate@pinned → your current specwhat you changed on the stack after deploying (a drop up edit, a scale bump…).

Every top-level resource key — and every field within it — falls into one of four classes:

ClassUpstream moved?Local moved?On upgrade
unchangednononothing to do.
upstream-onlyyesnoapplied automatically.
local-onlynoyespreserved — your drift is kept.
conflictyesyes (differently)needs a per-key resolution.

Resources added or removed upstream are surfaced too (added upstream → created; removed upstream → dropped, unless you had modified it, which is a conflict). This is a spec-level diff, per top-level key — not a textual three-way merge. The stack spec is small and structured, so keep it that way.

Note

The instantiate-time variable values are not persisted (secrets never are), so a variable-driven field is compared against the template version's default. A value you overrode at drop new time therefore reads as local drift — which it accurately is: a divergence from the template default that the upgrade will preserve.

drop stack outdated

$ drop stack outdated                # list every template-derived stack + an "update available" flag
$ drop stack outdated shop           # the full three-way diff for one stack

With a name it renders a per-key table — upstream vs local drift vs conflict — so you can see at a glance what an upgrade would pull and what it would leave alone. The console shows the same signal as an "update available" banner on the stack page (it calls the same /outdated endpoint) whenever the stack is behind its template's latest.

drop stack upgrade

$ drop stack upgrade shop --dry-run                    # the reconcile plan, nothing applied
$ drop stack upgrade shop                              # apply (prompts to confirm the plan)
$ drop stack upgrade shop --to 3                        # upgrade to a specific version (default: latest)
$ drop stack upgrade shop --take-upstream db --keep-local api

Upgrade applies every non-conflicting upstream change automatically and preserves your local drift. A conflict — the same resource key changed on both axes — must be resolved per key: --take-upstream <key> swaps in the template's version wholesale, --keep-local <key> keeps yours. An upgrade that leaves any conflict unresolved is refused (the API returns a 409 listing the unresolved keys) — nothing is applied until you decide. The merged spec then feeds the standard drop up dry-run plan, which you confirm before it executes; a successful upgrade re-pins from_template_version to the target and is audited as stack.upgrade. Resources the template dropped are flagged (not torn down) unless you pass --prune.

Conflict resolution is per top-level key

Choosing --keep-local <key> keeps your entire version of that resource, so any non-conflicting upstream field change on the same key is not applied to it (pick --take-upstream if you want the template's). Every other key still upgrades independently.

In the console the banner opens a diff view: the C1 canvas re-fed the merged graph with added / changed / removed / conflict node badges, a per-conflict take upstream / keep local toggle, and an Upgrade button that runs the same dry-run plan modal before it executes.

Seeding the catalog

The two examples ship as ready-to-publish templates. With the API running locally (dev-auth on by default):

$ make seed-templates       # idempotent — publishes `guestbook` + `notes` if absent

It runs scripts/seed-templates.mjs against DROP_API (set DROP_TOKEN for a CI service token instead of the local dev token). Each seed publishes an image-pinned stack with a db_storage variable and a secret session_secret, so instantiating one exercises the whole loop end-to-end.
