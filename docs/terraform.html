Terraform (IaC)

The same AWS components from the AWS deployment guide, as infrastructure-as-code under infra/terraform/. One shared foundation config owns the long-lived resources (ECR, S3, ACM, RDS, Secrets); then either an eks root (Variant A — Kubernetes) or an ecs root (Variant B — Fargate) reads it via remote state and runs Drop behind one internal ALB. Works with Terraform ≥ 1.10 or OpenTofu ≥ 1.10.

Two ways, same result

This page is the IaC equivalent of the AWS deployment guide — that one is the click-by-click aws CLI walkthrough, this one declares the same infrastructure as code. Pick one. Substitute tofu for terraform throughout if you use OpenTofu.

Layout

The configuration is split into independent configs, each with its own state file:

ConfigWhat it manages
foundation/Shared, long-lived resources both runtimes need: ECR repo, private S3 bucket, the S3-access IAM policy, the *.<base_domain> ACM cert, RDS PostgreSQL, the RDS security group, and three Secrets Manager secrets.
eks/Variant A. Runs Drop on Amazon EKS (Kubernetes) via the Helm chart. Reads foundation through a terraform_remote_state data source.
ecs/Variant B. Runs Drop on Amazon ECS/Fargate (no Kubernetes). Also reads foundation via remote state.

You apply foundation once, then either eks or ecs (or both, side by side). Each keeps its own state, so they never collide:

# apply order (teardown is the reverse)
foundation        # must exist first — exports outputs the others read
   └── eks  OR  ecs   # reads foundation's remote state, then deploys Drop
Order matters

Never apply eks/ecs before foundation has been applied at least once — their remote-state lookup finds nothing and the plan fails.

Prerequisites

Terraform here consumes these; it does not create them. Have them ready first:

You provideWhy
An S3 state bucket (with versioning on)Holds the remote state for every config. Named in each backend.tf; intentionally not managed by Terraform (bootstrap chicken-and-egg).
An existing VPC + private subnetsPassed as vpc_id / private_subnet_ids to foundation. Workloads run in the private subnets; no VPC is provisioned.
A Route53 hosted zone for base_domainfoundation looks it up (route53_zone_id) for ACM DNS validation and the api / wildcard records.
The Drop image in ECRBuild the bundles, then the image, then push (see below). The single image carries both api and edge roles.
Terraform / OpenTofu ≥ 1.10 + AWS CLIThe version floor is required for S3-native state locking.

Create the state bucket once, up front:

aws s3api create-bucket --bucket drop-tfstate --region us-east-1
aws s3api put-bucket-versioning --bucket drop-tfstate \
  --versioning-configuration Status=Enabled

Build and push the image (the Dockerfile copies the prebuilt esbuild bundles, so build those first):

# from the repo root
node build.mjs
docker build -f infra/Dockerfile -t <account>.dkr.ecr.<region>.amazonaws.com/drop:<tag> .

aws ecr get-login-password --region <region> \
  | docker login --username AWS --password-stdin <account>.dkr.ecr.<region>.amazonaws.com
docker push <account>.dkr.ecr.<region>.amazonaws.com/drop:<tag>

The ECR repo is created by foundation — so apply foundation first, then build & push to it, then apply eks/ecs referencing that <tag>.

State & locking — S3-native, no DynamoDB

Every backend.tf sets use_lockfile = true. On Terraform ≥ 1.10 / OpenTofu ≥ 1.10 this replaces the old DynamoDB lock table — there is no table to create, reference, or pay for.

terraform {
  backend "s3" {
    bucket       = "drop-tfstate"
    key          = "drop/foundation.tfstate"
    region       = "us-east-1"
    encrypt      = true
    use_lockfile = true   # S3-native lock; no DynamoDB
  }
}

When a run starts, Terraform writes a small <key>.tflock object next to the state via an S3 conditional write; if it already exists, the run refuses to start. It is deleted when the run finishes. Clear a stale lock (after a killed run) with terraform force-unlock <LOCK_ID>.

One AWS account, many states

A single account holds foundation + eks + ecs at once without conflict, because each writes to a different key in the same bucket:

s3://drop-tfstate/
  drop/foundation.tfstate   (+ .tflock while locked)
  drop/eks.tfstate          (+ .tflock while locked)
  drop/ecs.tfstate          (+ .tflock while locked)

For multiple isolated environments in one account, keep the configs and vary the key prefix per environment (staging/foundation.tfstate, prod/foundation.tfstate) with matching *.tfvars.

foundation

The shared layer. Providers: aws ~> 5.0 and random ~> 3.6 (the latter generates DROP_SESSION_SECRET in-state). It creates:

  • ECR — the Drop image repository + a lifecycle policy.
  • S3 — the private sites bucket (versioned, encrypted, public access blocked) and an IAM policy granting Drop's S3 access.
  • ACM — a *.<base_domain> wildcard certificate, DNS-validated via Route53.
  • RDS — a PostgreSQL instance (via terraform-aws-modules/rds/aws) and its security group.
  • Secrets Manager — three secrets: DROP_SESSION_SECRET (generated), DROP_GOOGLE_CLIENT_SECRET, and DROP_DATABASE_URL.

Variant A — EKS (Kubernetes)

Providers: aws, helm ~> 2.0, kubernetes ~> 2.0, http ~> 3.0. It provisions an EKS cluster (terraform-aws-modules/eks/aws) in the foundation's VPC, wires IRSA so Drop's ServiceAccount assumes the S3 IAM role, installs the AWS Load Balancer Controller, syncs the three Secrets Manager values into a Kubernetes Secret, and installs the Drop Helm chart (api + edge) behind one internal ALB Ingress — then points the api and wildcard Route53 records at that ALB.

Note

The Drop API runs Postgres schema migrations on boot under a pg_advisory_lock, so a multi-replica rollout is safe — one pod migrates, the rest wait then serve. The edge connects read-only and never migrates.

Variant B — ECS (Fargate)

Provider: aws only — no Kubernetes. It builds the runtime directly: ALB + app security groups (and an RDS ingress rule from the app SG), an internal ALB with api and edge target groups and an HTTPS listener (host rule routes api.<base_domain> → api, everything else → edge), an ECS cluster, execution + task IAM roles (the task role gets the foundation's S3 policy; the execution role reads the three secrets), CloudWatch log groups, api and edge task definitions and services, and the api + wildcard Route53 records.

Apply workflow

Each config is initialized and applied on its own, from inside its directory. backend.tf ships a placeholder bucket, so pass the real values at init time with -backend-config (the key is already set per config); supply environment inputs with a *.tfvars file.

# 1 · foundation
cd infra/terraform/foundation
cp terraform.tfvars.example terraform.tfvars   # then edit (or use TF_VAR_ for secrets)
export TF_VAR_db_password=… TF_VAR_google_client_secret=…
terraform init \
  -backend-config="bucket=drop-tfstate" \
  -backend-config="key=drop/foundation.tfstate" \
  -backend-config="region=us-east-1"
terraform apply -var-file=terraform.tfvars

# 2 · build & push the image now that the ECR repo exists (see Prerequisites)

# 3 · pick ONE runtime — eks here; swap 'eks' for 'ecs' for Fargate
cd ../eks
terraform init -backend-config="bucket=drop-tfstate" -backend-config="key=drop/eks.tfstate" -backend-config="region=us-east-1"
terraform apply -var-file=terraform.tfvars

To tear down, reverse the order: destroy eks/ecs first, then foundation.

Secrets

Never put real secrets in *.tfvars. The sensitive foundation inputs (db_password, google_client_secret) are best supplied via TF_VAR_… environment variables; the running app reads DROP_SESSION_SECRET, DROP_GOOGLE_CLIENT_SECRET, and DROP_DATABASE_URL from Secrets Manager (owned by foundation), so the variants never handle them directly.
