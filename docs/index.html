Ship internal sites and apps.
Nothing ever leaves.

Drop is a self-hosted publisher for your own network — a Surge/Netlify/Vercel-style static-site host and a Heroku/Fly-style compute platform. Push a built folder, or deploy a container app with a managed Postgres and write-only secrets — both get a confidential URL on *.drop.example.com, under one set of names, auth, and tooling. Everything lives in your own S3, your own Postgres, and your own cluster, behind your own Google Workspace.

Get started → CLI & MCP Architecture Deploy 
$ drop publish ./dist myapp           # static site
  ✓ live at https://myapp.drop.example.com  (internal-only)

$ drop deploy ./api myapi             # container app (drop.yaml app:)
  ✓ live at https://myapi.drop.example.com
$ drop db:create myapi-db             # managed Postgres (CloudNativePG)
$ printf "$PW" | drop secrets set myapi DATABASE_PASSWORD --stdin  # write-only secret
$ drop restart myapi                  # apply it
Two planes, one product

Static sites serve bytes from S3 at the edge. Compute (apps + databases + secrets) is opt-in — enabled only when the API has a Kubernetes cluster; without one, Drop is static-only. See CLI, MCP & Dashboard and examples/DATABASE_APPS.md.

Quickstart

Install the CLI straight from your Drop server. The installer is served by the API itself, so it auto-configures this instance's URL — the commands below then just work. Needs only Node 18+.

# 1 · install the CLI (downloads from this Drop, writes config for you)
curl -fsSL https://api.drop.example.com/install.sh | sh

# 2 · sign in — Google, brokered by the Drop server (no secrets on your machine)
drop login

# 3 · publish a built folder → a live, internal-only URL
drop publish ./dist myapp
Served live

When you read these docs on a deployed Drop, the URL above is filled in with this instance's real address — copy it and run it as-is. Prefer not to install? Run the CLI via npx instead — see CLI, MCP & Dashboard.

Drive it from an AI client (MCP)

Any MCP client — Claude Code/Desktop, Cursor, … — can publish and manage sites in natural language. After installing, point it at the bundled drop-mcp server (installed alongside drop):

{
  "mcpServers": {
    "drop": {
      "command": "drop-mcp",
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}

Why Drop

CONFIDENTIAL

Stays inside the company

Sensitive, agent-generated analysis never touches a third-party SaaS — it sits in your S3, served internal-only, gated by Workspace login (and optional per-site password).

PUBLISH

Reports · decks · dashboards

Any static bundle becomes a live internal URL in seconds — no deploy pipeline, no ports, nothing leaving the network.

AGENT-NATIVE

CLI & MCP

Install the CLI with one line (or run it via npx), and let any AI client publish directly through the built-in MCP server.

CONTROLLED

Roles & visibility

Per-site owner / editor / viewer roles, platform admins, and public / private / password visibility — all backed by Postgres for real reporting.

COMPUTE

Container apps

Deploy a container with drop deploy — a Deployment + Service + KEDA HTTPScaledObject. Apps scale to zero when idle; the edge wakes the pod on the first request to <app>.drop.example.com.

DATA

Managed Postgres

drop db:create provisions a CloudNativePG cluster in your namespace with Barman backups to S3. drop db:password sets/rotates the role password — printed once, never the connection password stored where it can be read back.

WRITE-ONLY

App secrets

Set, rotate, and delete per-app secrets — never readable back — injected as env vars. Backed by the kube Secret store or AWS Secrets Manager via the External Secrets Operator. Values never touch drop.yaml, git, logs, or responses.

OPERATE

Lifecycle & console

restart / stop (true offline) / start from the CLI, MCP, or the React console — workloads grouped by type with live status, a per-app drawer for logs/secrets/lifecycle, per-DB connection + password, and an admin all-workloads view.

How it fits together

Two small, stateless services share a Postgres metadata store and an S3 byte store; the compute plane adds an opt-in Kubernetes cluster:

  • api — the control plane: Google login, publishing, deploy, rollback, sharing, visibility, admin, the dashboard, and — when compute is enabled — apps, databases, secrets, and lifecycle. Owns schema migrations; the only writer to Postgres and the cluster.
  • edge — the data plane: serves published site bytes by hostname, and dispatches app hostnames to the in-cluster KEDA HTTP interceptor (scale-from-zero). Read-only against Postgres.

Static-site bytes live in S3 (sites/<name>/files/<verId>/…); site / app / database metadata lives in Postgres; apps, databases, and secrets are Kubernetes objects in per-tenant namespaces. The sites.type discriminator — site, app, or database — shares one name namespace. See the architecture for diagrams of the publish, serve, and compute flows.

Compute platform

Beyond static sites, Drop runs container apps and managed databases on Kubernetes (EKS in prod; k3s-on-podman locally), with the same names, auth, and CLI/MCP/console. It's opt-in — the API enables it only when it has a cluster; otherwise /v1/apps and /v1/databases return 501 and Drop is static-only.

  • Container apps — declare an app: section in drop.yaml and drop deploy <dir>. The API translates it to a Deployment + Service + KEDA HTTPScaledObject; apps scale to zero when idle and the edge wakes them on first request.
  • Managed databases — drop db:create provisions a CloudNativePG Postgres in your namespace with Barman backups to S3; drop db:password sets/rotates the role password. The connection ref is returned — never the password.
  • Write-only secrets — drop secrets set --stdin / ls / rm manage per-app secrets that are never readable back and inject as env vars (backend kube or AWS Secrets Manager via the External Secrets Operator).
  • Lifecycle — drop restart / stop / start (stop = true offline) from the CLI, MCP, or the console.
  • Per-tenant isolation — every owner gets a namespace with a default-deny NetworkPolicy and resource quota.

What you can ship

Anything that builds to static files: multi-page HTML, single-file reports, reveal.js / slide decks, and full client-rendered SPAs (Vite, React, Vue, Svelte, Astro). Route-aware SPA fallback, clean URLs, redirects, custom headers, CORS, and HTTP password are all configured per-site with a small drop.yaml.

And — with the compute plane — anything that runs in a container: an HTTP API, a server-rendered app (Next.js, Remix), a worker fronted by HTTP. Add an app: section to the same drop.yaml, give it a managed database and write-only secrets, and drop deploy.
