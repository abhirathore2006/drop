Ship internal sites.
Nothing ever leaves.

Drop is a self-hosted, Surge/Netlify/Vercel-style static-site publisher for your own network. Push a built folder — an HTML report, a slide deck, a Vite/React dashboard — and get a confidential URL on *.drop.example.com. Everything lives in your own S3 and your own Postgres, behind your own Google Workspace.

Get started → Architecture Deploy 
$ drop publish ./dist myapp
  ▸ packing ./dist
  ▸ dropping…
  ✓ live at https://myapp.drop.example.com  (internal-only)

Quickstart

Install the CLI straight from your Drop server. The installer is served by the API itself, so it auto-configures this instance's URL — the commands below then just work. Needs only Node 18+.

# 1 · install the CLI (downloads from this Drop, writes config for you)
curl -fsSL https://api.drop.example.com/install.sh | sh

# 2 · sign in — Google, brokered by the Drop server (no secrets on your machine)
drop login

# 3 · publish a built folder → a live, internal-only URL
drop publish ./dist myapp
Served live

When you read these docs on a deployed Drop, the URL above is filled in with this instance's real address — copy it and run it as-is. Prefer not to install? Run the CLI via npx instead — see CLI, MCP & Dashboard.

Drive it from an AI client (MCP)

Any MCP client — Claude Code/Desktop, Cursor, … — can publish and manage sites in natural language. After installing, point it at the bundled drop-mcp server (installed alongside drop):

{
  "mcpServers": {
    "drop": {
      "command": "drop-mcp",
      "env": { "DROP_API": "https://api.drop.example.com" }
    }
  }
}

Why Drop

CONFIDENTIAL

Stays inside the company

Sensitive, agent-generated analysis never touches a third-party SaaS — it sits in your S3, served internal-only, gated by Workspace login (and optional per-site password).

PUBLISH

Reports · decks · dashboards

Any static bundle becomes a live internal URL in seconds — no deploy pipeline, no ports, nothing leaving the network.

AGENT-NATIVE

CLI & MCP

Install the CLI with one line (or run it via npx), and let any AI client publish directly through the built-in MCP server.

CONTROLLED

Roles & visibility

Per-site owner / editor / viewer roles, platform admins, and public / private / password visibility — all backed by Postgres for real reporting.

How it fits together

Two small, stateless services share a Postgres metadata store and an S3 byte store:

  • api — the control plane: Google login, publishing, rollback, sharing, visibility, admin, and the dashboard. Owns schema migrations.
  • edge — the data plane: serves published bytes by hostname, applies per-site config and visibility. Read-only against Postgres.

File bytes live in S3 (sites/<name>/files/<verId>/…); everything else — sites, members, versions, users, auth handles — lives in Postgres. See the architecture for diagrams of the publish and serve flows.

What you can publish

Anything that builds to static files: multi-page HTML, single-file reports, reveal.js / slide decks, and full client-rendered SPAs (Vite, React, Vue, Svelte, Astro). Route-aware SPA fallback, clean URLs, redirects, custom headers, CORS, and HTTP password are all configured per-site with a small drop.yaml.
