Roles & Visibility

Two orthogonal axes: roles decide who can manage a site; visibility decides who can view the served pages.

These are independent. A site can be public (anyone with the URL sees it) but still have a tight set of owners and editors. A site can be private (no one can view it yet) while several people manage and publish it. Roles gate the API and dashboard; visibility gates the edge.

Roles & permissions

There are two role axes. Permissions are defined in code in src/authz/permissions.ts — not scattered through handlers.

Platform role

Stored on users.role as admin or member. An admin can see and manage all sites and users; a member is a normal user. The DROP_ADMINS env var (comma-separated emails) seeds admins when the API boots; thereafter admin status is DB-managed. See Deployment for the env var.

Per-site role

Stored on site_members.role as owner, editor, or viewer. There is exactly one owner per site. The same role gates both the static-site actions and the compute-plane actions (apps, databases, secrets). The map is:

  • owner — everything: read, logs, publish, deploy, db:create, rollback, configure, share, transfer, delete.
  • editor — read, logs, publish, deploy, db:create, rollback, and lifecycle (restart / stop / start).
  • viewer — read only.

Here read means seeing the workload (site, app, or database) in the dashboard, plus its versions, status, and settings; logs means reading an app's pod logs; configure means setting visibility, the DB password, and write-only app secrets. deploy ships a container-app revision and db:create provisions/updates a managed Postgres database — both are editor+, alongside publish.

Note

can(actor, action) is the single authority check used everywhere. Platform admins are all-powerful on every site; a non-member can do nothing. transfer hands ownership to another member and demotes the old owner to editor — so there is still exactly one owner.

Warn

logs is gated above plain read/viewer. Pod logs can echo env-injected secrets (an app's DATABASE_PASSWORD and friends are exposed to the process as environment variables), so reading them is an editor+ action even though a viewer can already see the app's status. A viewer is deliberately metadata-only and never sees the credential-bearing log stream.

Warn

App secrets are write-only and owner/admin-only. Setting, rotating, or deleting a secret is the configure permission (same gate as visibility and the DB password). Editors can deploy and restart apps but cannot manage secrets, and no role can read a secret value back — the API/MCP/CLI only ever return key names and metadata. See CLI, MCP & Dashboard for drop secrets set --stdin / ls / rm.

Visibility

Each site has a visibility of public, password, or private. Set it with POST /v1/sites/:name/visibility (the configure permission) or from the dashboard.

  • public default — served openly to anyone with the URL.
  • password — the edge requires HTTP basic-auth against the site's password_hash. Publishing a drop.yaml with a basicAuth block also turns this on.
  • private — only members (and platform admins) may view.
Warn

Edge enforcement of private viewing (viewer authentication) is the next feature. Until then a private site returns 403 at the edge — it fails closed and is never served openly. Owners and editors can still manage and publish; only the served page is gated.

Admin

Platform admins (seeded by DROP_ADMINS, then DB-managed) get an all-sites view.

  • GET /v1/me returns { admin: true } for platform admins.
  • GET /v1/admin/sites?cursor=&limit=&prefix= pages through every site (keyset-paginated, name-prefix filterable) regardless of ownership.
  • The dashboard shows an all sites tab backed by the same endpoint.
GET /v1/me            → { "admin": true }
GET /v1/admin/sites?limit=50&prefix=team-&cursor=<name>
Note

Admin status never bypasses per-site ownership or drop.yaml name-ownership. A publish still has to satisfy the name claim in the config; being an admin does not let you steal a name that another site already owns.
