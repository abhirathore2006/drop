Guides

Auth — per-app end-user pools

A managed auth resource gives one of your apps its own pool of end users — sign-up, sign-in, OAuth providers, and JWT-based sessions — running as a GoTrue engine in your tenant namespace with the users stored as rows in your own Postgres. It is a workload type, like a database or a cache: you drop auth create it, bind it to an app, and Drop owns its lifecycle and wiring — never the auth engine's internals (password hashing, session rotation, token issuance — the most dangerous code one can write, which Drop writes none of).

Drop's platform login is NEVER shared with your apps

The login you use for the Drop console/CLI authenticates operators. A managed auth resource authenticates your app's end users — an entirely separate pool with its own methods and its own JWTs. The two never mix, by design.

What works in v1 (and what's deferred)

Everything that needs to send an email is deferred to a named future task (“K-mail”: an org SMTP relay + the email-dependent flows). The drop.yaml schema already reserves the smtp:/email: keys, so K-mail is purely additive — a config written today lights up when it ships.

The engine (pinned)

The default engine is GoTrue (MIT), pinned to a concrete image tag — docker.io/supabase/gotrue:v2.170.0 — never a moving :v2 or :latest, so a tenant's login surface can't shift under a silent upstream push. Bump it deliberately, tracking GoTrue's CVE advisories; the Helm value auth.engineImage (env DROP_AUTH_ENGINE_IMAGE) overrides it and is where you point an air-gap mirror. GoTrue sits behind an AuthEngine port (image + env-mapping + admin-route abstraction), so a future swap to another engine is a new adapter, not a redesign.

Create an auth resource

An auth resource requires a same-org database — its engine schema and your users' rows live there. Bind an existing one with --db, or create one first with --with-db (CLI-side sugar: it runs drop db create <name>-db, then binds).

drop db create appdb
drop auth create login --db appdb
#   ✓ auth ready — engine gotrue, JWT HS256, signup open
#      login/API base: https://auth--login.<baseDomain>
drop auth create login --with-db            # create login-db first, then bind
drop auth create login --db appdb --signup closed   # admin-created users only

You can also declare an auth: section in drop.yaml: { providers, redirect_urls, jwt_ttl, signup }. The engine is reachable at auth--<name>.<baseDomain> — the -- reserved-prefix convention (a real resource name can never contain --, so the host is unambiguous). GoTrue's REST API is the auth API — /token, /signup, /user, /admin/users, the OAuth /authorize flow — curl-able from day one.

JWT signing — HS256, and the JWKS consequence

v1 ships HS256 (symmetric), not asymmetric keys — an honest deviation

The plan preferred asymmetric per-resource keys with a public JWKS URL. OSS GoTrue does not support that: it signs and verifies with a single shared GOTRUE_JWT_SECRET and serves no /.well-known/jwks.json (that is a Supabase-platform feature, not in the OSS image). Rather than fork the engine, v1 ships HS256 honestly. The consequence: a binding app verifies tokens with the shared secret (AUTH_JWT_SECRET, injected via the write-only path) — not a public key. When a future engine adds asymmetric signing, the AuthEngine port grows a JWKS surface without changing the resource model.

The signing secret is a 256-bit value generated at create into a write-only <name>-auth-keys Secret. It is never returned to any client, never in a manifest, never in the console. Each resource has its own secret, so a token minted by resource A is cryptographically rejected by resource B (an isolation guarantee the test suite asserts).

Rotating keys (with a grace window)

drop auth rotate-keys login

Rotation re-mints the secret and re-applies the engine. GoTrue itself verifies only the current secret, so the grace window is realized for your bound apps: Drop re-injects the new secret and the previous one — AUTH_JWT_SECRET plus AUTH_JWT_SECRET_PREVIOUS — and restarts each consumer, so tokens signed just before the rotation still verify until the next rotate/redeploy overwrites the previous value. Keep access-token TTLs short (1h default) so a disabled user loses access quickly.

Bind an auth resource to an app

app:
  image: my-org/web:1
  uses:
    - auth: login       # an auth resource in the SAME org

On deploy, Drop resolves the resource (same-org — a cross-org bind is a 400), reads back its signing secret, and writes into the app's write-only secret:

VariableWhat it is
AUTH_URLhttps://auth--<name>.<baseDomain> — the engine's base URL your app calls / redirects to.
AUTH_JWT_SECRETThe shared HS256 secret your app verifies incoming tokens with (write-only path — never in a manifest).
AUTH_JWT_SECRET_PREVIOUSPresent only after a key rotation, for the grace window (see above).
No anon / service-role API keys — a deviation from the Supabase split

Supabase clients expect a publishable “anon” key and a secret “service_role” key. OSS GoTrue has no such keys — its API is key-less and authorized purely by the JWT you present. So v1 injects AUTH_URL + AUTH_JWT_SECRET and stops there; there is no AUTH_PUBLISHABLE_KEY/AUTH_SECRET_KEY pair. Server-side, Drop mints a short-lived service-role JWT (signed with the resource secret) only to proxy the admin API — it never leaves the control plane.

Multiple auth resources

Bind more than one and each variable is prefixed with the resource key so they don't collide — e.g. LOGIN_AUTH_URL. A single binding is unprefixed. Inside a stack, an auth resource declares its db: and apps reference it with uses: [{ auth: <key> }].

Roles & permissions for your app

App roles are NOT Drop's platform roles

This is your application's authorization model — vocabulary you define (admin, editor, billing:read…), resolved for each end user and stamped into their JWT. It is entirely disjoint from the platform roles that decide who may operate Drop (owner/admin/editor on a workload). The two never mix, by design.

Turn it on at create time — auth.rbac: true in the auth: config (or the stack's auth resource). This does two things: it plumbs GoTrue's custom access-token hook onto the engine, and it makes drop auth rbac-seed available to install the schema.

auth:
  rbac: true          # seed the app RBAC schema + wire the JWT claims hook

The three tables (the Supabase pattern)

Two many-to-many hops — users ↔ roles ↔ permissions — seeded into your app's own bound Postgres:

TableColumnsWhat it is
app_rolesid, name (unique)Your role vocabulary.
app_role_permissionsrole_id, permissionThe permissions each role grants.
app_user_rolesuser_id (uuid), role_idWhich users hold which roles (user_id = GoTrue's auth.users.id).

The claims hook

Alongside the tables, a Postgres function public.drop_access_token_hook(event jsonb) returns jsonb is created. GoTrue calls it on every issued JWT (via GOTRUE_HOOK_CUSTOM_ACCESS_TOKEN_ENABLED / _URI, plumbed automatically when rbac: true). It joins the two link tables on event->>'user_id' and stamps the resolved arrays into the token claims:

{ "sub": "…", "email": "…", "roles": ["admin"], "permissions": ["notes:write"] }
Engine compatibility (verified)

The pg-functions access-token hook is supported by the pinned engine supabase/gotrue:v2.170.0 (2025-03-06); the hook infrastructure is present by v2.145.0 (2024-03-26). GoTrue's URI parser uses only the schema + function from pg-functions://postgres/public/drop_access_token_hook — the leading postgres segment is ignored — so the URI is correct even though your bound database is named app. GoTrue calls the function over its existing connection (as the app role, which owns the tables + function), so no cross-role grants are needed.

Applying the seed (v1 — honest)

The seed is applied by you via psql — not silently by the control plane

The plan called these "engine-provision assets". In practice the control plane has no SQL-execution path against a tenant database yet (that arrives with the SQL console), and the bound database may pre-exist the auth resource — so Drop will not silently run DDL against your data. Instead the seed ships as (a) exported SQL assets, (b) a CLI command that prints the SQL, and (c) the hook env, wired the moment you set rbac: true. All DDL is idempotent (IF NOT EXISTS / CREATE OR REPLACE), so re-running is safe.

drop db proxy appdb                                  # authenticated tunnel; prints a local port
drop auth rbac-seed login | psql "postgres://app@127.0.0.1:<port>/app"

Grant a user a role with plain SQL (the vocabulary is yours):

insert into app_roles (name) values ('admin') on conflict do nothing;
insert into app_user_roles (user_id, role_id)
  select '<user-uuid>', id from app_roles where name = 'admin';

Reading it in your app — @drop/auth

The in-repo SDK @drop/auth (zero runtime dependencies) has two halves. The client wraps GoTrue's REST API, reading the injected AUTH_URL:

import { createAuthClient } from "@drop/auth";
const auth = createAuthClient();                 // AUTH_URL comes from the binding
const { access_token } = await auth.signIn(email, password);
// also: signUp(email,password), signOut(token), getUser(token), refresh(refreshToken)

The server half verifies an incoming request's token locally — HS256 with the injected AUTH_JWT_SECRET (and AUTH_JWT_SECRET_PREVIOUS during a rotation grace window) — and returns the user plus the hook's roles/permissions:

import { verifyRequest } from "@drop/auth";
const { user, roles, permissions } = await verifyRequest(req);
if (!roles.includes("admin")) return forbidden();

verifyRequest is framework-agnostic: pass a raw token string, a Fetch Request, a Headers, a Node req ({ headers }), or a plain headers object — it finds the bearer token, and throws on an expired / tampered / missing one. It is async and server-only (it lazy-loads node:crypto, so client bundles that only use createAuthClient stay clean).

Enforcing in Postgres RLS (a documented pattern, not machinery)

Because your app rows and your users live in the same bound Postgres, you can additionally enforce authorization in the database with Row-Level Security that reads the same claims. This is a pattern you own — Drop provisions neither the policies nor the claims injection:

-- 1) at the start of a request, after verifyRequest, put the claims into the session:
select set_config('request.jwt.claims', $1, true);   -- $1 = JSON.stringify(claims)

-- 2) policies then read them:
alter table notes enable row level security;
create policy admins_write on notes for all
  using ( (current_setting('request.jwt.claims', true)::jsonb -> 'roles') ? 'admin' );
Pattern, not platform

Drop ships the roles/permissions tables + the claims hook. RLS policies that reference those claims — and the set_config that injects them per request (there is no PostgREST doing it for you) — are entirely yours to write. Documented here so the capability is discoverable.

OAuth providers

Declare each provider's own client id under auth.providers (client ids are not secret — they live in the spec). The client secret is written out-of-band, never to the spec:

auth:
  providers:
    google: { client_id: "…apps.googleusercontent.com" }
    oidc:   { client_id: "my-app", issuer: "https://idp.example.com" }
  redirect_urls: [ "https://myapp.<baseDomain>/auth/callback" ]
drop secrets set login GOTRUE_EXTERNAL_GOOGLE_SECRET=…    # the provider secret → the engine's write-only Secret
ProviderSecret env key (via drop secrets set)
googleGOTRUE_EXTERNAL_GOOGLE_SECRET
githubGOTRUE_EXTERNAL_GITHUB_SECRET
oidc (generic)GOTRUE_EXTERNAL_KEYCLOAK_SECRET

Redirect topology is per-app callbacks (https://<app>.<base>/auth/callback) — there is deliberately no centralized broker (a shared broker drifts back toward shared platform auth). The generic oidc kind maps to GoTrue's Keycloak provider, which accepts an arbitrary issuer.

Admin the users (the no-SMTP onboarding path)

Because verification emails are deferred, the primary onboarding path is admin-created users. This is configure-gated (owner/admin) and audited (auth.user.create / auth.user.disable / auth.user.delete). The control plane mints a short-TTL service-role token and proxies GoTrue's admin API server-side — the token never leaves the server.

drop auth users create login alice@example.com    # prints a temp password ONCE
drop auth users ls login
drop auth users rm login <user-id>

The console's Auth detail page shows providers, signup state, key age, and the same user admin (list / search / create-with-temp-password / disable / delete) with the created password shown once. MCP exposes auth_create, auth_status and auth_config (read the config surface — never key material) only — there is deliberately no end-user CRUD via agents in v1.

ADR: the edge visibility exemption

The single most security-sensitive change in the visibility model — reviewed as an ADR

Context. Every other served workload passes the edge's platform visibility/session gate. An auth host cannot: the whole point of auth--<name> is that unauthenticated users reach /token and /signup to obtain a session — login IS the auth. A session gate in front of the login endpoint is a chicken-and-egg lockout.

Decision. The edge special-cases hosts of the form auth--<name> whose resolved workload is genuinely type: auth: it skips the platform visibility gate and proxies to the engine like an app. In exchange it applies a per-IP token-bucket rate limit to the sensitive POST paths (/token, /signup, /verify, /recover; default 10/min/IP, config-capped) before proxying, returning 429 with Retry-After when a client exceeds it — so the exemption can't be abused as an unthrottled credential-stuffing or account-enumeration endpoint.

Scope (the carefully-reviewed part). The exemption is granted only when the resolved workload is type: auth. A site literally named auth with a preview label also matches the auth-- prefix, but is not an auth resource — it, and every other -- preview host, keeps its gates. The check is on the resolved type, not the string prefix; both cases are covered by tests (an auth host is exempt; a preview -- host of a private site still returns 403).

Consequences. The rate limiter is in-process per edge replica (matching the metrics collector's posture): a multi-replica edge multiplies the effective limit by replica count — acceptable for a v1 abuse brake, not a billing meter. This exemption is the one hole in the default-closed visibility model and is called out here so it is never re-derived silently.

How it runs, status & deletion

The engine is a GoTrue Deployment pinned to 1/1 (auth can't cold-start on a login — a scaled-to-zero engine would drop the first sign-in), a ClusterIP Service on 9999, and an HTTPScaledObject registering auth--<name> with the interceptor (min:1 keeps it warm). It connects to the bound database over the reused B1 binding mechanics — the CNPG <db>-app creds Secret + the cluster CA mounted read-only + sslmode=verify-full. The detail page / drop auth config / MCP auth_status surface providers, signup, key age and live status — never key material. Deleting the resource tears down the engine and its keys Secret; the users stay in the bound database (deleting an auth resource never drops your user rows).

Security-critical dependency

Drop becomes a security-critical dependency for every app that uses this: key rotation, engine CVE tracking (the pin above + advisory watch), and short token TTLs so a disabled user loses access quickly. The engine image is pinned and mirror-able for exactly this reason.
