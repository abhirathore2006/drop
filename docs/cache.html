Guides

Cache — managed Valkey

A cache is a managed, single-replica Valkey (a Redis-compatible in-memory store) you create in one command and bind to an app so its connection URL — password included — arrives as an environment variable. It is deliberately tiny: the anti-Redis-Cluster. No HA, no clustering, no tuning knobs — a fast local cache or a BullMQ-style job queue for an internal app.

EPHEMERAL by default — a restart loses ALL data

Unless you pass --persistent, a cache has no disk: any restart — a redeploy, a node move, a crash, an out-of-memory kill — starts it empty. That is the right default for a cache (rebuildable) and acceptable for a queue you can re-enqueue. If you need the data to survive restarts, create it --persistent (a small disk with periodic snapshots), and remember that even then a cache has no backups — deleting it always wipes the data. For durable data, use a managed database or a bucket.

A cache is a workload type

Like a site, app, database or bucket, a cache is a row in the shared resource namespace with an owner, an org, roles and an audit trail. It appears in drop ls, the console groups it under Caches, and it is deleted the same way.

Create a cache

drop cache create sessions
#   ▸ creating cache sessions…
#   ✓ cache ready — 256Mi, EPHEMERAL (a restart loses all data)
#      host: sessions.<tenant>.svc.cluster.local:6379
#   ✓ connection URL — shown once, store it now:
#      redis://:<password>@sessions.<tenant>.svc.cluster.local:6379
drop cache create jobs --memory 512Mi --persistent   # bigger + survives restarts
drop cache create sessions --org acme                # create in a team org

--memory takes a k8s quantity clamped to 64Mi–1Gi (default 256Mi); it sizes the pod memory limit and, when --persistent, the disk. You can also declare a cache: section in drop.yaml ({ memory, persistent }). The connection URL is printed exactly once — the password is embedded in it, and the platform never returns it again. In practice you rarely copy it by hand: bind the cache to an app (below) and Drop injects it for you.

RevealOnce connection URL

Create reveals the REDIS_URL (with the password) one time. There is no rotate command for a cache — if you lose the URL, delete and recreate (which wipes the data, ephemeral or not). This mirrors drop db password's reveal-once posture.

Bind a cache to an app

Declare the dependency in your app's drop.yaml with a uses edge — the same mechanism that binds a database or a bucket:

app:
  image: my-org/worker:1
  uses:
    - cache: sessions       # a cache in the SAME org

On deploy, Drop resolves the cache (it must be in the same org — a cross-org bind is refused with a 400), reads back its password, and writes the composed URL straight into the app's write-only secret, so the pod boots with:

VariableWhat it is
REDIS_URLredis://:<password>@<name>.<tenant>.svc.cluster.local:6379 — a standard Redis URL with auth, ready for any Redis/Valkey client (or BullMQ).

Because the URL rides the write-only secret path, the password never appears in the manifest, the deploy response, or the console.

Multiple caches

Bind more than one and each URL is prefixed with the resource name so they don't collide — e.g. SESSIONS_REDIS_URL, JOBS_REDIS_URL. A single binding is unprefixed (REDIS_URL). Inside a stack, the prefix is the resource key. This mirrors the bucket-binding convention exactly.

How it runs

A cache is a single-replica Valkey Deployment plus a ClusterIP Service on port 6379 in your tenant namespace. Authentication (requirepass) is a platform-generated password delivered to the container through a Kubernetes Secret — never as plaintext in the pod spec. The pod's memory limit is your requested memory; Valkey's maxmemory is set just under it (so writes are refused before the pod is OOM-killed rather than losing the process abruptly). A --persistent cache adds one small ReadWriteOnce disk mounted at /data with periodic RDB snapshots, and the Deployment uses a Recreate rollout so the disk never double-attaches. There is no eviction policy (noeviction) — a full queue refuses writes rather than silently dropping jobs.

Reaching it from outside the cluster

By default a cache is reachable only from apps in the same tenant namespace (the tenant network policy is default-deny; intra-namespace traffic is allowed). To connect a local client for debugging, expose it over the L4 plane — a TLS-Valkey rides the shared SNI port and a plain connection rides a dedicated port — exactly like exposing a database for psql.

Status & deletion

The cache detail page (and drop cache ls / MCP cache_status) surfaces the host, memory, persistence and live status — never the password.

drop cache rm sessions           # tears down the Valkey AND its data (no confirmation of contents — a cache has no backup)

Quota

A cache occupies one slot against the per-org workload cap. A persistent cache's disk request also counts toward the org's storage budget (when one is set), alongside database PVC requests and bucket bytes — GET /v1/orgs/<slug>/usage reports a caches figure in its storage section. An ephemeral cache has no disk and costs nothing against the storage budget.

Queues & queue-scaled workers

The same primitive backs BullMQ-style job queues. Pair a cache with a worker process declaring scale_on and it scales on queue depth — down to zero replicas when the queue is empty — via a KEDA ScaledObject on the redis-lists trigger, no new infrastructure required. Cron-style scheduled jobs also exist, via app schedules.

Audit

cache.create and cache.delete are recorded in the audit trail; the password is never logged.
