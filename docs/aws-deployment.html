Reference

AWS deployment

Deploy Drop on AWS two ways from one container image: Variant A on Amazon EKS via the Helm chart, or Variant B on ECS Fargate (no Kubernetes). Both share a foundation — one ECR image, a private S3 bucket, an ACM wildcard cert, RDS PostgreSQL 18, and three Secrets Manager secrets — and both put Drop behind one internal ALB. Drop is internal/confidential: nothing here is internet-facing; the ALB is reachable only from the corp network / VPN.

Terraform

Prefer infrastructure-as-code? The same components are scaffolded as Terraform in infra/terraform/ — a foundation config plus eks and ecs root configs that read it via remote state (S3-native locking, no DynamoDB). See the Terraform page. The AWS CLI commands below are the click-by-click equivalent.

Shared foundation

Everything below is required by both deployment variants (EKS/Helm and ECS/EC2): one ECR image, a private S3 bucket, an ACM wildcard cert, an RDS PostgreSQL 18 instance, and three Secrets Manager secrets — wired together by the shell vars you export once here.

Prerequisites & vars

Install AWS CLI v2, Docker, and Node 24 (Drop's runtime). You'll need an existing VPC with private subnets reachable from the corp network / VPN — Drop is internal, so nothing here is internet-facing. Export these once; every command in this guide reuses them.

# Identity / region
export REGION=us-east-1
export ACCOUNT=$(aws sts get-caller-identity --query Account --output text)

# Existing network (fill in your own IDs)
export VPC_ID=vpc-0123456789abcdef0
export PRIVATE_SUBNETS="subnet-0aaa111 subnet-0bbb222"   # 2+ AZs

# Drop domain — sites live at *.$BASE_DOMAIN, control plane at api.$BASE_DOMAIN
export BASE_DOMAIN=drop.example.com

# S3 bucket name for site bytes (must be globally unique — add a suffix if taken)
export BUCKET=drop-sites

# Route 53 hosted zone that serves $BASE_DOMAIN
export ZONE_ID=Z0123456789ABCDEFGHIJ

# Corp network / VPN range allowed to reach the ALB on :443
export CORP_CIDR=10.0.0.0/8

# Filled in by later steps:  $ALB_SG $APP_SG $RDS_SG $CERT_ARN $ECR_URI $DB_ENDPOINT
Note

The corp CIDR / VPN range that may reach the ALB on :443 is exported as $CORP_CIDR (e.g. 10.0.0.0/8) and reused when the security groups are created in the variant-specific sections — keep it set.

1 — ECR: build & push the image

Drop ships one image (infra/Dockerfile) containing both bundles; the role is chosen at run time by overriding the command (node dist/api.js or node dist/edge.js). The image copies a prebuilt bundle — it runs no npm — so you build the bundle on the host first with node build.mjs, then docker build.

# Create the repo (one repo for both roles)
aws ecr create-repository \
  --repository-name drop \
  --image-scanning-configuration scanOnPush=true \
  --region "$REGION"

export ECR_URI=$ACCOUNT.dkr.ecr.$REGION.amazonaws.com/drop

# Log Docker in to ECR
aws ecr get-login-password --region "$REGION" \
  | docker login --username AWS --password-stdin \
      "$ACCOUNT.dkr.ecr.$REGION.amazonaws.com"

# Build the self-contained bundle on the host (esbuild → dist/), then the image
node build.mjs
docker build -f infra/Dockerfile -t "$ECR_URI:0.1.0" .

# Push
docker push "$ECR_URI:0.1.0"
Note

The container listens on 8080 (DROP_HTTP_PORT=8080, the default). Health checks: the api answers GET /healthz and the edge answers GET /_drop_health — both return HTTP 200 with body ok.

2 — S3: private bucket + IAM policy

File bytes live in a private bucket. The app authenticates with an IAM role (IRSA on EKS, task role on ECS, instance role on EC2) — never static keys.

# Create the bucket (us-east-1 takes no LocationConstraint)
if [ "$REGION" = "us-east-1" ]; then
  aws s3api create-bucket --bucket "$BUCKET" --region "$REGION"
else
  aws s3api create-bucket --bucket "$BUCKET" --region "$REGION" \
    --create-bucket-configuration LocationConstraint="$REGION"
fi

# Lock it down: block all public access, default encryption
aws s3api put-public-access-block --bucket "$BUCKET" \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

aws s3api put-bucket-encryption --bucket "$BUCKET" \
  --server-side-encryption-configuration \
  '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

Attach this policy document to the app role (created in the variant-specific section). Object-level actions on /*, list at the bucket root:

cat > drop-s3-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DropObjectRW",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::$BUCKET/*"
    },
    {
      "Sid": "DropList",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::$BUCKET"
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name drop-s3 \
  --policy-document file://drop-s3-policy.json
Note

Note the arn:aws:iam::$ACCOUNT:policy/drop-s3 returned here — you attach it to the IRSA role (EKS) or task/instance role (ECS/EC2) in the variant section.

3 — ACM: wildcard certificate

One wildcard cert for *.$BASE_DOMAIN covers both api.$BASE_DOMAIN and every site host (each a single label under $BASE_DOMAIN). Use DNS validation.

export CERT_ARN=$(aws acm request-certificate \
  --domain-name "*.$BASE_DOMAIN" \
  --validation-method DNS \
  --region "$REGION" \
  --query CertificateArn --output text)

# Read the CNAME ACM wants you to publish
aws acm describe-certificate --certificate-arn "$CERT_ARN" --region "$REGION" \
  --query 'Certificate.DomainValidationOptions[0].ResourceRecord'

Create that CNAME in Route 53 (substitute the Name/Value from the output above), then wait for validation:

cat > acm-validation.json <<EOF
{
  "Changes": [{
    "Action": "UPSERT",
    "ResourceRecordSet": {
      "Name": "_REPLACE_WITH_ACM_NAME",
      "Type": "CNAME",
      "TTL": 300,
      "ResourceRecords": [{ "Value": "_REPLACE_WITH_ACM_VALUE" }]
    }
  }]
}
EOF

aws route53 change-resource-record-sets \
  --hosted-zone-id "$ZONE_ID" \
  --change-batch file://acm-validation.json

aws acm wait certificate-validated --certificate-arn "$CERT_ARN" --region "$REGION"
Note

A single wildcard does not cover the apex $BASE_DOMAIN itself — Drop only needs the one-label hosts (api.$BASE_DOMAIN and each <site>.$BASE_DOMAIN), so *.$BASE_DOMAIN is sufficient.

4 — RDS: PostgreSQL 18

All metadata (sites, members, versions, users, auth) lives in Postgres. The api runs schema migrations on boot under a pg_advisory_lock (so a multi-replica rollout is safe — one pod migrates, the rest wait); the edge connects read-only and never migrates. Create the instance private, inside $PRIVATE_SUBNETS.

# RDS security group — ingress 5432 is opened from APP-SG in the variant section
export RDS_SG=$(aws ec2 create-security-group \
  --group-name drop-rds-sg \
  --description "Drop RDS Postgres" \
  --vpc-id "$VPC_ID" \
  --query GroupId --output text)

# DB subnet group across the private subnets
aws rds create-db-subnet-group \
  --db-subnet-group-name drop-db-subnets \
  --db-subnet-group-description "Drop private subnets" \
  --subnet-ids $PRIVATE_SUBNETS

# Generate and stash the DB password (used to build DROP_DATABASE_URL in step 5)
export DB_PASSWORD=$(openssl rand -hex 24)

# PostgreSQL 18, private (no public IP)
aws rds create-db-instance \
  --db-instance-identifier drop-db \
  --engine postgres \
  --engine-version 18 \
  --db-instance-class db.t4g.small \
  --allocated-storage 20 \
  --storage-type gp3 \
  --storage-encrypted \
  --master-username dropadmin \
  --master-user-password "$DB_PASSWORD" \
  --db-name drop \
  --db-subnet-group-name drop-db-subnets \
  --vpc-security-group-ids "$RDS_SG" \
  --no-publicly-accessible \
  --backup-retention-period 7

aws rds wait db-instance-available --db-instance-identifier drop-db

# Capture the endpoint host for the connection string
export DB_ENDPOINT=$(aws rds describe-db-instances \
  --db-instance-identifier drop-db \
  --query 'DBInstances[0].Endpoint.Address' --output text)
Note

$DB_PASSWORD only exists in this shell. Capture it into DROP_DATABASE_URL in step 5 immediately — it is not retrievable from RDS afterward.

5 — Secrets Manager

Drop reads three secret values at runtime. Create one secret each — named exactly for the env var it carries — so the variant section can mount them by name / ARN.

Secret nameWhat it is
DROP_GOOGLE_CLIENT_SECRETClient secret of your Google "Web application" OAuth client (redirect URI https://api.$BASE_DOMAIN/auth/callback).
DROP_SESSION_SECRETRandom key that signs Drop session tokens — openssl rand -hex 32.
DROP_DATABASE_URLpostgres:// URL built from the RDS endpoint above.
# Google OAuth client secret (from the Google Cloud console)
aws secretsmanager create-secret \
  --name DROP_GOOGLE_CLIENT_SECRET \
  --secret-string "GOCSPX-xxxxxxxxxxxxxxxx" \
  --region "$REGION"

# Session signing secret
aws secretsmanager create-secret \
  --name DROP_SESSION_SECRET \
  --secret-string "$(openssl rand -hex 32)" \
  --region "$REGION"

# Build the connection string from the RDS endpoint, then store it
export DROP_DATABASE_URL="postgres://dropadmin:$DB_PASSWORD@$DB_ENDPOINT:5432/drop"

aws secretsmanager create-secret \
  --name DROP_DATABASE_URL \
  --secret-string "$DROP_DATABASE_URL" \
  --region "$REGION"
Note

These three are the only secrets. Everything else is plain env, set per role in the variant section — api: DROP_DEV_AUTH=0, DROP_BASE_DOMAIN, DROP_PUBLIC_URL=https://api.$BASE_DOMAIN, DROP_S3_BUCKET, DROP_S3_REGION, DROP_ALLOWED_DOMAINS, DROP_ADMINS, DROP_GOOGLE_CLIENT_ID; edge: DROP_BASE_DOMAIN, DROP_S3_BUCKET, DROP_S3_REGION, DROP_EDGE_DISK_CACHE=/cache.

Variant A — Kubernetes (EKS)

Run Drop on Amazon EKS: the api and edge run as two Deployments from one image, S3 access flows through an IRSA-bound ServiceAccount (no static keys), and a single internal ALB — provisioned by the AWS Load Balancer Controller from the chart's Ingress — does host-based routing for api.<baseDomain> and *.<baseDomain>. This variant assumes the shared foundation (VPC, private subnets, ACM wildcard cert, RDS, S3, ECR, Secrets Manager) is already in place.

The whole section uses one consistent set of shell variables. Set them once from your foundation outputs:

# Foundation outputs — reuse these across every command below
export REGION=us-east-1
export ACCOUNT=123456789012
export VPC_ID=vpc-0123456789abcdef0
export PRIVATE_SUBNETS="subnet-0aaa111 subnet-0bbb222"   # 2+ AZs, private
export BASE_DOMAIN=drop.example.com
export RDS_SG=sg-0rds0123456789abc                       # RDS-SG from the foundation (ingress 5432 from cluster)
export CERT_ARN=arn:aws:acm:$REGION:$ACCOUNT:certificate/abcd-1234   # *.$BASE_DOMAIN
export ECR_URI=$ACCOUNT.dkr.ecr.$REGION.amazonaws.com/drop
export DB_ENDPOINT=drop-db.xxxx.$REGION.rds.amazonaws.com
export BUCKET=drop-sites
export ZONE_ID=Z0123456789ABCDEFGHIJ                     # Route 53 hosted zone for $BASE_DOMAIN

export CLUSTER=drop
Note

The ACM cert is a single wildcard for *.$BASE_DOMAIN. Because api.$BASE_DOMAIN and every site host are one label under $BASE_DOMAIN, that one cert covers the API and all sites — no per-site certs. On EKS the ALB security group (ALB-SG, ingress 443 from $CORP_CIDR) and the app security group (APP-SG, ingress 8080 from ALB-SG) are managed by the AWS Load Balancer Controller and EKS respectively; you only wire RDS-SG to the cluster in step 5.

1. Create the cluster in the private subnets

Create an EKS cluster with a managed node group placed in $PRIVATE_SUBNETS, and enable the IAM OIDC provider in the same step (required for IRSA in step 2). eksctl takes the subnet list comma-separated:

eksctl create cluster \
  --name $CLUSTER \
  --region $REGION \
  --version 1.30 \
  --vpc-private-subnets $(echo $PRIVATE_SUBNETS | tr ' ' ',') \
  --node-private-networking \
  --managed \
  --nodes 2 --nodes-min 2 --nodes-max 6 \
  --node-type m6i.large \
  --with-oidc

# Point kubectl at the new cluster
aws eks update-kubeconfig --name $CLUSTER --region $REGION
Tip

Prefer the console or an existing IaC pipeline? Create the cluster with aws eks create-cluster / aws eks create-nodegroup into the same $PRIVATE_SUBNETS, then run eksctl utils associate-iam-oidc-provider --cluster $CLUSTER --region $REGION --approve to get the OIDC provider that the next steps depend on.

2. IRSA: bind the Drop ServiceAccount to the S3 IAM role

Drop reaches S3 through the pod's IAM role — no access keys. The S3 IAM policy (drop-s3) was already created in foundation step 2; here we have eksctl create an IAM role bound to the drop ServiceAccount in the drop namespace. The Helm chart creates that ServiceAccount (serviceAccount.create=true) and we annotate it with the role ARN in step 4.

# Bind an IAM role to the drop/drop ServiceAccount (IRSA).
# --role-only: eksctl makes the role; the chart creates the ServiceAccount.
eksctl create iamserviceaccount \
  --cluster $CLUSTER --region $REGION \
  --namespace drop --name drop \
  --attach-policy-arn arn:aws:iam::$ACCOUNT:policy/drop-s3 \
  --role-name drop-s3 \
  --role-only \
  --approve

export DROP_ROLE_ARN=arn:aws:iam::$ACCOUNT:role/drop-s3
Note

--role-only tells eksctl to create just the IAM role (with the OIDC trust policy for system:serviceaccount:drop:drop) and not the ServiceAccount object — the Helm chart owns that. We pass $DROP_ROLE_ARN to serviceAccount.annotations."eks.amazonaws.com/role-arn" in step 4 so the two line up.

3. Install the AWS Load Balancer Controller

The controller turns the chart's alb Ingress into a real internal ALB with target groups and listeners. Give it its own IAM policy and IRSA role, then install it via Helm.

# Download and create the controller's IAM policy
curl -fsSL -o /tmp/alb-iam-policy.json \
  https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.8.1/docs/install/iam_policy.json

aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file:///tmp/alb-iam-policy.json

# IRSA role for the controller's ServiceAccount in kube-system
eksctl create iamserviceaccount \
  --cluster $CLUSTER --region $REGION \
  --namespace kube-system --name aws-load-balancer-controller \
  --attach-policy-arn arn:aws:iam::$ACCOUNT:policy/AWSLoadBalancerControllerIAMPolicy \
  --role-name AmazonEKSLoadBalancerControllerRole \
  --approve

# Install the controller via Helm (it discovers the VPC from cluster metadata)
helm repo add eks https://aws.github.io/eks-charts
helm repo update

helm upgrade --install aws-load-balancer-controller eks/aws-load-balancer-controller \
  --namespace kube-system \
  --set clusterName=$CLUSTER \
  --set region=$REGION \
  --set vpcId=$VPC_ID \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller

# Wait until it's ready
kubectl -n kube-system rollout status deploy/aws-load-balancer-controller

4. Install Drop with the Helm chart

The Secret holding the three sensitive values must exist in the drop namespace before install. Pull them from AWS Secrets Manager and create a Kubernetes Secret whose keys match the chart's expected key names (DROP_GOOGLE_CLIENT_SECRET, DROP_SESSION_SECRET, DROP_DATABASE_URL), then reference it via secret.existingSecret.

kubectl create namespace drop --dry-run=client -o yaml | kubectl apply -f -

# Pull secrets from AWS Secrets Manager into the chart's expected keys.
# The secret names match the foundation (named for the env var each carries).
kubectl -n drop create secret generic drop-secrets \
  --from-literal=DROP_GOOGLE_CLIENT_SECRET="$(aws secretsmanager get-secret-value \
      --secret-id DROP_GOOGLE_CLIENT_SECRET --region $REGION --query SecretString --output text)" \
  --from-literal=DROP_SESSION_SECRET="$(aws secretsmanager get-secret-value \
      --secret-id DROP_SESSION_SECRET --region $REGION --query SecretString --output text)" \
  --from-literal=DROP_DATABASE_URL="$(aws secretsmanager get-secret-value \
      --secret-id DROP_DATABASE_URL --region $REGION --query SecretString --output text)"

Now install the chart at infra/helm/drop. The ALB annotations make the controller build one internal ALB with an HTTPS:443 listener using the ACM wildcard cert, IP-mode target groups (pods are targeted directly), and a health check that the API answers at /healthz.

helm upgrade --install drop infra/helm/drop \
  --namespace drop --create-namespace \
  --set image.repository=$ECR_URI \
  --set image.tag=0.1.0 \
  --set baseDomain=$BASE_DOMAIN \
  --set apiHost=api.$BASE_DOMAIN \
  --set config.s3Bucket=$BUCKET \
  --set config.s3Region=$REGION \
  --set config.allowedDomains=example.com \
  --set config.admins=admin@example.com \
  --set googleClientId=YOUR_ID.apps.googleusercontent.com \
  --set secret.existingSecret=drop-secrets \
  --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=$DROP_ROLE_ARN \
  --set ingress.className=alb \
  --set ingress.annotations."alb\.ingress\.kubernetes\.io/scheme"=internal \
  --set ingress.annotations."alb\.ingress\.kubernetes\.io/target-type"=ip \
  --set ingress.annotations."alb\.ingress\.kubernetes\.io/listen-ports"='[{"HTTPS":443}]' \
  --set ingress.annotations."alb\.ingress\.kubernetes\.io/certificate-arn"=$CERT_ARN \
  --set ingress.annotations."alb\.ingress\.kubernetes\.io/healthcheck-path"=/healthz

Default publicUrl is https://$apiHost, so your Google OAuth "Web application" client must list https://api.$BASE_DOMAIN/auth/callback as an authorized redirect URI.

Note

The healthcheck-path=/healthz here is only the Ingress-level default. The chart already sets per-Service ALB health-check annotations — /healthz on the api Service and /_drop_health on the edge Service — which the AWS Load Balancer Controller applies to each target group, so the edge target group is checked on the correct path automatically. No manual console fix needed.

Note

The API runs schema migrations on boot under a Postgres pg_advisory_lock, so rolling out multiple api replicas is safe — one pod migrates while the others wait, then all serve. The edge connects read-only and never migrates. Health checks: api answers GET /healthz, edge answers GET /_drop_health (both 200 "ok").

5. Allow the cluster to reach RDS

The chart targets pods directly (IP-mode), so database traffic egresses from the EKS nodes. Add an ingress rule on RDS-SG that permits Postgres (5432) from the cluster's node/primary security group.

# Discover the cluster security group EKS attaches to nodes/control-plane ENIs
export NODE_SG=$(aws eks describe-cluster --name $CLUSTER --region $REGION \
  --query 'cluster.resourcesVpcConfig.clusterSecurityGroupId' --output text)

# RDS-SG: ingress 5432 from the EKS cluster SG
aws ec2 authorize-security-group-ingress \
  --group-id $RDS_SG \
  --protocol tcp --port 5432 \
  --source-group $NODE_SG \
  --region $REGION
Tip

If your node group uses a separate custom security group (the APP-SG role from the foundation), point --source-group at that SG instead so the rule matches the foundation's SG chain (ALB-SG → APP-SG → RDS-SG).

6. Point DNS at the controller-created ALB

The controller provisions the ALB once the Ingress is admitted. Read its DNS name and hosted-zone ID off the Ingress, then create Route 53 ALIAS records for api.$BASE_DOMAIN and the wildcard *.$BASE_DOMAIN.

# Wait for the ALB to be provisioned, then grab its DNS name
kubectl -n drop get ingress drop \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
export ALB_DNS=$(kubectl -n drop get ingress drop \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')

# The ALB's own canonical hosted-zone ID (for the ALIAS target)
export ALB_ZONE=$(aws elbv2 describe-load-balancers --region $REGION \
  --query "LoadBalancers[?DNSName=='$ALB_DNS'].CanonicalHostedZoneId | [0]" \
  --output text)

# ALIAS api.$BASE_DOMAIN + *.$BASE_DOMAIN -> the ALB
cat > /tmp/drop-dns.json <<JSON
{
  "Comment": "Drop ALB aliases",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "api.$BASE_DOMAIN",
        "Type": "A",
        "AliasTarget": { "HostedZoneId": "$ALB_ZONE", "DNSName": "$ALB_DNS", "EvaluateTargetHealth": true }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "*.$BASE_DOMAIN",
        "Type": "A",
        "AliasTarget": { "HostedZoneId": "$ALB_ZONE", "DNSName": "$ALB_DNS", "EvaluateTargetHealth": true }
      }
    }
  ]
}
JSON

aws route53 change-resource-record-sets \
  --hosted-zone-id $ZONE_ID \
  --change-batch file:///tmp/drop-dns.json
Note

One ALB, host-based routing: requests to api.$BASE_DOMAIN hit the api target group (8080), and everything else under *.$BASE_DOMAIN hits the edge target group (8080). Both records are ALIASes to the same internal ALB; the wildcard cert covers both. Because the ALB is internal it resolves to private IPs — use a private hosted zone associated with $VPC_ID (or internal DNS) for these records. Verify from inside the corp network / VPN: curl -I https://api.$BASE_DOMAIN/healthz should return 200.

1. Security groups

Three groups: ALB-SG accepts HTTPS from the corp CIDR/VPN only (Drop is internal/confidential); APP-SG accepts 8080 only from the ALB; and the existing RDS-SG gets a 5432 ingress rule from APP-SG so the tasks can reach Postgres.

# ALB security group — HTTPS 443 from the corp network / VPN only
ALB_SG=$(aws ec2 create-security-group --region $REGION \
  --group-name drop-alb-sg --description "Drop internal ALB" \
  --vpc-id $VPC_ID --query GroupId --output text)
aws ec2 authorize-security-group-ingress --region $REGION \
  --group-id $ALB_SG --protocol tcp --port 443 --cidr $CORP_CIDR

# App (task) security group — 8080 only from the ALB
APP_SG=$(aws ec2 create-security-group --region $REGION \
  --group-name drop-app-sg --description "Drop ECS tasks" \
  --vpc-id $VPC_ID --query GroupId --output text)
aws ec2 authorize-security-group-ingress --region $REGION \
  --group-id $APP_SG --protocol tcp --port 8080 --source-group $ALB_SG

# Wire RDS-SG to accept Postgres 5432 from the tasks
aws ec2 authorize-security-group-ingress --region $REGION \
  --group-id $RDS_SG --protocol tcp --port 5432 --source-group $APP_SG

2. Internal ALB, target groups, and the HTTPS listener

Create the ALB with --scheme internal (it is reachable only on the corp network / VPN, never from the public internet). Both target groups use --target-type ip on port 8080 — required for Fargate awsvpc tasks — but differ in their health-check path: the api TG checks /healthz, the edge TG checks /_drop_health (both return 200 "ok").

# Internal ALB across the private subnets
ALB_ARN=$(aws elbv2 create-load-balancer --region $REGION \
  --name drop-alb --scheme internal --type application \
  --subnets $PRIVATE_SUBNETS --security-groups $ALB_SG \
  --query 'LoadBalancers[0].LoadBalancerArn' --output text)

# api target group — health check GET /healthz
API_TG=$(aws elbv2 create-target-group --region $REGION \
  --name drop-api-tg --protocol HTTP --port 8080 --target-type ip \
  --vpc-id $VPC_ID \
  --health-check-protocol HTTP --health-check-path /healthz \
  --matcher HttpCode=200 \
  --query 'TargetGroups[0].TargetGroupArn' --output text)

# edge target group — health check GET /_drop_health
EDGE_TG=$(aws elbv2 create-target-group --region $REGION \
  --name drop-edge-tg --protocol HTTP --port 8080 --target-type ip \
  --vpc-id $VPC_ID \
  --health-check-protocol HTTP --health-check-path /_drop_health \
  --matcher HttpCode=200 \
  --query 'TargetGroups[0].TargetGroupArn' --output text)

# HTTPS:443 listener — DEFAULT action forwards to the edge TG (*.baseDomain)
LISTENER_ARN=$(aws elbv2 create-listener --region $REGION \
  --load-balancer-arn $ALB_ARN \
  --protocol HTTPS --port 443 \
  --certificates CertificateArn=$CERT_ARN \
  --ssl-policy ELBSecurityPolicy-TLS13-1-2-2021-06 \
  --default-actions Type=forward,TargetGroupArn=$EDGE_TG \
  --query 'Listeners[0].ListenerArn' --output text)

# Host-header rule: api.<baseDomain> -> api TG (everything else falls through to edge)
aws elbv2 create-rule --region $REGION \
  --listener-arn $LISTENER_ARN --priority 10 \
  --conditions Field=host-header,HostHeaderConfig="{Values=[api.$BASE_DOMAIN]}" \
  --actions Type=forward,TargetGroupArn=$API_TG
Note

The single ACM wildcard cert for *.$BASE_DOMAIN covers both api.$BASE_DOMAIN and the site hosts (each one label under $BASE_DOMAIN), so the one HTTPS listener serves every host. The default action sends *.$BASE_DOMAIN to the edge; only the explicit api.$BASE_DOMAIN rule diverts to the api.

3. IAM roles — execution role and task role

Fargate needs two roles. The task execution role lets ECS pull from ECR, write logs, and read the secret values at launch. The task role is the identity the app code runs as — it carries the S3 policy (no static keys).

# Trust policy shared by both roles — assumed by ECS tasks
cat > /tmp/ecs-trust.json <<'JSON'
{ "Version": "2012-10-17",
  "Statement": [{ "Effect": "Allow",
    "Principal": { "Service": "ecs-tasks.amazonaws.com" },
    "Action": "sts:AssumeRole" }] }
JSON

# Execution role: ECR pull + CloudWatch Logs (managed policy) ...
aws iam create-role --role-name drop-exec-role \
  --assume-role-policy-document file:///tmp/ecs-trust.json
aws iam attach-role-policy --role-name drop-exec-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

# ... plus permission to read the three Secrets Manager secrets at launch
cat > /tmp/exec-secrets.json <<JSON
{ "Version": "2012-10-17",
  "Statement": [{ "Effect": "Allow",
    "Action": "secretsmanager:GetSecretValue",
    "Resource": ["$SEC_DB","$SEC_SESSION","$SEC_GOOGLE"] }] }
JSON
aws iam put-role-policy --role-name drop-exec-role \
  --policy-name drop-read-secrets --policy-document file:///tmp/exec-secrets.json

# Task role: the app's runtime identity, carrying the S3 access policy
aws iam create-role --role-name drop-task-role \
  --assume-role-policy-document file:///tmp/ecs-trust.json

cat > /tmp/s3-policy.json <<JSON
{ "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow",
      "Action": ["s3:GetObject","s3:PutObject","s3:DeleteObject"],
      "Resource": "arn:aws:s3:::$BUCKET/*" },
    { "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::$BUCKET" } ] }
JSON
aws iam put-role-policy --role-name drop-task-role \
  --policy-name drop-s3 --policy-document file:///tmp/s3-policy.json

EXEC_ROLE_ARN=arn:aws:iam::$ACCOUNT:role/drop-exec-role
TASK_ROLE_ARN=arn:aws:iam::$ACCOUNT:role/drop-task-role

Both task definitions use the same image $ECR_URI and map container port 8080; they differ only in the command override and their environment. Plain config goes in environment; the three secrets are pulled by ARN via secrets[].valueFrom (resolved by the execution role). Create the log group first.

aws logs create-log-group --region $REGION --log-group-name /ecs/drop || true

5. ECS services (api and edge)

Create the cluster, then one Fargate service per role. Both run in $PRIVATE_SUBNETS with APP-SG and no public IP; each registers itself into its target group, where the ALB health-checks it on the role's path.

aws ecs create-cluster --region $REGION --cluster-name drop

# turn the space-separated subnet list into the comma form the CLI wants
SUBNETS_CSV=$(echo $PRIVATE_SUBNETS | tr ' ' ',')
NETCFG="awsvpcConfiguration={subnets=[$SUBNETS_CSV],securityGroups=[$APP_SG],assignPublicIp=DISABLED}"

# api service -> api target group (container "api", port 8080)
aws ecs create-service --region $REGION --cluster drop \
  --service-name drop-api --task-definition drop-api \
  --desired-count 2 --launch-type FARGATE \
  --network-configuration "$NETCFG" \
  --health-check-grace-period-seconds 60 \
  --load-balancers targetGroupArn=$API_TG,containerName=api,containerPort=8080

# edge service -> edge target group (container "edge", port 8080)
aws ecs create-service --region $REGION --cluster drop \
  --service-name drop-edge --task-definition drop-edge \
  --desired-count 2 --launch-type FARGATE \
  --network-configuration "$NETCFG" \
  --health-check-grace-period-seconds 60 \
  --load-balancers targetGroupArn=$EDGE_TG,containerName=edge,containerPort=8080
Note

Bring the api service up first and let it reach a steady state. On boot the api runs schema migrations under a pg_advisory_lock (multi-replica safe — one task migrates, the rest wait), and the edge connects read-only assuming the schema already exists.

6. DNS — Route 53 ALIAS records

Point both api.$BASE_DOMAIN and the wildcard *.$BASE_DOMAIN at the internal ALB with ALIAS A records. Fetch the ALB's DNS name and its canonical hosted-zone ID first.

ALB_DNS=$(aws elbv2 describe-load-balancers --region $REGION \
  --load-balancer-arns $ALB_ARN \
  --query 'LoadBalancers[0].DNSName' --output text)
ALB_ZONE=$(aws elbv2 describe-load-balancers --region $REGION \
  --load-balancer-arns $ALB_ARN \
  --query 'LoadBalancers[0].CanonicalHostedZoneId' --output text)

cat > /tmp/dns.json <<JSON
{ "Changes": [
  { "Action": "UPSERT",
    "ResourceRecordSet": {
      "Name": "api.$BASE_DOMAIN", "Type": "A",
      "AliasTarget": { "HostedZoneId": "$ALB_ZONE",
        "DNSName": "$ALB_DNS", "EvaluateTargetHealth": true } } },
  { "Action": "UPSERT",
    "ResourceRecordSet": {
      "Name": "*.$BASE_DOMAIN", "Type": "A",
      "AliasTarget": { "HostedZoneId": "$ALB_ZONE",
        "DNSName": "$ALB_DNS", "EvaluateTargetHealth": true } } }
] }
JSON
aws route53 change-resource-record-sets --hosted-zone-id $ZONE_ID \
  --change-batch file:///tmp/dns.json
Note

Because the ALB is internal, these records resolve to private IPs. Use a private hosted zone associated with $VPC_ID (or your internal DNS) so corp-network / VPN clients resolve api.$BASE_DOMAIN and the site hosts. Set the Google OAuth "Web application" client's redirect URI to https://api.$BASE_DOMAIN/auth/callback.

Plain-EC2 alternative

Note

If you'd rather not run a container orchestrator at all, you can skip ECS and run the prebuilt bundle directly on EC2 instances behind the same internal ALB. Build with node build.mjs, copy dist/ to the host, and run two systemd units — one executing node dist/api.js and one node dist/edge.js, both listening on 8080 (set DROP_HTTP_PORT=8080 and the same env/secrets shown above; source secrets from Secrets Manager at start). Give the instances APP-SG and an instance profile carrying the same S3 policy from step 3 (no static keys). Register the api instances into the api target group (/healthz) and the edge instances into the edge target group (/_drop_health) using --target-type instance instead of ip; the listener, host-header rule, and Route 53 records are identical.
