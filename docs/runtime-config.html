Guides

Runtime config & feature flags

Every config change shouldn't be a redeploy. When you just need to flip a flag or nudge a knob, Drop gives each app a small runtime key/value store: drop config set <app> KEY=value, and the app reads the change live. It is deliberately small — a per-app KV, not a flag platform (no targeting rules, no percentage rollouts, no environments-within-an-app). Values are non-secret by definition; secrets keep going through the write-only secret path.

Config vs. secrets — the one distinction that matters

Config is readable in plaintext (in the console, over the API, in the SDK). Secrets are write-only — set once, injected into the pod, never returned. So put feature flags, tunables, copy, and non-sensitive URLs in config; put API keys, passwords, and tokens in drop secrets set. Drop enforces the line: the config store refuses credential-looking values (see below).

The model

A workload of type app owns a flat map of string keys to string values, plus a monotonically-increasing version that acts as an ETag. The store lives in the control plane's metastore (app_configs), so it is audited, editable from the console, and needs no schema of yours.

  • Keys are env-var-ish: a letter or underscore, then letters, digits, _ or . (so FEATURE_NEW_UI and feature.newUi both work), up to 128 chars.
  • Values are plain strings, capped at 4 KiB. Parse them app-side (a "true"/"25"/JSON blob is just a string here).
  • Version bumps on every set/rm that actually changes something. It's the ETag the SDK polls against — an unchanged poll costs a 304.

Setting config from the CLI

# set / update a key (non-secret)
$ drop config set billing FEATURE_NEW_UI=true
  ✓ billing: set FEATURE_NEW_UI (config version 1)

# list an app's config — key, value (plaintext), version
$ drop config ls billing
  { "config": { "FEATURE_NEW_UI": "true" }, "version": 1 }

# remove a key
$ drop config rm billing FEATURE_NEW_UI
  ✓ billing: removed FEATURE_NEW_UI (config version 2)

The same three actions live on the app's page in the console as an inline config table — add, edit, and remove rows, with values shown in plaintext (a note distinguishes it from the write-only secrets panel just above it). Both surfaces are gated on the configure permission; every mutation is audited (config.set / config.rm).

Credential-looking values are refused

If a key looks secret (API_KEY, DB_PASSWORD, STRIPE_SECRET, …) or a value looks like an opaque credential (a long, high-entropy, whitespace-free token), Drop rejects it — client-side and server-side — with:

✗ looks like a secret — use `drop secrets set` (config values are non-secret and stored in plaintext)

This reuses the same entropy/keyword heuristic that keeps credentials out of published templates. Prose, URLs, UUIDs, and short tunables pass; opaque tokens don't. If you genuinely have a secret, it belongs in the write-only secret path.

Reading config in your app: @drop/config

The read path is an HTTP endpoint plus a tiny, dependency-free SDK. @drop/config polls GET /v1/apps/<app>/config with an If-None-Match ETag on an interval (30 s by default), caches the map in memory, and fires onChange only when the version advances — so a steady app makes one cheap conditional request every 30 s and does nothing until something actually changes.

// zero-config inside a Drop app — DROP_CONFIG_URL + DROP_CONFIG_TOKEN are injected
import { createConfigClient } from "@drop/config";

const config = createConfigClient();               // polls every 30s

config.get("FEATURE_NEW_UI");                       // "true" | undefined (in-memory, no network)
config.getAll();                                    // { FEATURE_NEW_UI: "true", … } (a copy)

const off = config.onChange((map, version) => {    // fires when the version ETag changes
  console.log("config changed to v" + version, map);
});
// off()            — unsubscribe
// config.refresh() — poll once, now
// config.stop()    — stop the background poll

Outside a Drop app (a test, a script, a custom runtime) pass what you need explicitly:

createConfigClient({
  url:   "https://api.drop.example.com/v1/apps/billing/config",
  token: "drop_st_…",       // a config.read-scoped token
  pollMs: 5000,             // override the interval (0 disables the background poll)
});

The injected environment

Apps don't authenticate the read with anything new. On your first drop config set for an app, Drop lazily mints a per-app read token — an ordinary service token whose only scope is an implicit config.read:<app> — and injects two values into the app's write-only secret:

Env varWhat it is
DROP_CONFIG_URLThe endpoint @drop/config polls (<api>/v1/apps/<app>/config).
DROP_CONFIG_TOKENA read-only, config-scoped bearer token — it can read this app's config and nothing else.

Because they're injected via the secret path, they land in the pod on its next restart or deploy (like any secret). After that first pickup, config changes are hot — the SDK polls; no restart is needed. Minting is lazy (an app that never uses config never gets a token) and idempotent (subsequent sets don't re-mint). The token read scope is config.read only, so it can never read your app's metadata, secrets, or logs.

When to use this — and when a table in your database is fine

Honestly: if your app already has a managed database, a settings table in it is a perfectly good place for runtime config, and you may not need this at all. Reach for Drop's runtime config when you want:

  • Audit — every change is recorded (config.set/config.rm) with who and when, in the same trail as the rest of your platform activity.
  • Console editing — a non-developer (or you, without a psql session) can flip a flag from the app's page.
  • No schema — nothing to migrate, and it works for apps with no database at all.
  • Live reads — the ETag-polling SDK gives you change notifications without wiring up your own.

If instead you want row-level structure, transactions with the rest of your data, or very high write rates, use your database. This KV is for the small, cross-cutting knobs — not your application's data model.

Not a flag platform

There is deliberately no per-user targeting, no percentage rollout, and no A/B bucketing here — that scope belongs to a dedicated experimentation tool. Drop's runtime config is the 20% that removes the redeploy tax on the other 80% of "I just need to change one value" moments.
