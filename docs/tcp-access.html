Guides

TCP access (the L4 plane)

Everything Drop serves over HTTPS rides an L7 load balancer and a Google-authenticated session. But raw-TCP workloads — psql from your laptop, a Redis client, an MQTT broker, gRPC over plain TCP — have no HTTP request to route or session to check. The L4 plane is a small in-cluster TCP router (edge-tcp) that gives those workloads a path, using the only two routing keys that exist below HTTP: the TLS SNI hostname, or a dedicated port.

Read this first — what L4 does NOT do

The L4 plane does not authenticate platform identities. A TCP SYN carries no Google session and no role, so the router cannot check who you are the way the HTTP edge does. TCP exposure is therefore opt-in per resource, default off, and security rests on protocol-native auth (the database password, Redis requirepass, your own TLS client certs) plus NetworkPolicy fencing — the tenant default-deny gains a single allow-from-edge-tcp rule only for a workload you explicitly expose. If you need real per-user authorization and audit over a database connection, use db:proxy instead — it is the strongly-authorized alternative and the two are complements, not competitors.

Two routing modes: SNI vs port

At L4 there is no Host header. The router has exactly two ways to decide where a connection goes:

ModeRouting keyPorts usedGood for
SNIThe TLS server_name (<name>.<baseDomain>) the client puts in its ClientHelloShared, well-known (e.g. 5432) — one port serves many workloadsAny TLS protocol that sends SNI: modern libpq (PG 14+), TLS Redis, most GUI tools
PortThe port number alone — no parsing, no peekingOne dedicated port per workload, allocated from a range (default 7000–7099)Non-SNI protocols: plain Redis, MQTT, anything that speaks bytes before (or without) TLS

SNI is the primary mechanism because it is scarce-port-frugal — one shared listener fronts every SNI-routable workload. Dedicated ports are the fallback, and they are genuinely scarce (an NLB defaults to ~50 listeners), so the port range is capped and metered.

The router never terminates TLS

In SNI mode edge-tcp peeks the ClientHello to read the one cleartext field it needs — the SNI — and then splices the bytes straight through to the upstream. The TLS handshake completes end-to-end between your client and the workload; the router is a blind pipe and never sees your plaintext or your credentials.

Postgres on a shared port

libpq doesn't open with a TLS ClientHello — it first sends an 8-byte cleartext SSLRequest and waits for a one-byte answer before it will start TLS. So on the shared Postgres port the router speaks just enough of the protocol to get a routing key:

# the handshake edge-tcp relays, in order:
client  → router    # SSLRequest (8 bytes, cleartext)
router  → client    # 'S'  — the router answers ITSELF, so the client sends its ClientHello
client  → router    # ClientHello (TLS)  → router peeks the SNI, resolves the upstream
router  → upstream  # the ORIGINAL SSLRequest, replayed verbatim
upstream→ router    # 'S'  — read and discarded (the client already got its 'S')
router  → upstream  # ClientHello + splice  → TLS now completes end-to-end

Because the single 'S' byte is identical whether the router or the server sends it, both ends see a perfectly well-formed Postgres-over-TLS handshake while the encryption stays end-to-end. A client that tries GSSAPI encryption first (GSSENCRequest) is answered 'N' — the router doesn't offer GSS transport — and libpq then falls back to an SSLRequest, which re-enters the path above.

sslmode=require is the default

A plaintext Postgres connection (sslmode=disable) sends a cleartext startup packet with no SNI — there is no routing key on a shared port, so the router closes it. Connect with at least sslmode=require (and prefer verify-full with the cluster CA — that's what the first-class DB binding already sets for in-cluster apps). If your client is too old to send SNI, use a dedicated port or db:proxy.

Limits & accounting

Every connection is bounded so a long-lived or idle socket can't pin router memory or starve a tenant:

Per connection
  • Idle timeout — a spliced connection with no bytes in either direction for the timeout window has both sockets destroyed.
  • Handshake budget — a peer that dawdles through the SNI peek / preamble (or never finishes it) is dropped rather than holding a slot.
  • Per-workload concurrency cap — once a workload hits its concurrent-connection ceiling, new connections are refused (no upstream is even dialed).
  • Byte counters — bytes in/out and duration are recorded per closed connection, feeding the same metrics seam the WebSocket plane uses.

Because a TCP SYN can't wake a scaled-to-zero pod (the KEDA HTTP interceptor only wakes on HTTP), any app that exposes TCP must run with scale.min ≥ 1. Managed databases (CNPG) are always-on and unaffected.

db:proxy (authenticated tunnel)

When you need real per-user authorization and an audit trail on a database connection — the thing L4 structurally cannot provide — reach for db:proxy. drop db proxy <db> opens a local TCP listener; each connection rides an authenticated WebSocket to the control plane, which authorizes you (a per-user connect grant), audits the open, and splices to the database's <db>-rw Service in-cluster. Think of L4 as convenient, protocol-native reach, and db:proxy as the guarded door — the two are complements, not competitors.

Just reading a table? Use the SQL console

For a quick read-only look — no local tunnel, no client — the SQL console (drop db query <db> "select …", or the box on the database detail page) runs a single read-only, audited, capped query API-side. Reach for the tunnel here when you need to write, run a migration, or drive an interactive psql session — the console deliberately refuses writes.

# open a tunnel; each connection is authorized + audited
drop db proxy mydb                      # prints: listening on 127.0.0.1:<ephemeral-port>
drop db proxy mydb --port 6543          # or pin a local port
# then, in another shell:
psql "host=127.0.0.1 port=6543 sslmode=disable"

How it contrasts with L4 expose:

db:proxy (this)L4 expose
WhoPer-user platform authz — your Drop role/`connect` grant is checked on every tunnelNone — a TCP SYN carries no identity; security is protocol-native (the DB password) + NetworkPolicy
AuditEvery open is recorded (db.tunnel.open)Not per-connection (only the expose/unexpose config change is audited)

The ticket model (replay-proof)

Each accepted local connection first asks the API for a single-use tunnel ticket (POST /v1/databases/:name/tunnel-ticket, authorized by connect) — a random secret, 60 s TTL, bound to (you, this database), stored only as a hash. The WebSocket upgrade presents the ticket in ?ticket=…; the API redeems it exactly once (an atomic latch), so a captured ticket can't be replayed and a ticket for one database can't open another. The db.tunnel.open audit event is written at redemption (the connection), not issuance (which can be speculative). A per-user concurrent-tunnel cap bounds fan-out.

Honest about the hops (and TLS)

The local hop — your psql to the 127.0.0.1 listener — is plain loopback (that's why sslmode=disable is fine there; the loopback + the authenticated, audited tunnel is the security boundary). CNPG still serves TLS at the database layer, so if you keep TLS on, the handshake negotiates end-to-end over the tunnel and the DB certificate is verified by your client. This is deliberately not a raw kubectl port-forward — port-forward bypasses platform authz entirely; the whole point of db:proxy is that it doesn't.

Deployment posture

The tunnel dials the DB Service DNS from the API. In the normal in-cluster production deployment (DROP_TUNNEL_DIRECT=1) that's a direct dial to <db>-rw.<ns>.svc:5432. If the API runs outside the cluster (e.g. a local dev API with a remote cluster) it has no route to that Service, so the tunnel upgrade returns 501 — issue the ticket from, and run the tunnel against, an in-cluster API. There is no MCP tool for db:proxy (an agent opening a raw database socket is out of scope); it is a CLI feature for a human at a psql prompt.

Local parity

Locally, nginx grows a stream {} block (listen 5432; ssl_preread on;) that forwards the shared port and the dynamic range to edge-tcp exactly as the production NLB will. It ships off by default — port 5432 is already taken by Postgres on most dev machines — and is enabled with make tls TCP_PORTS=1.

Exposing a workload

TCP exposure is opt-in, default off, and audited. Turning it on for an app or a database records one row in the tcp_endpoints registry, adds a single tenant NetworkPolicy that lets only edge-tcp reach only that workload's pods, and — for a dedicated port — publishes the port on the edge-tcp Service so the NLB grows a listener for it. The tcp.expose / tcp.unexpose events land in the audit trail.

# a database over TLS-SNI on the shared Postgres port (no port consumed)
drop db expose pg                       # sugar for: drop expose pg --sni --protocol postgres
# an app on a dedicated port (non-SNI protocols: plain Redis, MQTT, …)
drop expose cache --port --protocol redis
# turn it off again
drop unexpose cache

Under the hood the CLI/MCP call POST /v1/sites/:name/expose with {mode: "sni"|"port", protocol?: "postgres"|"redis"|"tcp"} (authorized by the ship/manage tier — a site owner/editor or org owner/admin/member); DELETE /v1/sites/:name/expose unexposes. A --port exposure allocates the lowest free port from the range (default 7000–7099); an exhausted pool returns 409 — which is the nudge to prefer --sni.

Expose first, then deploy the TCP service

An app whose drop.yaml declares services: [{ protocol: tcp }] is refused at deploy unless it already has an expose row — so the order is expose, then deploy. And because a TCP SYN can't wake a scaled-to-zero pod, a TCP-exposed app must run with scale.min ≥ 1 (enforced both at expose and at deploy). Managed databases are always-on and unaffected.

Connect strings

Expose returns the connect string, and it also shows on the resource's detail page and in drop expose ls:

ModeConnect stringNotes
SNI (postgres)<name>.<baseDomain>:5432Connect with at least sslmode=require (prefer verify-full with the cluster CA) — the SNI hostname is the routing key.
Port<lb-host>:<port>The raw LB host (DROP_TCP_LB_HOST, defaults to the base domain) on the allocated port — no SNI needed.
 
$ drop expose ls
NAME    TYPE      MODE  PROTOCOL  PORT  CONNECT
pg      database  sni   postgres  —     pg.drop.example.com:5432   (sslmode=require)
cache   app       port  redis     7000  drop.example.com:7000

The console mirrors this: an exposure panel on the app and database detail pages shows the current mode/protocol, a copyable connect string, and expose / unexpose controls. MCP exposes the same via the expose and unexpose tools.
