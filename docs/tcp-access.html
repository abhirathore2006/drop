TCP access (the L4 plane)

Everything Drop serves over HTTPS rides an L7 load balancer and a Google-authenticated session. But raw-TCP workloads — psql from your laptop, a Redis client, an MQTT broker, gRPC over plain TCP — have no HTTP request to route or session to check. The L4 plane is a small in-cluster TCP router (edge-tcp) that gives those workloads a path, using the only two routing keys that exist below HTTP: the TLS SNI hostname, or a dedicated port.

Read this first — what L4 does NOT do

The L4 plane does not authenticate platform identities. A TCP SYN carries no Google session and no role, so the router cannot check who you are the way the HTTP edge does. TCP exposure is therefore opt-in per resource, default off, and security rests on protocol-native auth (the database password, Redis requirepass, your own TLS client certs) plus NetworkPolicy fencing — the tenant default-deny gains a single allow-from-edge-tcp rule only for a workload you explicitly expose. If you need real per-user authorization and audit over a database connection, use db:proxy instead — it is the strongly-authorized alternative and the two are complements, not competitors.

Two routing modes: SNI vs port

At L4 there is no Host header. The router has exactly two ways to decide where a connection goes:

ModeRouting keyPorts usedGood for
SNIThe TLS server_name (<name>.<baseDomain>) the client puts in its ClientHelloShared, well-known (e.g. 5432) — one port serves many workloadsAny TLS protocol that sends SNI: modern libpq (PG 14+), TLS Redis, most GUI tools
PortThe port number alone — no parsing, no peekingOne dedicated port per workload, allocated from a range (default 7000–7099)Non-SNI protocols: plain Redis, MQTT, anything that speaks bytes before (or without) TLS

SNI is the primary mechanism because it is scarce-port-frugal — one shared listener fronts every SNI-routable workload. Dedicated ports are the fallback, and they are genuinely scarce (an NLB defaults to ~50 listeners), so the port range is capped and metered.

The router never terminates TLS

In SNI mode edge-tcp peeks the ClientHello to read the one cleartext field it needs — the SNI — and then splices the bytes straight through to the upstream. The TLS handshake completes end-to-end between your client and the workload; the router is a blind pipe and never sees your plaintext or your credentials.

Postgres on a shared port

libpq doesn't open with a TLS ClientHello — it first sends an 8-byte cleartext SSLRequest and waits for a one-byte answer before it will start TLS. So on the shared Postgres port the router speaks just enough of the protocol to get a routing key:

# the handshake edge-tcp relays, in order:
client  → router    # SSLRequest (8 bytes, cleartext)
router  → client    # 'S'  — the router answers ITSELF, so the client sends its ClientHello
client  → router    # ClientHello (TLS)  → router peeks the SNI, resolves the upstream
router  → upstream  # the ORIGINAL SSLRequest, replayed verbatim
upstream→ router    # 'S'  — read and discarded (the client already got its 'S')
router  → upstream  # ClientHello + splice  → TLS now completes end-to-end

Because the single 'S' byte is identical whether the router or the server sends it, both ends see a perfectly well-formed Postgres-over-TLS handshake while the encryption stays end-to-end. A client that tries GSSAPI encryption first (GSSENCRequest) is answered 'N' — the router doesn't offer GSS transport — and libpq then falls back to an SSLRequest, which re-enters the path above.

sslmode=require is the default

A plaintext Postgres connection (sslmode=disable) sends a cleartext startup packet with no SNI — there is no routing key on a shared port, so the router closes it. Connect with at least sslmode=require (and prefer verify-full with the cluster CA — that's what the first-class DB binding already sets for in-cluster apps). If your client is too old to send SNI, use a dedicated port or db:proxy.

Limits & accounting

Every connection is bounded so a long-lived or idle socket can't pin router memory or starve a tenant:

Per connection
  • Idle timeout — a spliced connection with no bytes in either direction for the timeout window has both sockets destroyed.
  • Handshake budget — a peer that dawdles through the SNI peek / preamble (or never finishes it) is dropped rather than holding a slot.
  • Per-workload concurrency cap — once a workload hits its concurrent-connection ceiling, new connections are refused (no upstream is even dialed).
  • Byte counters — bytes in/out and duration are recorded per closed connection, feeding the same metrics seam the WebSocket plane uses.

Because a TCP SYN can't wake a scaled-to-zero pod (the KEDA HTTP interceptor only wakes on HTTP), any app that exposes TCP must run with scale.min ≥ 1. Managed databases (CNPG) are always-on and unaffected.

The authorized alternative: db:proxy

When you need real per-user authorization and an audit trail on a database connection — the thing L4 structurally cannot provide — reach for db:proxy. It opens a local listener whose connections ride an authenticated WebSocket to the control plane, which checks your role and splices to the database in-cluster. It also works on deployments with no L4 plane at all and needs no exposure opt-in. Think of L4 as convenient, protocol-native reach, and db:proxy as the guarded door.

Local parity

Locally, nginx grows a stream {} block (listen 5432; ssl_preread on;) that forwards the shared port and the dynamic range to edge-tcp exactly as the production NLB will. It ships off by default — port 5432 is already taken by Postgres on most dev machines — and is enabled with make tls TCP_PORTS=1.
