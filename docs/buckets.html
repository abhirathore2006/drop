Guides

Buckets — tenant object storage

Apps need somewhere to put uploads, exports and artifacts. A bucket is a managed S3-compatible object store, isolated per tenant, that you create in one command and bind to an app so its credentials arrive as environment variables — never in drop.yaml, never in a manifest, never in a response body after the one time they are shown.

A bucket is a workload type

Like a site, an app or a database, a bucket is a row in the shared resource namespace with an owner, an org, roles and an audit trail. It shows up in drop ls, the console groups it under Buckets, and it is deleted the same way. There is no object browser in the console — reading and writing objects is your app's job, not the platform's.

Create a bucket

drop bucket create avatars
#   ▸ creating bucket avatars…
#   ✓ bucket avatars ready
#      endpoint: http://…  bucket: …  prefix: buckets/<tenant>/avatars/
#   ✓ credentials — shown once, store them now:
#      S3_ACCESS_KEY_ID=…
#      S3_SECRET_ACCESS_KEY=…

Add --org <slug> to create it in a team org (defaults to your personal org). The credentials are printed exactly once — the platform never stores them and never returns them again. In practice you rarely copy them by hand: bind the bucket to an app (below) and Drop injects them for you.

RevealOnce credentials

Create and rotate reveal the access key one time. If you lose it, rotate to mint a fresh one — you can't ask for the old one back. This mirrors drop db password.

Bind a bucket to an app

Declare the dependency in your app's drop.yaml with a uses edge — the same mechanism that binds a database:

app:
  image: my-org/gallery:1
  uses:
    - bucket: avatars        # a bucket in the SAME org

On deploy, Drop resolves the bucket (it must be in the same org — a cross-org bind is refused with a 400), provisions its credentials, and writes them straight into the app's write-only secret, so the pod boots with these environment variables:

VariableWhat it is
S3_ENDPOINTThe S3 endpoint reachable from your pod (omitted in prod → your SDK uses the AWS default for the region).
S3_BUCKETThe bucket name to address.
S3_PREFIXYour tenant's key prefix — prepend it to every object key. This is the isolation boundary.
S3_ACCESS_KEY_ID / S3_SECRET_ACCESS_KEYThe access credentials.

Point your S3 client at S3_ENDPOINT/S3_BUCKET and write keys under S3_PREFIX. Because the creds ride the write-only secret path, they never appear in the manifest, the deploy response, or the console.

Multiple buckets

Bind more than one and each set of variables is prefixed with the resource name so they don't collide — e.g. AVATARS_S3_BUCKET, EXPORTS_S3_BUCKET. A single binding is unprefixed (S3_BUCKET). Inside a stack, the prefix is the resource key.

The isolation model

Every bucket maps to a prefix buckets/<tenant-namespace>/<name>/. Because the prefix leads with the tenant's namespace, two tenants' prefixes can never overlap — one tenant can neither read/write nor even enumerate another's keys. In production this is enforced by a per-tenant, prefix-scoped IAM policy (the template lives at infra/terraform/eks/bucket-policy.tf.example): the object grant is limited to arn:aws:s3:::<bucket>/<prefix>* and the ListBucket grant is conditioned to s3:prefix = <prefix>*.

Local vs production credentials

Locally (Floci/MinIO), a bucket is just a prefix inside the platform's own S3 bucket, and the credentials Drop hands you are the platform's shared test keys — they are not prefix-scoped, which is fine for dev where every tenant shares one key pair anyway. In production those shared keys are unsafe: the BucketStore port must be backed by an implementation that mints a per-tenant, prefix-scoped IAM/STS credential from the Terraform policy template above. The platform default (v1) is the local prefix store; the production key-minting store is a named follow-up, and the port comment points at the template.

Rotate credentials

drop bucket rotate avatars      # owner only; the new key is printed once

Rotation re-mints the access credentials. Apps that were bound to the bucket keep using their injected copy until you redeploy them (which re-reads the fresh credentials). In the console, the bucket's detail page has a rotate credentials button that reveals the new key once.

Usage & deletion

The bucket detail page (and drop bucket ls / MCP bucket_status) surfaces the endpoint, bucket, prefix, total size and object count — never the credentials. Size is an eventually-consistent sweep of object sizes under the prefix.

drop bucket rm avatars           # refuses (409) if the bucket holds objects
drop bucket rm avatars --force   # deletes it AND wipes its contents (audited)

Deleting a non-empty bucket requires --force so its contents can't be wiped by accident; the console danger zone shows the object count and asks you to type the name to confirm.

Quota

Buckets count toward two org limits. They occupy a slot against the per-org workload cap, and their object bytes count toward the org's storage budget (when one is set) alongside database PVC requests — see GET /v1/orgs/<slug>/usage, which now reports a storage section ({ databases: {count, requestedBytes}, buckets: {count, bytes}, budget }). The budget and the per-database storage cap are admin-adjustable per org via GET/PUT /v1/admin/orgs/<slug>/quotas; the storage figure is an approximate, eventually-consistent computation and is documented as such.

Audit

bucket.create, bucket.rotate and bucket.delete are recorded in the audit trail; the credentials themselves are never logged.
