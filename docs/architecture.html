Architecture

Two stateless Hono services share a Postgres metadata store and an S3 byte store. The api is the control plane (auth, publish, admin, dashboard; owns migrations); the edge is the data plane (serves bytes by hostname, read-only against Postgres).

File bytes live in S3 (sites/<name>/files/<verId>/…). Everything else — sites, members, versions, users, auth handles — lives in Postgres. Ports: api 8473, edge 8474, Postgres 5432, S3/Floci 4566 (local), nginx 443 (local HTTPS, optional).

1. System overview

The api and edge are the only moving parts; both scale horizontally because all shared state is in Postgres + S3. The edge never writes and never migrates.

2. Publish flow

drop publish ./dist myapp streams a tarball to the api, which writes the bytes to S3 and flips the live pointer in Postgres. The atomic name claim is INSERT … ON CONFLICT DO NOTHING (first writer wins); the pointer flip is a row-locked transaction (it replaced the old S3 ETag CAS).

3. Serve flow

A browser requests https://<name>.drop.example.com/path; the edge resolves the site, enforces visibility, and serves bytes. The long-lived in-memory cache holds only the small per-site pointer (10 s TTL) — asset bytes are never retained in memory: they're served from the node-local disk cache (OS page cache keeps hot files at RAM speed) or fetched per-request from S3 and freed after the response, so memory use is bounded by concurrency, not by site count.

4. Data model

Exactly one owner per site is enforced by a partial unique index (unique(site_name) where role='owner'). Deleting a site cascades to its members and versions. Authorization is two-axis — platform role (users.role) + per-site role (site_members.role), resolved by can(actor, action) in src/authz/permissions.ts. Visibility is an independent axis (who may view the served pages) from roles (who may manage the site).

5. Local development topology

Two options. make start runs api/edge as Node processes against Floci (S3) and Postgres in podman — fast iteration over http://…:<port>. The docker compose stack runs everything in containers behind nginx for trusted HTTPS on :443, mirroring the prod ingress: https://api.drop.localhost/ and https://<name>.drop.localhost/. nginx sets X-Forwarded-Host, which the edge reads so the site name survives the proxy. See Getting started.

6. Production topology (Kubernetes)

Helm (infra/helm/drop) deploys api + edge behind one ingress. Postgres is an external managed database (RDS / CloudSQL); S3 is the real bucket via IRSA. Migrations run on api-pod boot under a pg_advisory_lock, so a multi-replica rollout is safe — one pod migrates, the rest wait then serve. DROP_DATABASE_URL is injected from a Secret into both deployments. See Deployment.
