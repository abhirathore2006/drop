Architecture

Two stateless Hono services share a Postgres metadata store and an S3 byte store. The api is the control plane (auth, publish, admin, dashboard; owns migrations); the edge is the data plane (serves bytes by hostname, read-only against Postgres).

File bytes live in S3 (sites/<name>/files/<verId>/…). Everything else — sites, members, versions, users, auth handles, and organisations (the group that owns every resource — see §8) — lives in Postgres. Ports: api 8473, edge 8474, Postgres 5432, S3/Floci 4566 (local), nginx 443 (local HTTPS, optional).

Drop is also an opt-in compute platform: alongside static sites it runs container apps, managed databases, and write-only secrets on a Kubernetes cluster. Compute is enabled only when the api has DROP_KUBECONFIG; otherwise /v1/apps & /v1/databases return 501 and Drop is static-only. See §7 Compute plane. Compute port: k3s 6443 (local).

1. System overview

The api and edge are the only moving parts; both scale horizontally because all shared state is in Postgres + S3. The edge never writes and never migrates.

2. Publish flow

drop publish ./dist myapp streams a tarball to the api, which writes the bytes to S3 and flips the live pointer in Postgres. The atomic name claim is INSERT … ON CONFLICT DO NOTHING (first writer wins); the pointer flip is a row-locked transaction (it replaced the old S3 ETag CAS).

3. Serve flow

A browser requests https://<name>.drop.example.com/path; the edge resolves the site, enforces visibility, and serves bytes. The long-lived in-memory cache holds only the small per-site pointer (10 s TTL) — asset bytes are never retained in memory: they're served from the node-local disk cache (OS page cache keeps hot files at RAM speed) or fetched per-request from S3 and freed after the response, so memory use is bounded by concurrency, not by site count.

4. Data model

Exactly one owner per site is enforced by a partial unique index (unique(site_name) where role='owner'). Deleting a site cascades to its members and versions. Every resource (site/app/database) also belongs to exactly one organisation via sites.org_id (see §8). Authorization is resolved by can(actor, action) in src/authz/permissions.ts as the union of three layers: platform-admin (all-powerful) OR the actor's org-wide role on the resource's org OR a per-resource site_members.role (an additive grant — the broader of the two wins, never the narrower). Visibility is an independent axis (who may view the served pages) from roles (who may manage the resource).

5. Local development topology

Two options. make start runs api/edge as Node processes against Floci (S3) and Postgres in podman — fast iteration over http://…:<port>. The docker compose stack runs everything in containers behind nginx for trusted HTTPS on :443, mirroring the prod ingress: https://api.drop.localhost/ and https://<name>.drop.localhost/. nginx sets X-Forwarded-Host, which the edge reads so the site name survives the proxy. See Getting started.

6. Production topology (Kubernetes)

Helm (infra/helm/drop) deploys api + edge behind one ingress. Postgres is an external managed database (RDS / CloudSQL); S3 is the real bucket via IRSA. Migrations run on api-pod boot under a pg_advisory_lock, so a multi-replica rollout is safe — one pod migrates, the rest wait then serve. DROP_DATABASE_URL is injected from a Secret into both deployments. See Deployment.

7. Compute plane (apps · databases · secrets)

Opt-in, enabled by DROP_KUBECONFIG. The api is the only cluster writer; it talks to the Kubernetes API over plain node:https with server-side apply (no @kubernetes/client-node), behind a KubeClient port (FakeKube in tests). Everything a tenant owns lives in a per-tenant namespace drop-t-<slug(email)>-<hash>, locked down by:

  • NetworkPolicy — default-deny; the egress allowlist excludes the cluster pod/service CIDRs (DROP_BLOCKED_EGRESS_CIDRS) so tenants can't reach each other or the platform DB.
  • ResourceQuota + LimitRange (no unbounded pods; cpu/memory capped), PodSecurity (baseline) labels, and the gVisor RuntimeClass for untrusted images in prod.
  • Storage — a managed database's PVC is capped at 1Gi per database today (requests over it are rejected at create with a 400). A fuller per-app + per-tenant, admin-adjustable storage quota is planned (see Future.md §10).

Apps

drop deploy → appManifests translates the drop.yaml app: section into a Deployment + Service + HTTPScaledObject (KEDA HTTP add-on) + ingress NetworkPolicy. No replicas on the Deployment — KEDA owns the count (0..max). The edge proxies <name>.drop.example.com to the KEDA interceptor via node:http (preserving the Host header), which wakes a scaled-to-zero pod on the first request.

LifecycleWhat the api does to the cluster
restartBumps a pod-template annotation (rolls the pods; applies new secrets/config)
stopPauses the ScaledObject (paused-replicas: 0) and scales to 0 — true offline; won't wake on traffic
startUn-pauses; restores the configured scale
Note

runtime_state in Postgres makes a stop survive redeploys, so a stopped app stays offline until you start it.

Image delivery

An app can run a prebuilt app.image from drop.yaml (BYO-image, unchanged), or you can build & push through Drop with no registry credentials on the client — the same command works locally and in prod. drop push <dir> [name] builds the Dockerfile locally, tags it drop.local/<app>:<tag>, streams a docker save tarball to PUT /v1/apps/:name/image, and prints the in-cluster ref; drop deploy <dir> --build chains that push then deploys. The CLI builder is docker by default (DROP_BUILDER=podman to switch).

The api never embeds a registry client — it depends on an ImageStore port (FakeImageStore in tests) with the backend chosen at deploy time (DROP_IMAGE_BACKEND):

BackendHow the tarball reaches the clusterIn-cluster ref
containerd (default, local)Imports the archive into the k3s node's containerd via <runtime> exec -i <k3s> ctr … images import -, then verifies the node holds the exact refdrop.local/<app>:<tag> — never resolved; non-latest tag → IfNotPresent, so the pod uses the imported image with no registry and no imagePullSecret
registry (prod)skopeo copy docker-archive:… docker://… pushes to a registry / ECR (DROP_IMAGE_REGISTRY) — no Docker daemon in the api pod<registry>/<app>:<tag>; the Deployment references the operator-configured DROP_IMAGE_REGISTRY_PULL_SECRET (or node IRSA for ECR)

The CLI tags the image and declares the same <tag> to the api so the imported ref and the Deployment's image string match exactly (otherwise the pod ImagePullBackOffs against drop.local); a fresh tag per build rolls the pods. Uploads larger than DROP_IMAGE_MAX_BYTES (default 2 GiB) are rejected. Server env: DROP_IMAGE_RUNTIME (default podman), DROP_K3S_CONTAINER (default k3s), DROP_CONTAINERD_SOCK.

CLI only

Image push is exposed on the CLI (push, deploy --build), not yet as an MCP tool. In-cluster builds (no local Docker) are a future item.

WebSockets

WebSockets pass through the public edge to type=app workloads. Upgrades bypass Hono: a Node-level server.on('upgrade') handler (src/edge/ws-proxy.ts) runs the same visibility gate as the HTTP path before the handshake — a private site gets a 403 and a password site a 401 written to the raw socket, so a viewer-blocked upgrade never half-opens (no 101). Past the gate the edge opens the upgrade against the same KEDA interceptor the HTTP path uses (Host <name>.drop.example.com, so KEDA routes and wakes the pod) and splices the two sockets byte-for-byte — the app computes Sec-WebSocket-Accept and emits the 101; the edge is a transparent tunnel. A fallback (DROP_WS_DIRECT=1) dials the app Service in-cluster directly with a wake shim, for interceptor builds that reject WS.

Each connection is bounded: a per-host concurrent-upgrade cap (DROP_WS_MAX_PER_HOST, default 100) and an idle timeout (DROP_WS_IDLE_TIMEOUT_MS, default 5 min) that destroys both sockets, so long-lived sockets can't pin edge memory; bytes in/out are counted in one place for a future metrics slice.

Cold start

On a scale-from-zero app the first upgrade waits for KEDA to wake a pod, so cold-start latency applies at the WS handshake. Apps that need instant WebSockets should set scale.min: 1 so a pod is always live (broadcast / presence state is per-pod anyway — see examples/chat-ws). In prod, also raise the ALB idle timeout (default 60 s) via an ingress annotation so the load balancer doesn't drop quiet sockets before the edge's 5-minute timeout would.

Databases

drop db create → a CloudNativePG Cluster (Postgres 18, single instance), with backups via the Barman Cloud Plugin (ObjectStore + ScheduledBackup, method plugin) to S3 (local Floci, prod IRSA). The app user/db are bootstrapped from a platform-owned <db>-app Secret. drop db password rotates the role password with a one-shot, idempotent in-namespace ALTER ROLE Job (a role changes its own password — no superuser), then syncs the Secret. Apps connect in-namespace to <db>-rw.

Secrets

Write-only per-app secrets behind a SecretStore port, with the backend chosen at deploy time (DROP_SECRET_BACKEND):

BackendHow the api persists a keyHow it reaches the pod
kubeMerge-patches the <app>-secret Secret per key (set never prunes siblings)Directly — the Secret is the source
awsWrites one AWS Secrets Manager secret per key at drop/<ns>/<app>/<KEY> and reconciles an ExternalSecret (explicit per-key remoteRefs)The External Secrets Operator syncs it into <app>-secret

Either way the Deployment envFroms <app>-secret (listed after <app>-env, so a secret wins on a key collision; optional so an absent Secret never blocks startup). The metastore holds only key names + a fingerprint.

Write-only

A secret value is never returned, logged, or persisted outside the secret manager and the pod env — there is no read-back path.

Local = prod shape

Locally the cluster is k3s-in-podman with KEDA, CNPG, and ESO installed by make compute-up; the aws secrets backend and CNPG backups run against Floci's S3 + Secrets-Manager emulation, so the local stack exercises the same code paths as EKS. See CLI, MCP & Dashboard for the commands and console.

8. Organisations (resource grouping + permissions)

An organisation is a logical group that owns resources, with org-level roles. Every site/app/database belongs to exactly one org (sites.org_id, added in migration m0004). There are two kinds:

  • Personal — auto-created (idempotently) on each user's first login. Its namespace is the user's existing per-tenant namespace, so the backfill moves no workload. One per user (unique(created_by) where kind='personal').
  • Team — created explicitly with drop org create <slug> [name]; the creator becomes owner and the org gets its own namespace (orgSlugNamespace(slug)).

Org roles apply org-wide (every resource in the org), resolved in src/authz/permissions.ts:

Org roleCan
owner / adminEverything (incl. share, transfer, delete, and managing org members)
memberread, logs, publish, deploy, db:create, rollback, configure — i.e. ship + manage config/secrets, but not share/transfer/delete a resource
viewerread only

The effective check is a union: can() returns true if the actor is a platform admin, OR their org role on the resource's org allows the action, OR their per-resource site_members.role does (site_members remain as an additive grant layer — see §4). Create flows have no resource yet, so they check the org role directly (canCreateInOrg); publish, deploy, and db:create take --org <slug> and default to your personal org.

Manage orgs from any surface: CLI drop org create|ls|members|add|rm (e.g. drop org add <slug> <email> [role]) over /v1/orgs*, or the MCP tools org_create, org_list, org_add_member. See Roles & Visibility.
