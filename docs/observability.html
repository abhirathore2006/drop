Observability

Two signals, both produced entirely inside the network — no external prober, no third-party SaaS. Request metrics (G2): the edge meters every response it serves or proxies and rolls it up per minute. Uptime checks (G2b): the API polls each workload's own URL on an interval and records whether it answered. Everything lands in Postgres next to the rest of Drop's state, is readable over the same authenticated API, and is retained for 30 days. A third signal — retained pod logs (G4) — tails running workloads into gzipped S3 objects for time-range, grep-grade search long after the pod is gone.

Request metrics

Both edges — the HTTP edge and the L4 (TCP) edge — accumulate per-host counters in-process and flush them every ~15 seconds into a traffic_minutes rollup: one row per host per minute. Writing the rollup is the only metastore write the edge performs (it stays a migrations reader); multiple edge replicas UPSERT the same (host, minute) row additively.

ColumnWhat it counts (per host, per minute)
requestsResponses served (every static serve, app proxy, redirect, and auth denial) plus each closed WS/TCP stream.
bytes_in / bytes_outRequest/response body bytes. Static serves set an exact content-length; a streamed body with none contributes 0 (WS/TCP byte counts are exact).
p50_ms / p95_msApproximated upstream latency (see Percentile honesty).
s2xx / s4xx / s5xxResponse counts by status class. (1xx/3xx count toward requests but no class.)

A preview host <site>--<label> rolls up under its own key, separate from its parent site — the metering key is the resolved serving host, whatever it was.

Reading it

GET /v1/sites/<name>/metrics?range=1h|24h|7d (authz: read) returns a time-bucketed series plus window totals. The point count stays bounded regardless of range: 1h is per-minute, 24h aggregates to 10-minute buckets, 7d to hourly — all server-side.

{
  "range": "1h",
  "series": [
    { "minute": "2026-07-04T10:00:00.000Z", "requests": 812, "p50": 8, "p95": 62, "errors": 3, "bytesOut": 1734921 },
    …
  ],
  "totals": { "requests": 1234, "errors": 5, "bytesIn": 40213, "bytesOut": 3251902, "p50": 8, "p95": 62 }
}

The CLI surfaces the totals as a one-liner; the console detail panel shows the same numbers (sparkline charts land in M4):

$ drop status myapp
  myapp (last hour): 1234 req · p50 8ms · p95 62ms · 0.4% errors · 3.1 MiB out
Percentile honesty

p50/p95 are approximated from fixed histogram buckets — 5, 10, 25, 50, 100, 250, 500, 1000, 2500, 5000 ms — never true percentiles, because Drop doesn't retain per-request samples. When flush windows merge into the same minute (four flushes a minute, and across edge replicas): summed counts are exact; merged p95 is the max of the windows (conservative); merged p50 is a request-weighted average of the windows' p50 approximations. WS/TCP stream durations are deliberately not folded into the latency histogram — a connection lifetime is not a request latency, and mixing them would corrupt p95 — so a closed stream counts only toward requests and bytes.

Uptime checks

A poller inside the API probes each qualifying workload once on an interval (default 60s) and records the outcome into uptime_checks — one row per minute, last-write-wins. Who gets probed:

TypeProbed?How
Site (static)AlwaysHTTP GET through the edge.
AppOnly when scale.min ≥ 1 or healthcheck.keep_warm: trueHTTP GET through the edge. A scale-to-zero app without keep_warm is deliberately not probed — a probe would wake the pod.
DatabaseOnly when the API is in-cluster (DROP_TUNNEL_DIRECT=1)TCP connect to the CNPG rw Service. Otherwise skipped (the local API can't reach it).

HTTP probes go through the edge with the workload's own Host header, so they exercise real routing. A check is OK when the edge returns a non-5xx response within 5 seconds — so a password-gated 401 or a 404 still counts as reachable. A 5xx, a timeout, or a connection failure is DOWN.

# GET /v1/sites/<name>/uptime?range=24h  (authz: read)
{
  "range": "24h",
  "checks": [ { "minute": "2026-07-04T10:00:00.000Z", "ok": true, "latencyMs": 45, "status": 200 }, … ],
  "summary": { "last24hPct": 99.8, "lastCheck": { "ok": true, "latencyMs": 45, "status": 200, "at": "2026-07-04T10:00:00.000Z" } }
}

The site detail response embeds the same uptime summary, and the console shows it as a line:

  99.8% (24h) · 45ms

keep_warm

app:
  image: myapi:1
  healthcheck:
    keep_warm: true
Opt-in observability only

keep_warm opts a scale-to-zero app into uptime probing. It is purely an observability signal — it does not touch the KEDA scale bounds (min stays 0), so the pod still scales down between probes and real traffic. Its only effect is that the poller will now GET the app on each sweep (and thus wake it briefly) instead of skipping it. It stands alone: you don't need an HTTP healthcheck.path to use it. See healthcheck:.

Prometheus

GET /metrics on the API exposes the rollup in Prometheus text format, admin-gated. It reflects the last flushed minute per site — a cheap traffic_minutes query, not a live in-process counter (the API is not the meter; the edge is). Because each value is a per-minute snapshot rather than a monotonic total, every metric is a gauge. This is for operators who already run Prometheus — Drop doesn't ship dashboards.

# HELP drop_edge_requests Requests served for a site in the last flushed minute.
# TYPE drop_edge_requests gauge
drop_edge_requests{site="myapp"} 812
drop_edge_bytes_out{site="myapp"} 1734921
drop_edge_bytes_in{site="myapp"} 40213
drop_edge_p50_ms{site="myapp"} 8
drop_edge_p95_ms{site="myapp"} 62
drop_edge_status{site="myapp",class="2xx"} 809
drop_edge_status{site="myapp",class="4xx"} 2
drop_edge_status{site="myapp",class="5xx"} 1

Events & alerting

Deliberately minimal, event-driven, no pager pretensions. When something notable happens, the API writes a row to a per-org events feed (keyset-paged like the audit log) and — if the org has configured one — fires a single outbound webhook. Sinks v1 are the console notification badge + feed and that webhook; there is no on-call rotation, no escalation, no dashboards.

KindSeverityFires whenRecovers on
crashlooperrorA workload's pods keep restarting (a housekeeping sweep diffs restart counts per active namespace against the previous sweep).The app returns to Running its desired replicas.
deploy_failederrorA release Job halts a deploy (the old version keeps serving).The next successful deploy of that app.
stack_haltederrorA stack up halts partway (a step failed; applied steps persist).The next clean stack up.
quotawarningA create is refused for the org's workload cap or storage budget (a 429).— (the org's usage dropping isn't polled; the incident is deduped to one row until swept).
preview_expiringinfoThe sweep is about to reap a preview that saw edge traffic in the last hour.— (a point-in-time heads-up).
Dedup & resolve

At most one open incident per (org, site, kind). A repeat emit while one is open doesn't insert a new row — it bumps a count in the incident's detail and freshens its timestamp, so a flapping crash-loop or a throttled quota warning is one feed entry, not a flood. Recovery sets resolved_at (the incident closes); a later emit after a resolve opens a fresh incident. The unread badge counts open warning/error incidents across your orgs (folded into /v1/me) — info notices show in the feed but don't badge.

Outbound webhook

One webhook per org (drop org webhook set <url>, owner/admin-gated). On each new incident (and on recovery — never on a dedup bump) the API POSTs the event as JSON. Delivery runs in the API process — no queue, no new infra — with a bounded retry/backoff (3 attempts, exponential; a permanent 4xx stops early). The posture is honest best-effort, at-most-once: the retries ride over a transient webhook failure, but a process crash mid-delivery does not replay — the durable record is the events table (re-read the feed). A configured signing secret HMAC-SHA256-signs the body as X-Drop-Signature: sha256=<hex>.

# generic JSON (Teams incoming webhooks + any endpoint)
{ "kind": "crashloop", "title": "crash-loop: api", "severity": "error",
  "siteName": "api", "detail": { "restarts": 7, "count": 3 }, "resolved": false }

# a hooks.slack.com URL instead gets Slack's {text} shape (renders as a line)
{ "text": "🔴 [drop] error: crash-loop: api · api" }
Slack & Teams

A Slack incoming-webhook URL (hooks.slack.com) is auto-detected and sent Slack's minimal {text} body so it renders without a bespoke integration; a richer Slack Block Kit adapter is a follow-up. Teams incoming webhooks and any other endpoint receive the generic event JSON above.

Surfaces

# the org feed (any member; keyset-paged, newest first)

GET  /v1/orgs/<slug>/events?unresolved=1     # just the open warning/error count

MCP has no events surface — alerting is a human/ops concern, not an agent one.

Log retention & search

Live logs (drop logs -f) die with the pod — a crash post-mortem needs history. Full Loki is against this platform's weight class, so Drop takes the middle path: the API tails each running workload's pod logs (one follow stream per workload, resumable by sinceTime), batches the lines, and writes them as gzipped NDJSON objects in the S3 the platform already owns — one object per (site, hour) at logs/<site>/<hour>.ndjson.gz — with one log_objects index row per object. No log database, no per-line index.

Each NDJSON line is a structured record:

{ "ts": "2026-07-04T10:15:03.221Z", "site": "api", "pod": "api", "stream": "stdout", "line": "GET /health 200" }

Collection is bounded on three axes so it never becomes a fleet-wide firehose: only running, opted-in workloads are tailed; a cap on concurrent tails (DROP_LOG_MAX_CONCURRENT_TAILS, default 50); and a per-hour in-memory line cap that drops the oldest under flood. The posture is at-least-once: a collector restart resumes a tail from the last line's timestamp so it doesn't re-ingest the whole backlog, but a few lines either side of the boundary may duplicate — grep-grade history doesn't need dedup.

Search (grep-grade, honestly)

Search is time-range + text-match, not full-text. The range narrows to the set of objects via the index; the text match then streams through those objects server-side — a plain substring test (optionally case-insensitive) or an optional regex. There is no tokenizer and no inverted index: it is grep over the retained NDJSON, and the API says so. Objects are read newest-first up to a cap (default 1000 lines, a truncated flag when hit), so a wide range returns the freshest matches without reading — or buffering — the whole history.

# API (same `logs` capability as the live tail — see the warning below)
GET /v1/sites/<name>/logs/search?from=<iso|epoch>&to=<iso|epoch>&q=<pattern>[&regex=1][&i=1][&limit=1000]

# CLI — a window (2h/24h/7d) or an ISO timestamp; distinct from the live `-f` follow
drop logs api --since 2h --grep "ERROR"
drop logs api --since 2026-07-04T00:00:00Z --until 2026-07-04T06:00:00Z --grep "timeout" -i

In the console, the workload's logs panel has a history toggle beside the live tail: a quick 1h/24h/7d window + a search box render matching lines into the same log view. The live tail is untouched.

Secrets warning — read this

Retained logs can contain env-injected secret values — and persisting them makes that leak durable, not just momentary. Search is therefore behind the same logs capability as the live tail (strictly above viewer — a viewer is metadata-only). If an app logs its secrets, opt it out (below). Databases are EXCLUDED by default (Postgres logs are verbose and can echo query text / bound parameter values) — opt in explicitly.

drop.yamlDefaultEffect
app.log_retention: falseapps/sites onOpt this app OUT — its pod logs are never retained (still available live via -f).
database.log_retention: truedatabases offOpt this database IN — otherwise a DB's verbose logs are not collected.

Retention defaults to 7 days and is org-overridable via the quota table's log_retention_days key (0 disables retention). The housekeeping sweep resolves each site's window from its owning org, then deletes each expired S3 object first, then its index row — orphan-safe: a crash between the two just re-issues the idempotent object delete on the next sweep, and no bytes are ever stranded without an index to find them.

Retention & config

traffic_minutes, uptime_checks, and the events feed are all retained 30 days and swept in the API's housekeeping loop (a range delete over the minute/created_at index, not a table scan). Events are cheap and valuable for a post-mortem, so they ride the same window; an actively-flapping incident keeps a fresh timestamp (dedup bumps it), so only truly-stale rows are swept.

Env varDefaultEffect
DROP_METRICS_FLUSH_INTERVAL_MS15000How often each edge flushes its in-process counters to traffic_minutes.
DROP_UPTIME_INTERVAL_MS60000How often the API's uptime poller sweeps.
DROP_EDGE_INTERNAL_URL(unset)The internal edge origin the poller GETs (Host-routed), e.g. http://drop-edge.drop-system.svc. Unset → HTTP uptime probes are disabled (with a startup warning); database TCP probes still run.
DROP_METRICS_RETENTION_DAYS30Retention window for both rollup tables and the events feed.
DROP_LOGS_COLLECT1Master enable for the log collector (needs compute/kube). Set 0 to turn it off.
DROP_LOG_RETENTION_DAYS7Default retention for retained log objects — org-overridable via the log_retention_days quota key.
DROP_LOG_MAX_CONCURRENT_TAILS50Cap on concurrent pod-log follow streams — the collector's fan-out bound.

On crash loops

A workload's current restart count shows live on the app-status and stack-graph endpoints (straight from the pods). Persistent crash-loops are now also detected by the housekeeping sweep and surfaced as crashloop events — one open incident that resolves when the app recovers — so you get a feed entry + webhook, not just a live number.
