Guides

CI tokens (service accounts)

CI shouldn't borrow a person's login. A session token is unscoped, tied to a human, dies when they leave, and shows up in the audit trail as them — so an automated deploy is indistinguishable from a hand-run one. A service token is the alternative: an org-owned, scoped bearer credential built for automation. It carries exactly the permissions you grant it, survives staff changes, and audits as token:<name>@<org> so automation is legible.

The shape of a token

A secret looks like drop_st_<48 hex>. Drop stores only its SHA-256 hash — the plaintext is shown once, at create, and never again (same posture as a database password). If you lose it, mint a new one and revoke the old.

Scopes: what a token may do

A token's authority is the union of its scopes. Each scope reuses one of the ordinary permission verbs, optionally qualified to a single resource:

FormMeansExample
verbThat verb on every resource in the orgpublish — publish any site
verb:*Identical to a bare verb — the explicit "all resources" formdeploy:*
verb:nameThat verb on one named resource onlydeploy:myapp

The verbs are the same ones roles grant — read, logs, publish, deploy, db:create, rollback, configure, expose, share, transfer, delete. An unknown verb is rejected at create with a clear error, so a typo can't silently mint a powerless (or, worse, over-broad) token. Grant the narrowest set that does the job: a deploy pipeline usually needs only deploy:myapp (plus publish:myapp-web if it also ships a static front end).

A token can never touch admin or org management

Tokens are for resource actions only. The scope grammar has no verb for suspending users, changing platform roles, editing quotas, or adding/removing org members — and the authorization layer refuses a token principal on every admin and org-management route regardless of what you write. A token is also fenced to its own org: it can act only on resources that org owns, never across orgs.

Creating a token

Only an org owner or admin can mint or revoke tokens. From the CLI:

# a deploy token for one app, expiring in 90 days
$ drop token create --org acme --name ci-deploy --scope deploy:myapp,publish:myapp-web --expires 90d
  ✓ token ci-deploy created (st_9f3c…)
     scopes:  deploy:myapp, publish:myapp-web
     expires: 2026-10-02T12:00:00.000Z
  ✓ secret — shown once, store it now (export it as DROP_TOKEN in CI):
     drop_st_1a2b3c…

--expires accepts 90d, 12w, 6mo, or 1y; omit it for a token that never expires. List and revoke are org-scoped too:

$ drop token ls --org acme
$ drop token revoke --org acme st_9f3c…

The console mirrors this under Settings › Tokens (visible to owners/admins): a scope builder (verb + optional resource, add/remove rows), a one-time reveal of the secret, last-used and expiry columns, and a guarded revoke.

Using a token in CI

Non-interactive tooling reads the token from the DROP_TOKEN environment variable — no drop login, no session file on disk. When DROP_TOKEN is set it takes precedence over any saved session; point the CLI at your instance with DROP_API. Store the secret in your CI provider's encrypted secrets, never in the repo.

# .github/workflows/deploy.yml — publish on every push to main
name: deploy
on:
  push:
    branches: [main]
jobs:
  publish:
    runs-on: ubuntu-latest
    env:
      DROP_API: https://api.drop.example.com
      DROP_TOKEN: ${{ secrets.DROP_TOKEN }}   # a drop_st_… service token
    steps:
      - uses: actions/checkout@v4
      - name: Install the Drop CLI
        run: curl -fsSL "$DROP_API/install.sh" | sh
      - name: Build
        run: npm ci && npm run build
      - name: Publish
        run: drop publish ./dist myapp-web --org acme
Preview deploys

A drop publish --preview flag (per-branch/per-PR preview URLs driven by the same token) is on the roadmap — this recipe will grow a PR-preview job when previews land. See the previews guide (coming soon) for the pattern.

Revocation & expiry

Every request re-checks the token against the database — a single indexed hash lookup — so control is immediate:

  • Revoke is a soft mark: the row stays (it keeps its audit value — you can still see what the token did) but the next request it makes returns 401. There is no grace period.
  • Expiry works the same way — once expires_at passes, the token stops authenticating.
  • Org deletion cascades: delete an org and its tokens go with it.

Because a token isn't a user, suspending people doesn't touch it and it doesn't touch them — rotate it on its own schedule (mint the new one, switch CI, revoke the old).

Everything a token does is audited as the token

A token-authenticated request records its actor as token:<name>@<org> in the append-only audit trail, so an automated deploy or delete is attributable to the credential — not to whoever happened to create it. Filter the trail by that actor to see exactly what a given pipeline has done.
