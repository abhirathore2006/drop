Guides

Local development (drop dev)

Drop is where apps end up, not where they're built — so running your code locally against its managed resources normally means hand-copying env and juggling kubectl port-forward. drop dev collapses that into one command: it opens authorized, audited tunnels to your app's bound databases, materializes an env with the connection hosts rewritten to localhost, overlays your local .env.dev, and runs your dev command with it — tearing the tunnels down when you Ctrl-C.

# from your app's directory (reads the app name from drop.yaml)
drop dev -- npm run dev
# or name the app and command explicitly
drop dev billing -- node server.js
It composes machinery you already have

drop dev invents no new plane. The tunnels are exactly db:proxy's authenticated WebSocket tunnels (the A3 ticket model); the env is the app's own non-secret config plus your local secrets; the process supervision is a plain child_process with inherited stdio. It's the same parts, wired for the inner loop.

What one drop dev does

  1. Fetches the app's dev-context — GET /v1/apps/:name/dev-context (authorized at the deploy tier). This returns the app's non-secret env, its DB/cache binding metadata (which resource, its in-cluster host/port, and which env var the binding injects — PGHOST/PGPORT for a database, REDIS_URL for a cache), and the names of the secret keys the app expects. No secret value ever crosses this endpoint.
  2. Opens a tunnel per bound database — reusing db:proxy, each on an allocated local port. Every connection rides a fresh single-use ticket over an authenticated tunnel, so it is per-user authorized and audited (see below).
  3. Materializes the env — the app's non-secret env, with each bound host rewritten to localhost:<port>, plus your .env.dev overlay for the secret values.
  4. Runs your command — the argv after --, or (if you give none) the app's web-process command from drop.yaml — inheriting stdio. Tunnels are torn down on Ctrl-C, SIGTERM, or when the child exits.
$ drop dev billing -- npm run dev
  ✓ overlaid 3 value(s) from .env.dev
  ✓ tunnel to billdb (database) on 127.0.0.1:55432 — rewrites PGHOST=localhost PGPORT=55432
  ⚠ cache "sessions" is not tunneled on this instance — set REDIS_URL in .env.dev to point at a local cache
  ▸ running: npm run dev
  # … your app boots against the managed DB over the authorized tunnel …
  # Ctrl-C → tunnels close

Secrets stay write-only — never pulled

This is the load-bearing rule. Drop's app secrets are write-only: you set them, the platform injects them into the pod, and nothing — not the API, not the console, not drop dev — ever reads them back. drop dev keeps that invariant intact. dev-context carries the names of the secret keys your app expects; you supply the values locally in .env.dev.

drop dev --check tells you exactly what to fill

Run --check to list the secret key names the app expects (names only, never values) and print a ready-to-fill .env.dev template — then it exits without opening anything.

$ drop dev billing --check
  billing: secret keys the app expects (NAMES only — values are never pulled):
    · JWT_SECRET
    · REDIS_URL
    · STRIPE_KEY
    binding: database billdb @ billdb-rw.drop-acme.svc.cluster.local:5432 (tunneled → PGHOST/PGPORT rewritten to localhost)
    binding: cache sessions @ sessions.drop-acme.svc.cluster.local:6379 (NOT tunneled here → set REDIS_URL in your .env.dev)

  ready-to-fill .env.dev template:
  # .env.dev — LOCAL dev credentials for billing.
  # Secrets are NEVER pulled from Drop — fill these with your OWN local/dev values.
  JWT_SECRET=
  REDIS_URL=
  STRIPE_KEY=
  # database binding(s): the tunnel rewrites PGHOST/PGPORT to a local port; supply the
  # connection creds (PGPASSWORD lives in the <db>-app Secret — never pulled; PGUSER/PGDATABASE default to 'app'):
  # billdb: PGPASSWORD=   PGUSER=app   PGDATABASE=app

Redirect that template into place and fill it with your own dev credentials:

drop dev billing --check > .env.dev   # then edit it; .env.dev is git-ignored, developer-owned
Why not just pull the real secrets?

Because a platform that can hand a developer's laptop the production Stripe key on demand is a platform that has quietly become the softest place to exfiltrate every secret it holds. Write-only secrets mean a compromised laptop (or a stolen session) can set a secret but never read one. drop dev refuses to be the hole in that wall — local dev uses local dev credentials, which is what you want anyway (you don't test against the live Stripe account).

The --env-file overlay

drop dev reads .env.dev from the current directory by default (absent → no overlay, that's fine). Point it elsewhere with --env-file <path> (an explicit path that doesn't exist is an error — a typo shouldn't silently run with no secrets). The file is a plain KEY=VALUE list: blank and # lines are skipped, a leading export  is tolerated, and surrounding quotes are stripped. No interpolation — these are credentials, not a shell.

Precedence, lowest to highest — the tunnel always wins because the allocated local port is the authoritative address:

LayerSourceExample
1. app envthe app's non-secret env: (the <name>-env source)NODE_ENV, LOG_LEVEL
2. overlayyour .env.dev — beats the app env for the same keyPGPASSWORD, STRIPE_KEY, REDIS_URL
3. tunnelhost rewrites — beat bothPGHOST=localhost, PGPORT=<port>

For a cache, the rewrite swaps only the host inside REDIS_URL to the local tunnel, keeping the userinfo/password from your .env.dev — so your local credential still authenticates while traffic routes through the tunnel.

The in-cluster-tunnel constraint (same as db:proxy)

Tunnels dial the DB Service in-cluster

Exactly like db:proxy and the SQL console, the tunnel splices to <db>-rw.<ns>.svc from the API process. In the normal in-cluster deployment (DROP_TUNNEL_DIRECT=1) that's a direct dial and everything just works. If you run drop dev against an API that lives outside the cluster (a local dev API pointed at a remote cluster), it has no route to that Service and the per-connection dial returns 501 — surfaced, never fatal.

You have two clean ways to develop:

  • Against a real in-cluster deployment — the recommended path. Tunnels light up, PGHOST points through the authorized tunnel to the managed database, and every connection is audited.
  • Against a local out-of-cluster API — pass --no-tunnel. drop dev then skips tunnels entirely and materializes the env from the app env + .env.dev only. Point PGHOST/REDIS_URL at your own local Postgres/Valkey in .env.dev — the classic inner loop, no cluster required.
# local API, local Postgres — env-only, no tunnels
drop dev billing --no-tunnel -- npm run dev
Caches aren't tunneled yet

Today only databases have a tunnel endpoint (/v1/databases/:name/tunnel). A bound cache is reported in --check and its REDIS_URL is listed as a key to fill — point it at a local Valkey/Redis in .env.dev. The dev-context already carries the cache binding metadata, so the day a cache tunnel lands, drop dev tunnels it with no change to your workflow.

Tunnels are authorized and audited

Because the tunnels are db:proxy's, a drop dev session against a prod-org database is exactly as visible and gated as any other tunnel: each connection is authorized by your per-user connect grant, and every open is recorded as a db.tunnel.open audit event (written at redemption — the actual connection — not at ticket issuance). A dev loop against a real database is a first-class, auditable action, not a back door. Teams that want a hard wall put dev in its own environment and scope roles accordingly.

The dev-context read itself isn't audited

It carries no secret — non-secret env, binding hosts, and key names — so like GET .../secrets (which returns key names too) it isn't audited. The security-relevant event is the tunnel open, and that already is.

Command reference

drop dev [app] [-- cmd…]        # app defaults to drop.yaml's app.name; cmd to the web process command
  --check                       # list secret KEY NAMES + print a .env.dev template, then exit
  --env-file <path>             # overlay local values from this file (default: .env.dev if present)
  --no-tunnel                   # skip tunnels; take connection hosts from .env.dev / the app env
